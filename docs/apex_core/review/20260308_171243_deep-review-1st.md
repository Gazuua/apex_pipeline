## Critical (6건)

### C-01. context_provider와 on_accept 간 코어 인덱스 불일치 (TOCTOU)
- **발견**: [아키텍처] [clangd LSP]
- **위치**: `server.cpp:89-93` (context_provider) / `server.cpp:261-265` (on_accept)
- **문제**: context_provider는 `next_core_.load(relaxed)`로 소켓을 IOCP 바인딩하고, on_accept은 `next_core_.fetch_add(1, relaxed)`로 코어를 결정. 멀티코어에서 IOCP 바인딩 코어와 실제 세션 할당 코어가 불일치 가능.
- **수정**: on_accept()에서 단일 fetch_add로 코어를 결정한 뒤, 해당 코어의 io_context에 소켓을 바인딩하는 구조로 변경. context_provider 콜백 제거.

### C-02. cross_core_call 타임아웃 후 참조 캡처 func 실행 시 UAF
- **발견**: [보안/안전성]
- **위치**: `cross_core_call.hpp:125-128` (void), `:67` (non-void)
- **문제**: 타임아웃 CAS 성공 후 호출자 코루틴이 재개되어 스택 프레임 해제. 그러나 타겟 코어의 task 람다가 CAS 이전에 func 실행을 시작한 경우, 참조 캡처된 코루틴 로컬 변수가 UAF.
- **수정**: 드레인 측에서 CAS를 먼저 수행하고, CAS 성공 시에만 func을 호출하도록 순서 변경.

### C-03. cross_core_call Result<R> 소멸 스레드 안전성 미강제
- **발견**: [아키텍처]
- **위치**: `cross_core_call.hpp:69-100`
- **문제**: 타임아웃 시 `optional<R>`이 타겟 코어 스레드에서 소멸. R이 thread-affine RAII 타입이면 UB. 주석 경고만 있고 컴파일 타임 강제 없음.
- **수정**: `static_assert(std::is_nothrow_destructible_v<R>)` 추가. 헤더 주석 보강.

### C-04. MPSC Queue 슬롯 경쟁 조건 — sequence number 부재
- **발견**: [보안/안전성]
- **위치**: `mpsc_queue.hpp:131-135`
- **문제**: head CAS 성공 후 slot.data 쓰기 전에 프로듀서가 선점되면, consumer가 같은 슬롯을 소비 후 다른 프로듀서가 동일 슬롯에 재진입 가능. slot당 sequence counter 부재.
- **수정**: 각 Slot에 `std::atomic<size_t> sequence` 필드 추가 (Vyukov MPMC 알고리즘). 또는 현재 구현이 실제 사용 패턴(단일 io_context drain_timer → 단일 consumer)에서 안전한 근거를 주석으로 명확히 문서화.

### C-05. process_frames 핫패스에서 매 프레임 std::vector 힙 할당
- **발견**: [성능]
- **위치**: `server.cpp:339-344`
- **문제**: `payload_copy`를 `std::vector<uint8_t>`로 매 프레임마다 heap-allocate. 10만 msg/sec 기준 초당 20만 번 malloc/free.
- **수정**: 스택 기반 SmallBuffer 패턴 적용 (4KB 이하 스택, 초과 시 heap fallback). 또는 SlabPool 연동.

### C-06. TCP_NODELAY 소켓 옵션 미설정
- **발견**: [성능]
- **위치**: `tcp_acceptor.cpp:50-55`
- **문제**: Nagle 알고리즘 활성 상태. 소형 패킷에서 최대 40ms 레이턴시 추가. 게임 서버급 성능 목표와 불일치.
- **수정**: `on_accept_` 콜백 호출 전 `tcp::no_delay(true)` 설정. ServerConfig에 옵션화.
