## Critical 이슈 (6건)

### C-1. Session::send() / send_raw() — io_context 스레드에서 동기 블로킹 write

- **파일:** `src/session.cpp:38-44, 50-56`
- **신뢰도:** 95 (3개 에이전트가 독립적으로 발견)
- **발견:** 세션 에이전트, 서버 에이전트, LSP 에이전트

`boost::asio::write()`는 동기 블로킹 호출이다. `Server::process_frames()`는 `async_read_some` 완료 핸들러 내에서 io_context 스레드에서 실행되며, 그 안에서 `session->send()`를 호출한다. 즉 io_context 스레드가 전송 완료까지 블로킹된다. 단일 io_context 서버에서 클라이언트 하나가 수신 버퍼를 채우면 해당 스레드 전체가 멈춰 다른 세션의 read/accept가 전부 중단된다.

채팅 브로드캐스트 시나리오(`for_each`로 각 세션에 `send()` 호출)에서는 N개 세션에 대해 N번 블로킹 write가 발생하며, 지연이 선형으로 누적된다.

```cpp
// session.cpp:38
boost::asio::write(socket_, boost::asio::buffer(frame), ec); // 블로킹!
```

**수정 방향:** `async_write` + 세션별 송신 큐(`std::deque<std::vector<uint8_t>>`)로 교체.

### C-2. TimingWheel::tick() — 콜백에서 reschedule() 호출 시 UAF

- **파일:** `src/timing_wheel.cpp:123-128`
- **신뢰도:** 92
- **발견:** 저수준 에이전트

`tick()`에서 Phase 2가 엔트리를 슬롯 링크드 리스트에서 제거한 뒤, Phase 3에서 콜백을 실행한다. 콜백 반환 후에 `entries_[id] = nullptr`과 `delete e`가 실행된다. 그런데 콜백이 `reschedule(id)`를 호출하면:

1. `reschedule()`에서 `entries_[id]`가 아직 nullptr이 아님 (이미 분리된 엔트리를 가리키는 중)
2. `remove_entry(entry, old_slot)` 호출 — entry의 `prev/next`는 이미 nullptr인 상태
3. `remove_entry`가 `slots_[slot_idx].head`를 nullptr로 덮어써서 해당 슬롯의 유효한 엔트리 리스트를 끊어버림

**재현 시나리오:**
```cpp
TimingWheel tw(64, [&](EntryId id) {
    tw.reschedule(id, 5);  // 하트비트 패턴 — 만료 시 재스케줄
});
auto id = tw.schedule(1);
tw.tick(); tw.tick(); // 만료 -> reschedule -> 메모리 corruption
```

**수정 방법:** `entries_[id] = nullptr`과 `free_ids_.push_back(e->id)`를 콜백 호출 전으로 이동:
```cpp
for (auto* e : expired) {
    entries_[e->id] = nullptr;   // 먼저 클리어
    free_ids_.push_back(e->id);
    on_expire_(e->id);           // 이제 reschedule(id)는 nullptr을 보고 -> 새 id로 schedule()
    delete e;
}
```

### C-3. TcpAcceptor::do_accept() — `[this]` 캡처에 의한 수명 미연장

- **파일:** `src/tcp_acceptor.cpp:36`
- **신뢰도:** 90
- **발견:** 서버 에이전트

`async_accept` 람다가 `[this]`를 캡처하지만 `TcpAcceptor`의 수명을 연장하지 않는다. `stop()`이 `acceptor_.close(ec)`를 호출한 뒤 `TcpAcceptor` 객체가 파괴되면, io_context 스레드의 큐에 남아있는 완료 콜백이 이미 소멸된 `this`에 접근하게 된다.

구체적 시나리오: `Server` 소멸자가 `stop()` 호출 -> `TcpAcceptor` 멤버 소멸 -> io_context 스레드에서 완료 콜백 실행 -> `this->running_` 접근 -> UB.

현재 예제(스택에 할당된 `Server`가 `io_ctx.run()` 이전에 선언)에서는 역순 소멸 덕분에 안전하다. 그러나 힙 할당이나 람다 캡처로 `Server`를 사용하면 즉시 UB가 발생한다.

**수정 방향:** `std::enable_shared_from_this` + `shared_from_this()` 캡처 사용, 또는 사용 패턴 강제("Server는 반드시 io_ctx.run() 반환 후 소멸") + 내부 `post()` 가드 추가.

### C-4. Server::do_read() — process_frames() 후 is_open() 재확인 누락

- **파일:** `src/server.cpp:95-101`
- **신뢰도:** 88 (2개 에이전트가 발견)
- **발견:** 서버 에이전트, LSP 에이전트

`process_frames(session)` 이후, 핸들러가 세션을 닫았을 수 있다(`session->close()` 또는 에러). 그러나 `touch_session()`과 `do_read()`가 무조건 호출된다:

```cpp
process_frames(session);                        // 내부에서 세션이 닫힐 수 있음
session_mgr_.touch_session(session->id());      // 닫힌 세션에 touch!
do_read(session);                               // is_open() 가드로 걸리지만
```

닫힌 세션에 `touch_session()`을 호출하면 타이머가 소생하여, `on_timer_expire`가 이미 제거된 세션 ID로 실행될 수 있다.

**수정 방법:** `process_frames()` 이후 `if (!session->is_open()) return;` 추가.

### C-5. Server::do_read() — RingBuffer 가득 참 시 수신 데이터 무음 손실

- **파일:** `src/server.cpp:86-92`
- **신뢰도:** 85 (2개 에이전트가 발견)
- **발견:** 서버 에이전트, LSP 에이전트

`recv_buf.writable()`이 빈 span을 반환하면 복사 루프가 조용히 중단된다:

```cpp
while (copied < bytes) {
    auto w = recv_buf.writable();
    if (w.empty()) break;   // 버퍼 가득 참 -> 나머지 데이터 조용히 버림
    ...
}
```

남은 `bytes - copied` 바이트는 임시 버퍼에 그대로 남아 다음 `do_read()`에서 덮어씌워진다. 프레임 중간이 절단되어 이후 `FrameCodec::try_decode`가 영구적으로 잘못된 오프셋을 파싱하게 된다. 에러 로깅도 없고 세션 종료도 없다.

**수정 방법:** `copied < bytes`이고 `w.empty()`인 경우 세션을 종료:
```cpp
if (w.empty()) {
    session->close();
    session_mgr_.remove_session(session->id());
    return;
}
```

### C-6. SessionManager::on_timer_expire() — 콜백에서 remove_session() 호출 시 댕글링 이터레이터

- **파일:** `src/session_manager.cpp:103-110`
- **신뢰도:** 80
- **발견:** 세션 에이전트

```cpp
auto session = session_it->second;     // shared_ptr 복사

if (timeout_callback_) {
    timeout_callback_(session);        // 내부에서 remove_session(session_id) 호출 가능
}

session->close();
sessions_.erase(session_it);           // session_it가 이미 무효화됐을 수 있음!
```

`timeout_callback_`이 `remove_session()`을 내부적으로 호출하면 `sessions_`에서 동일 항목이 먼저 삭제되어 `session_it`가 댕글링 이터레이터가 된다. 이후 `sessions_.erase(session_it)`는 UB이다.

**수정 방법:** 콜백 호출 전에 맵에서 먼저 제거:
```cpp
auto session = session_it->second;
sessions_.erase(session_it);           // 먼저 제거

if (timeout_callback_) {
    timeout_callback_(session);        // remove_session()은 이제 no-op (이미 제거됨)
}

session->close();
```

### I-1. ServiceBase::bind_dispatcher() — 호출 순서 미강제

### I-2. Session::State::Closing — 미사용 enum 값

- **파일:** `include/apex/core/session.hpp:26`
- **신뢰도:** 85
- **발견:** 세션 에이전트

헤더에 생명주기를 `Connected -> Active -> Closing -> Closed`로 문서화하고 `State::Closing` enum을 정의했으나, 실제로 `State::Closing`을 설정하는 코드가 전혀 없다. `session->close()`는 Active에서 Closed로 직접 전이한다. 설계와 구현 사이의 불일치.

추가로 `set_state()`가 public(line 58)으로 노출되어 있어, 외부에서 임의 상태 전이가 가능하다.

### I-3. next_power_of_2() — overflow 반환 0을 호출자가 미처리

- **파일:** `include/apex/core/detail/math_utils.hpp:12`, `mpsc_queue.hpp:82`
- **신뢰도:** 84
- **발견:** 저수준 에이전트

`next_power_of_2()`가 overflow 시 0을 반환한다. 호출자(`MpscQueue`, `RingBuffer`, `TimingWheel`)는 이를 검사하지 않는다:
```cpp
capacity_ = detail::next_power_of_2(capacity < 1 ? 1 : capacity);
mask_ = capacity_ - 1;  // capacity_=0이면 mask_ = SIZE_MAX -> 힙 OOB
```

현실적으로 엄청난 크기의 입력이 필요하므로 발생 가능성은 낮지만, 방어 코드가 권장된다.

**수정 방법:** `next_power_of_2` 호출 후 `if (capacity_ == 0) throw std::overflow_error(...)` 추가.

### I-4. ErrorSender — nullptr API 계약 미문서화

- **파일:** `src/error_sender.cpp:17-19`
- **신뢰도:** 83
- **발견:** 세션 에이전트

`message`가 비어있을 때 FlatBuffers offset이 기본값 0(null)이 된다. 수신측에서 `resp->message()->c_str()`를 직접 호출하면 null 역참조로 크래시가 발생한다. API 계약에 이 동작이 명시되어 있지 않다.

### I-5. Server::stop() — 스레드 안전성 제약 미문서화

- **파일:** `src/server.cpp:49-51`, `include/apex/core/server.hpp`
- **신뢰도:** 82 (2개 에이전트가 발견)
- **발견:** 서버 에이전트, LSP 에이전트

`Server::stop()`은 `sessions_` 맵에 대한 데이터 레이스를 방지하기 위해 반드시 io_context 스레드에서 호출해야 한다. E2E 테스트는 `boost::asio::post()`로 올바르게 처리하고 있으나, `Server` 헤더에 스레드 안전성 문서가 없다. 사용자가 외부 스레드에서 `stop()`을 직접 호출하면 UB가 발생한다.

**수정 방향:** 내부적으로 `post(io_ctx_, ...)` 가드 추가, 또는 헤더 주석에 제약 명시.

## 핵심 결론

가장 임팩트가 큰 이슈는 **C-1 (동기 블로킹 write)** 이다. 현재 단일 io_context 아키텍처에서 동기 `boost::asio::write()`를 사용하면 서버 처리량이 가장 느린 클라이언트의 TCP 수신 속도에 종속된다. "고성능 실시간 서버 파이프라인"을 표방하는 프로젝트에서 이것이 핵심 약점이다.

"즉시 수정" 단계(C-2, C-4, C-5, C-6)는 총 12줄 미만의 코드 변경으로 적용 가능하며, 설계 변경 전에도 견고성을 크게 개선한다.
