# 테스트 견고성 번들: BACKLOG-118 + BACKLOG-119

## 개요

- **브랜치**: `feature/test-robustness-118-119-115`
- **백로그**: BACKLOG-118 (reinterpret_cast UB), BACKLOG-119 (타이밍 의존 테스트), BACKLOG-115 (이미 수정 확인 → 히스토리)
- **스코프**: apex_core 테스트 코드

## BACKLOG-118: reinterpret_cast UB 제거

### 문제

`test_service_lifecycle.cpp:33-53`에서 미초기화 `aligned_storage`에 `reinterpret_cast`로 더미 참조 생성.
테스트가 해당 참조를 통해 멤버에 접근하지는 않으나 C++ 표준상 UB.

### 해결

4개 더미 객체를 실제 인스턴스로 교체. GTest fixture(`TEST_F`) 도입:

| 더미 | 교체 방법 |
|------|-----------|
| `dummy_server()` | `Server(minimal ServerConfig)` |
| `dummy_registry()` | `ServiceRegistry()` (기본 생성) |
| `dummy_registry_view()` | `ServiceRegistryView({&registry_})` |
| `dummy_scheduler()` | `PeriodicTaskScheduler(io_ctx_)` |

- `aligned_storage` + `reinterpret_cast` + NOLINT 블록 전체 삭제
- `make_cfg_ctx` 헬퍼는 fixture 멤버 사용으로 단순화
- 개별 `TEST()` → `TEST_F()` 전환 (fixture: `ServiceLifecycleTest`)
- fixture 외부의 서비스 클래스 정의(MinimalService, TrackerService 등)는 그대로 유지

## BACKLOG-119: 타이밍 의존 sleep_for 수정

### 문제

4개 테스트 파일에서 raw `sleep_for` 사용. TSAN/ASAN 환경에서 타임아웃 불일치 가능.

### 수정 전략: 패턴별 최적 전략

**이벤트 대기** → `wait_for` 폴링 전환:
- `test_periodic_task_scheduler.cpp:57` — `sleep_for(80ms)` → `wait_for([&] { return count.load() >= 1; })`
- `test_server_error_paths.cpp:66` — `sleep_for(500ms)` → `wait_for` 스레드 종료 감지

**부재 증명** → `timeout_multiplier()` 스케일링:
- `test_connection_handler.cpp:156` — `sleep_for(50ms)` → `sleep_for(50ms * timeout_multiplier())`
- `test_periodic_task_scheduler.cpp:77` — `sleep_for(100ms)` → `sleep_for(100ms * timeout_multiplier())`

**의도적 블로킹** → 유지 + 주석 보강:
- `test_cross_core_call.cpp:117` — `sleep_for(50ms)` io_context 스레드 블로킹은 의도적 (타임아웃 테스트). 주석으로 의도 명시.

## BACKLOG-115: 이미 수정 확인

`test_tcp_acceptor.cpp:80-108`의 StopPreventsNewAccepts는 현재 코드에서 결정적 패턴으로 수정 완료.
`stop()` → `connect()` → `t.join()` (io_ctx 완전 drain) → `EXPECT_EQ(0)`.
BACKLOG_HISTORY로 이동.
