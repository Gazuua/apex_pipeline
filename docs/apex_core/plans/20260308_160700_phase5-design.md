## 1. 개요

Phase 5는 이후 모든 Phase의 기반이 되는 4개 항목을 구현한다:

| 항목 | 내용 |
|------|------|
| TOML 설정 | 계층형 AppConfig, `[server]` + `[logging]` 우선, 확장 가능 |
| spdlog 통합 | 프레임워크 내장 로거, 2레벨(apex/app), Console 텍스트 + File JSON |
| Graceful Shutdown | drain_timeout TOML 설정, 타임아웃 시 warn + 강제 종료 |
| CI/CD | GitHub Actions 4잡 매트릭스 (MSVC, GCC, ASAN, TSAN) |

### 내부 의존 순서

```
TOML 설정 로딩 → spdlog 초기화 (TOML 연동) → Graceful Shutdown 타임아웃 (TOML에서 읽기)
CI/CD는 독립 (별도 진행 가능)
```

## 2. TOML 설정 시스템

### 2.1 설정 파일 구조

```toml
# config/default.toml

[server]
port = 9000
num_cores = 0              # 0 = auto (hardware_concurrency)
mpsc_queue_capacity = 65536
drain_interval_us = 100
heartbeat_timeout_ticks = 300
recv_buf_capacity = 8192
timer_wheel_slots = 1024
handle_signals = true
drain_timeout_s = 25       # Graceful Shutdown 타임아웃 (초)

[logging]
level = "info"                 # 글로벌 기본 레벨 (app 로거)
framework_level = "info"       # apex 로거 레벨 (독립 제어)
pattern = "%Y-%m-%d %H:%M:%S.%e [%l] [%n] %v"  # 콘솔 포맷

[logging.console]
enabled = true

[logging.file]
enabled = false
path = "logs/apex.log"
max_size_mb = 100
max_files = 3
json = true                    # FileSink는 JSON 포맷
```

Phase 6/7에서 `[kafka]`, `[redis]` 섹션 추가 시 구조 변경 없이 확장 가능.
알 수 없는 섹션/키는 무시 (forward compatibility).

### 2.2 C++ 구조체

```cpp
// include/apex/core/config.hpp
namespace apex::core {

struct LogConsoleConfig {
    bool enabled = true;
};

struct LogFileConfig {
    bool enabled = false;
    std::string path = "logs/apex.log";
    size_t max_size_mb = 100;
    size_t max_files = 3;
    bool json = true;
};

struct LogConfig {
    std::string level = "info";
    std::string framework_level = "info";
    std::string pattern = "%Y-%m-%d %H:%M:%S.%e [%l] [%n] %v";
    LogConsoleConfig console;
    LogFileConfig file;
};

struct AppConfig {
    ServerConfig server;    // 기존 구조체에 drain_timeout 추가
    LogConfig logging;

    static AppConfig from_file(const std::string& path);
    static AppConfig defaults();
};

} // namespace apex::core
```

### 2.3 ServerConfig 변경

기존 `ServerConfig`에 1개 필드 추가:

```cpp
struct ServerConfig {
    // ... 기존 필드 ...
    std::chrono::seconds drain_timeout{25};  // 추가
};
```

### 2.4 사용 패턴

```cpp
int main() {
    auto config = AppConfig::from_file("config/default.toml");
    apex::core::init_logging(config.logging);
    Server(config.server)
        .add_service<EchoService>()
        .run();
}
```

`AppConfig::defaults()`로 TOML 파일 없이 기본값 생성 가능 (테스트, 기존 예제 호환).

## 3. spdlog 통합

### 3.1 로거 구조

| 로거 이름 | 용도 | 기본 레벨 |
|-----------|------|-----------|
| `"apex"` | 프레임워크 내부 (Server, CoreEngine, TcpAcceptor 등) | info |
| `"app"` | 서비스 코드 (ServiceBase 파생 클래스) | info |

두 로거는 동일한 Sink 세트를 공유하되, 레벨만 독립 제어.

### 3.2 초기화 / 종료 API

```cpp
// include/apex/core/logging.hpp
namespace apex::core {

void init_logging(const LogConfig& config);
void shutdown_logging();

} // namespace apex::core
```

`init_logging()`에서 Sink 생성 → 로거 2개 등록 → 레벨 설정.
`shutdown_logging()`은 `spdlog::shutdown()` 래핑.

### 3.3 Sink 구성

- **ConsoleSink**: 텍스트 포맷 (패턴 기반, 사람 가독성)
- **FileSink**: JSON 포맷 (커스텀 포매터, 머신 파싱)
  - rotating_file_sink (max_size_mb, max_files)

### 3.4 JSON 포맷 (FileSink)

```json
{"ts":"2026-03-08T14:23:01.123","level":"info","logger":"apex","msg":"Server started on port 9000"}
```

`spdlog::custom_flag_formatter` 상속으로 JSON 이스케이핑 보장.
Phase 6에서 `trace_id` 등 커스텀 필드 추가 시 동일 패턴으로 확장.

### 3.5 프레임워크 내부 로깅 원칙

- 핫패스(read_loop, process_frames)에는 로그 넣지 않음
- 라이프사이클 이벤트(start, stop, error)에만 로그 출력
- `spdlog::get("apex")` 호출은 초기화 시 로컬 캐싱 가능 (registry 뮤텍스 회피)
- Phase 5에서는 동기 모드 사용 (비동기는 KafkaSink Phase 6부터)

## 4. Graceful Shutdown 타임아웃

### 4.1 변경 포인트

`Server::poll_shutdown()`의 무한 폴링에 deadline 기반 타임아웃 추가.

### 4.2 멤버 추가

```cpp
// server.hpp
std::chrono::steady_clock::time_point shutdown_deadline_;
```

### 4.3 종료 흐름

```
stop()
  → acceptor 중지
  → begin_shutdown()
      → 모든 세션 close()
      → shutdown_deadline_ = now + drain_timeout
      → poll_shutdown()
          ├─ active_sessions == 0 → info 로그 + 정상 종료
          └─ now >= deadline → warn 로그 + 강제 종료
              → core_engine_->stop() → join() → drain_remaining()
              → 서비스 stop() → accept_io_.stop()
```

steady_clock 사용 (NTP 보정 면역).
타임아웃 후에도 종료 시퀀스는 동일 (io_context stop으로 코루틴 자연 소멸).

### 5.2 트리거

```yaml
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
```

1. checkout
2. vcpkg 캐시 복원 (actions/cache, 키: OS + vcpkg.json 해시)

5. cmake build
6. ctest 실행

## 6. 테스트 전략

### 6.1 새 테스트

| 대상 | 테스트 내용 | 종류 |
|------|-----------|------|
| AppConfig | TOML 정상 파싱 / 파일 없음 예외 / 필드 누락 기본값 / 타입 불일치 에러 | 단위 |
| init_logging | ConsoleSink만 / FileSink만 / 둘 다 / 레벨 독립 제어 | 단위 |
| drain_timeout | 타임아웃 내 정상 종료 / 타임아웃 만료 강제 종료 | 통합 |

### 6.2 에러 처리

| 상황 | 동작 |
|------|------|
| TOML 파일 없음 | `std::runtime_error` throw |
| TOML 파싱 에러 | `std::runtime_error` + toml++ 에러 메시지 |
| 필드 타입 불일치 | `std::invalid_argument` |
| 필드 누락 | 기본값 사용 (에러 아님) |
| 알 수 없는 섹션/키 | 무시 (forward compatibility) |
| 로그 파일 경로 접근 불가 | stderr 출력, 서버 정상 구동 |

콜드패스 예외 전략은 기존 프레임워크 패턴과 일관.
