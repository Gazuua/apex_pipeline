## 목표

현재 Server는 단일 io_context 기반으로 동작한다.
CoreEngine(io_context-per-core)과 통합하여 멀티코어 shared-nothing 아키텍처를 실현하고,
코어 간 요청-응답 패턴(cross_core_call)을 구현한다.

## 현재 상태

| 컴포넌트 | 상태 | 비고 |
|----------|------|------|
| CoreEngine | 완성 | io_context-per-core, MPSC drain, post_to/broadcast |
| MpscQueue | 완성 | lock-free bounded MPSC, trivially_copyable 제약 |
| Server | 완성 (싱글코어) | `Server(io_context&, Config)` — 단일 이벤트 루프 |
| cross_core_call | 미구현 | ADR-20 설계만 확정 |

**핵심 간극:** Server가 CoreEngine을 include조차 하지 않음. 완전히 분리된 상태.

## INT-1: Server-CoreEngine 통합

### 설계 방향

**단일 Server가 CoreEngine을 소유하고, 코어별 리소스를 관리한다.**

기존 `Server(io_context&)` 시그니처를 유지하면서 멀티코어 모드를 추가한다.
싱글코어 모드(기존)와 멀티코어 모드가 공존하여 하위 호환성을 보장한다.

### ServerConfig 확장

```cpp
struct ServerConfig {
    uint16_t port = 9000;
    uint32_t heartbeat_timeout_ticks = 300;
    uint32_t tick_interval_ms = 100;
    size_t recv_buf_capacity = 8192;
    size_t timer_wheel_slots = 1024;

    // --- 멀티코어 설정 (v0.2.4 추가) ---
    uint32_t num_cores = 0;                 // 0 = 싱글코어 모드 (기존 호환)
    size_t mpsc_queue_capacity = 65536;
    std::chrono::microseconds drain_interval{100};
};
```

### 생성자 전략

```cpp
// 기존 (싱글코어) — 하위 호환
Server(io_context& io_ctx, Config config = {});

// 멀티코어 — CoreEngine 내부 생성
// num_cores > 0이면 io_ctx는 accept 전용, 코어별 io_context는 CoreEngine이 소유
```

`num_cores == 0`: 기존과 동일 (싱글코어, 외부 io_context 사용)
`num_cores > 0`: CoreEngine 내부 생성, 코어별 리소스 분리

### 코어별 리소스 분리

```
Server (num_cores=4)
├── CoreEngine (4 cores)
│   ├── core[0]: io_context + SessionManager + drain_timer
│   ├── core[1]: io_context + SessionManager + drain_timer
│   ├── core[2]: io_context + SessionManager + drain_timer
│   └── core[3]: io_context + SessionManager + drain_timer
├── MessageDispatcher (공유, read-only after setup)
├── TcpAcceptor (메인 io_context, accept 후 라운드로빈 분배)
└── Services (공유, 핸들러 등록만)
```

**결정 사항:**
- **SessionManager**: 코어별 독립 인스턴스 (세션은 코어에 고정)
- **MessageDispatcher**: 공유 (핸들러 테이블은 start() 이후 read-only)
- **TcpAcceptor**: 단일 (메인 io_context), accept 후 라운드로빈으로 코어 분배
- **TimingWheel**: SessionManager 내부이므로 코어별 독립

### 코어별 구조체

```cpp
/// 코어별 리소스 번들 (Server 내부)
struct PerCoreState {
    uint32_t core_id;
    SessionManager session_mgr;      // 이 코어의 세션만 관리
    // 향후 확장: per-core 메트릭, 메모리 풀 등
};
```

### Accept 분배 전략

```
TcpAcceptor (메인 io_context)
  → on_accept(socket)
    → core_id = next_core_.fetch_add(1) % num_cores  (라운드로빈)
    → socket을 target core의 io_context로 이관
    → target core에서 세션 생성 + read_loop 시작
```

소켓 이관은 `boost::asio::ip::tcp::socket::operator=(socket&&)` 후
대상 코어의 io_context에서 `co_spawn`으로 read_loop 시작.

구현 핵심:
```cpp
void Server::on_accept(tcp::socket socket) {
    auto core_id = next_core_.fetch_add(1, std::memory_order_relaxed) % core_count();
    auto& target_io = core_engine_->io_context(core_id);

    // 대상 코어의 io_context로 소켓 이관 + read_loop 시작
    boost::asio::co_spawn(target_io,
        [this, core_id, s = std::move(socket)]() mutable -> awaitable<void> {
            auto session = per_core_[core_id].session_mgr.create_session(std::move(s));
            co_await read_loop(session, core_id);
        },
        boost::asio::detached);
}
```

### read_loop / process_frames 변경

기존 시그니처에 `core_id` 파라미터 추가 (멀티코어 모드에서만 사용):

```cpp
awaitable<void> read_loop(SessionPtr session);                    // 싱글코어 (기존)
awaitable<void> read_loop(SessionPtr session, uint32_t core_id);  // 멀티코어
```

`process_frames`에서 `session_mgr_` 대신 `per_core_[core_id].session_mgr` 사용.

### start() / stop() 변경

```cpp
void Server::start() {
    // ... 서비스 시작, 핸들러 등록 (기존)

    if (multi_core_mode()) {
        core_engine_->set_message_handler([this](uint32_t cid, const CoreMessage& msg) {
            handle_core_message(cid, msg);
        });
        // CoreEngine::run()은 blocking — 별도 스레드 또는 내부 관리
        // TcpAcceptor는 메인 io_context에서 동작
    }
    acceptor_.start();
    start_tick_timer();
}

void Server::stop() {
    running_.store(false);
    acceptor_.stop();
    if (core_engine_) core_engine_->stop();
    // 코어별 세션 정리
    for (auto& pc : per_core_) {
        pc.session_mgr.close_all();
    }
}
```

### Server 멤버 변경

```cpp
class Server {
    // ... 기존 멤버 유지 ...

    // v0.2.4 멀티코어 추가
    std::unique_ptr<CoreEngine> core_engine_;              // null = 싱글코어
    std::vector<PerCoreState> per_core_;                   // 코어별 상태
    std::atomic<uint32_t> next_core_{0};                   // 라운드로빈 카운터

    [[nodiscard]] bool multi_core_mode() const noexcept {
        return core_engine_ != nullptr;
    }
    uint32_t core_count() const noexcept;
    void handle_core_message(uint32_t core_id, const CoreMessage& msg);
};
```

## INT-2: cross_core_call

### 설계 (ADR-20 기반)

Seastar `smp::submit_to(core, lambda) -> future<T>` 패턴을 Boost.Asio 코루틴으로 적용.

### API

```cpp
/// 대상 코어에서 callable을 실행하고 결과를 받아온다.
/// 호출 코어의 코루틴이 suspend되고, 대상 코어에서 실행 후 resume된다.
template <typename Func>
auto cross_core_call(uint32_t target_core, Func&& func)
    -> awaitable<std::invoke_result_t<Func>>;

/// fire-and-forget (응답 불필요)
template <typename Func>
bool cross_core_post(uint32_t target_core, Func&& func);
```

### 내부 메커니즘

```
호출 코어 (core 0)                    대상 코어 (core 2)
───────────────────                   ───────────────────
1. 요청 ID 발번 (atomic counter)
2. pending_map[req_id] = this_coro
3. CoreMessage{req_id, func_ptr}
   → post_to(core 2, msg)
4. co_await suspend                   5. drain_timer에서 msg 수신
                                      6. func() 실행, 결과 저장
                                      7. CoreMessage{req_id, result_ptr}
                                         → post_to(core 0, response)
8. drain에서 response 수신
9. pending_map[req_id] → resume
10. co_return result
```

### 핵심 구조체

```cpp
/// 타입 소거된 요청/응답 래퍼
struct CrossCoreRequest {
    uint64_t request_id;
    std::function<void()> execute;    // 대상 코어에서 실행
    // 결과는 shared_state에 저장
};

/// 요청-응답 공유 상태
template <typename T>
struct CrossCoreState {
    std::atomic<bool> ready{false};
    T result;                         // 또는 std::optional<T>
    boost::asio::io_context* caller_io_ctx;  // resume 대상
    // awaitable resume 메커니즘
};
```

### CoreMessage 활용

현재 `CoreMessage.data`는 `uint64_t` — 포인터를 담기에 충분하다.
cross_core_call 전용 CoreMessage::Type 추가:

```cpp
enum class Type : uint8_t {
    Shutdown = 0,
    DrainComplete,
    Custom,
    CrossCoreRequest,   // v0.2.4 추가
    CrossCoreResponse,  // v0.2.4 추가
};
```

`data` 필드에 `CrossCoreRequest*` 또는 `CrossCoreState*`를 `reinterpret_cast<uint64_t>`로 전달.

**수명 관리:** shared_ptr로 CrossCoreState를 관리하여 요청/응답 양쪽 코어에서 안전하게 접근.
요청 코어가 타임아웃으로 먼저 포기해도 대상 코어의 응답이 dangling되지 않도록 한다.

### 타임아웃 처리

```cpp
template <typename Func>
auto cross_core_call(uint32_t target_core, Func&& func,
                     std::chrono::milliseconds timeout = 5000ms)
    -> awaitable<Result<std::invoke_result_t<Func>>>;
```

- 타임아웃 시 `Result<T>` → `ErrorCode::CrossCoreTimeout` 반환
- TimingWheel 또는 steady_timer로 타임아웃 감시

## INT-3: 멀티코어 예제 + 테스트

### 예제: multicore_echo_server

```cpp
// examples/multicore_echo_server.cpp
int main() {
    boost::asio::io_context accept_io;
    Server server(accept_io, {
        .port = 9000,
        .num_cores = 4,           // 4코어 멀티코어 모드
    });
    server.add_service<EchoService>();
    server.start();
    accept_io.run();              // accept loop만 — 코어 스레드는 CoreEngine이 관리
}
```

| 테스트 | 검증 |
|--------|------|
| `MultiCoreServerStartStop` | num_cores=2, start/stop 정상 동작 |

| `MultiCoreSessionIsolation` | 코어 0의 세션은 코어 1의 SessionManager에 없음 |
| `CrossCoreCallBasic` | core 0 → core 1 람다 실행 + 결과 수신 |
| `CrossCoreCallTimeout` | 응답 지연 → ErrorCode::CrossCoreTimeout |

## 설계 결정 포인트 (구현 시 확정 필요)

다음 항목은 구현 시 트레이드오프를 고려하여 결정한다:

### D-1: Server::run() 블로킹 전략
- **Option A**: `server.start()` 후 `accept_io.run()` + CoreEngine 내부 스레드
  - 장점: 기존 패턴 유지, accept만 메인 스레드
  - 단점: CoreEngine::run()이 blocking이라 별도 관리 필요
- **Option B**: `server.run()`이 모든 것을 blocking으로 관리
  - 장점: 사용자 코드 단순
  - 단점: 기존 패턴과 다름

### D-2: cross_core_call 타입 소거 방식
- **Option A**: `std::function` + `shared_ptr<void>` (단순, 약간의 힙 할당)
- **Option B**: 커스텀 타입 소거 + SlabPool 할당 (고성능, 복잡)
- 초기 구현은 **Option A**로 시작, 벤치마크 결과에 따라 최적화

### D-3: 멀티코어 브로드캐스트
- 코어 0의 세션이 chat broadcast → 다른 코어 세션들에게 전달 방법
- **Option A**: cross_core_post로 각 코어에 전송 함수 실행 요청
- **Option B**: 공유 broadcast 큐 (lock-based)
- 권장: **Option A** (shared-nothing 철학 유지)
