## Critical (반드시 수정 필요) - 8건

### 1. MpscQueue: Full 판단 로직 결함
- **파일**: `core/include/apex/core/mpsc_queue.hpp:98-105`
- **복수 에이전트 지적**
- `slot.ready` 상태로 Full을 판정하는데, 다른 producer가 CAS 성공 후 아직 `ready=true`를 세팅하기 전이면 false-full이 발생함
- `head - tail >= capacity` 비교가 정확하지만 `tail_`이 non-atomic이라 producer에서 읽을 수 없는 구조적 한계

### 2. TimingWheel: 콜백 중 use-after-free
- **파일**: `core/src/timing_wheel.cpp:99-109`
- `on_expire_` 콜백 내부에서 같은 슬롯의 다른 타이머를 `cancel()`하면, 미리 저장해둔 `next` 포인터가 dangling
- 수정안: 콜백 호출 전에 만료 대상을 로컬 리스트로 수집한 후 일괄 호출

### 3. TimingWheel: schedule() + reschedule() off-by-one
- **파일**: `core/src/timing_wheel.cpp:49-54, 88-90`
- `ticks_from_now=1`일 때 `deadline_tick = current_tick_ + 1 - 1 = current_tick_`이 되어 즉시 실행됨
- **`reschedule()`에도 동일한 `- 1` 로직이 복제됨** (LSP 추가 발견)
- 수정안: 두 곳 모두 `- 1` 제거 또는 private 헬퍼로 추출

### 4. TimingWheel: EntryId 재사용 없음 -> 무한 메모리 성장
- **파일**: `core/src/timing_wheel.cpp:57-59`
- **복수 에이전트 지적**
- `next_id_`가 단조 증가하면서 `entries_` 벡터가 무한 성장. 장시간 운영 서버에서 OOM 위험
- free-list 기반 id 재사용 필요

### 5. ServiceBase: `name_`이 `string_view` -> 댕글링 위험
- **파일**: `core/include/apex/core/service_base.hpp:27, 64`
- **복수 에이전트 지적**
- 임시 `std::string`을 넘기면 즉시 dangling
- `std::string`으로 저장하고 `string_view`로 반환하는 패턴으로 변경

### 6. CoreEngine: `run()` 재호출 시 `io_ctx.restart()` 누락
- **파일**: `core/src/core_engine.cpp:63`
- `run()` 종료 후 재호출하면 `io_ctx`가 이미 stopped 상태라 즉시 리턴
- `io_ctx.restart()` 호출 또는 재사용 방어 필요

### 7. Frame.payload 수명 관리 부재
- **파일**: `core/include/apex/core/frame_codec.hpp:15`, `core/src/ring_buffer.cpp:77-84`
- `Frame::payload`가 `RingBuffer`의 내부 `linear_buf_`를 가리키는 `span`인데, `consume_frame` 후 접근하면 UAF
- `linearize()` 재호출 시에도 이전 span이 무효화됨. API 계약이 어디에도 명시되어 있지 않음

### 8. WireHeader: version 검증 누락
- **파일**: `core/src/wire_header.cpp:29`
- **3개 에이전트 지적**
- `ParseError::UnsupportedVersion` 에러 타입까지 선언해놓고 `parse()`에서 실제 검증을 안 함
- 2줄 수정으로 끝나는 가장 쉬운 Critical 이슈

## 긍정적 평가

- WireHeader의 big-endian 변환: `memcpy` 기반으로 strict aliasing 규칙 올바르게 준수
- `std::expected` 일관 사용: 에러 처리 흐름이 명확하고 현대적
- MessageDispatcher: `std::array<Handler, 65536>` O(1) 디스패치 - 가장 깔끔한 컴포넌트
- RingBuffer 설계: zero-copy linearize 컨셉 자체는 훌륭
- MAX_BODY_SIZE 16MB 제한: 기본적인 DoS 방어 존재
- CRTP ServiceBase 패턴: handle()/route\<T\>()의 타입 추론에 효과적 활용
- C++23 features(expected, span, string_view) 적극 활용
- cache-line alignment에 대한 의식적 적용 (head\_/tail\_ 분리는 올바름)
