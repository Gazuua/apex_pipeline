# CI Auto-Tag 설계

## 개요

main CI 전체 통과 시 git tag를 자동 생성하여, "이 커밋은 모든 검증을 통과한 상태"임을 보증하는 메커니즘.

## 버전 체계

```
v[메이저].[대].[중].[소]
 └── VERSION 파일 ──┘  └── CI 자동 ──┘
```

- **상위 3자리** (`메이저.대.중`): 루트 `VERSION` 파일에 기록. 마일스톤 시 수동 변경.
- **4번째 자리** (`소`): CI가 기존 git tag를 세서 자동 +1.
- 예: VERSION=`0.6.4`, 최신 태그=`v0.6.4.5` → 다음 태그=`v0.6.4.6`

## 구성 요소

### 1. VERSION 파일 (신규)

- 위치: 프로젝트 루트 `VERSION`
- 내용: base 버전 3자리 (예: `0.6.4`). 개행 외 공백 없음.
- 변경 시점: 마일스톤 달성 시 수동 커밋.
- 변경 시 4번째 자리는 0부터 재시작.

### 2. release-tag job (ci.yml 추가)

deploy job 이후 실행되는 CI job.

**실행 조건:**
- `github.ref == 'refs/heads/main'`
- `needs.changes.outputs.source == 'true'` (소스 코드 변경 있음)
- `needs.deploy.result == 'success'`

**동작:**
1. checkout (fetch-depth: 0 — 태그 히스토리 필요)
2. `VERSION` 파일 읽기 → base 버전 (예: `0.6.4`)
3. `git tag -l 'v{base}.*'`로 해당 base의 태그 목록 조회
4. 최대 small 번호 파싱 → +1. 태그 없으면 0부터 시작.
5. `git tag v{base}.{next}` → `git push origin v{base}.{next}`

**권한:** `permissions: contents: write`

**의존성 체인:**
```
build → bake-services → smoke-test ─┐
                      → helm-validation ─┤
                                         → deploy → release-tag
```

### 3. 태그 생성 조건 정리

| 시나리오 | 태그 생성 |
|----------|----------|
| 소스 변경 + 전체 CI 통과 | O |
| 소스 변경 + 어딘가 실패 | X |
| 문서만 변경 | X (source filter 미통과) |
| Go만 변경 | X (source filter 미통과) |
| deploy skipped (upstream 실패) | X |

## 범위 외 (명시적 제외)

- **vcpkg.json 버전 동기화**: apex-pipeline을 vcpkg 패키지로 배포하지 않으므로 불필요
- **GitHub Release 생성**: small 태그마다 릴리스는 소음. mid 이상에서 수동 생성.
- **문서 자동 갱신**: small 변경마다 불필요. mid 이상에서 기존 방식대로 에이전트가 갱신.

## 첫 적용

현재 git 태그 0개. VERSION 파일에 `0.6.4` 기록 후 첫 소스 변경 머지 시 `v0.6.4.0` 태그가 생성됨.
