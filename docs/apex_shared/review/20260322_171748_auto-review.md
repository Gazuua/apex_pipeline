## HIGH 이슈 (수정 완료)

### [S1] AdapterBase::close() — remaining 스택 변수 UAF
- **위치**: `adapter_base.hpp`, close() 내 spin-wait
- **문제**: `remaining`이 스택 지역 변수인데 타임아웃 후 반환하면 미실행 핸들러가 파괴된 스택에 atomic operation 수행
- **수정**: `std::shared_ptr<std::atomic<uint32_t>>`로 변경하여 핸들러 수명 보장
- **커밋**: `3927e6b`

### [L1] RedisMultiplexer::close() — backoff_timer_.cancel() 누락
- **위치**: `redis_multiplexer.cpp`, close()
- **문제**: reconnect_loop가 backoff_timer에서 대기 중일 때 close()가 타이머를 취소하지 않아 소멸 후 dangling reference
- **수정**: `backoff_timer_.cancel()` 추가
- **커밋**: `3927e6b`

### [L2] RedisMultiplexer 소멸자 — cancel_all_pending의 코루틴 resume UAF
- **위치**: `redis_multiplexer.cpp`, ~RedisMultiplexer()
- **문제**: `cancel_all_pending`의 `resolver.cancel()`이 코루틴 resume을 post하는데, this 파괴 후 resume 시 slab_ 접근 UAF
- **수정**: 소멸자에서는 `cancel_all_pending` 대신 직접 `slab_.destroy`로 해제
- **커밋**: `3927e6b`

### [L3] on_disconnect() — spawn_callback_ 거부 시 reconnect_active_ 잔류
- **위치**: `redis_multiplexer.cpp`, on_disconnect()
- **문제**: DRAINING 상태에서 spawn_callback_이 거부되면 ReconnectGuard가 미실행되어 reconnect_active_가 영구 true
- **수정**: DRAINING 이후 destroy 흐름이므로 실질적 영향 없음 확인. 주석 문서화
