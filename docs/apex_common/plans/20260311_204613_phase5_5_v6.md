## 1. 핫패스 아키텍처 개선 (최우선)

### 1.1 drain/tick 분리 — 이중 역할 해소

**[출처: 사용자 2.2.5 + 토론 합의]**

**현재 문제**:
- CoreEngine의 `drain_timer`가 두 가지 역할을 동시에 수행:
  1. MPSC inbox 소비 (cross-core 메시지 처리)
  2. `drain_callback` → `SessionManager::tick()` (하트비트 타이머 진행)
- 100μs 간격 설정이지만 실제 OS 타이머 해상도: Linux ~1-4ms, Windows ~15ms → 무의미
- 메시지 없어도 CPU 소비 (busy polling)
- cross-core 메시지의 실질 지연이 타이머 해상도에 종속

**해결책**: drain_timer의 이중 역할을 두 개의 독립 메커니즘으로 완전 분리

#### (A) 메시지 drain — `post()` + atomic coalescing flag + batch limit

```cpp
// CoreEngine::post_to() 내부 — enqueue 성공 후 즉시 알림
auto result = queue.enqueue(msg);
if (!result.has_value()) {
    return result;  // enqueue 실패 시 drain 알림 발행하지 않음
}
if (!drain_pending_[target_core].exchange(true, std::memory_order_acq_rel)) {
    boost::asio::post(cores_[target_core].io_ctx, [this, target_core] {
        drain_pending_[target_core].store(false, std::memory_order_release);
        drain_inbox(target_core);
    });
}
```

**[v5] batch-limited drain — starvation 방지**:

게임 서버에서 cross-core 메시징이 핫패스가 되는 시나리오 (존 위치 동기화, 브로드캐스트,
매치메이킹 등)에서 drain_inbox가 수천 건을 한 번에 처리하면 같은 코어의 read_loop, tick,
async_send가 밀릴 수 있다. batch 제한 + 재스케줄로 해결:

```cpp
void CoreEngine::drain_inbox(uint32_t core_id) {
    auto& core_ctx = *cores_[core_id];
    size_t processed = 0;

    while (processed < config_.drain_batch_limit) {
        auto msg = core_ctx.inbox->dequeue();
        if (!msg) break;
        // ... 메시지 처리 로직 ...
        ++processed;
    }

    // 큐에 남은 메시지가 있으면 즉시 재스케줄
    if (processed == config_.drain_batch_limit) {
        boost::asio::post(core_ctx.io_ctx, [this, core_id] {
            drain_inbox(core_id);
        });
    }
}
```

- batch 후 io_context에 양보 → read_loop, tick 등 실행 기회 보장
- 재스케줄이 `post()`이므로 io_context 큐 뒤에 붙음 → fair scheduling
- coalescing flag와 독립 동작 (재스케줄은 flag 무관하게 직접 post)

**CoreEngineConfig 추가**:
```cpp
struct CoreEngineConfig {
    uint32_t num_cores{0};
    size_t mpsc_queue_capacity{65536};
    std::chrono::milliseconds tick_interval{100};
    size_t drain_batch_limit{1024};  // [v5] 1회 drain 최대 메시지 수
};
```
기본값 1024는 보수적 시작점. 벤치마크에서 조절 예정.

- `post()`는 내부적으로 Linux에서 eventfd, Windows에서 IOCP를 사용 → 크로스플랫폼
- atomic coalescing으로 연속 enqueue 시 post() 폭주 방지
- MPSC 큐는 데이터 전달 + backpressure 용도로 유지, post()는 알림 신호만
- **폴링 지연 100μs+ → ~0μs**

**[v4] enqueue 실패 경로 명확화**:
v3까지의 코드에서 `queue.enqueue(msg)` 후 바로 drain 알림을 발행했으나,
enqueue가 실패(큐 full)할 수 있다는 분기가 누락되어 있었다.
수정된 코드에서는 enqueue 결과를 먼저 확인한 후에만 drain 알림을 발행한다.

또한 `cross_core_post` (cross_core_call.hpp 184~194행)에서도 `post_to` 실패 시
태스크를 delete하고 false를 반환하는 현재 패턴이, 이 변경 후에도 동일하게 동작함을 확인.

**[v2] re-check 패턴 분석 결과**:
`store(false)` → `drain_inbox()` 순서이므로 메시지 유실 가능성은 없다.
별도의 re-check 루프 불필요. 원래 패턴 그대로 유지.

**[v3] `drain_inbox()` 구현 시 기존 로직 이전 필수**:
현재 `start_drain_timer` 핸들러 내부의 `CrossCoreRequest`/`CrossCorePost` 타입별
분기 처리 + exception handling이 새로운 `drain_inbox(target_core)`에 반드시 이전되어야 한다.
[v5] 단, message passing 전환(1.6절) 후에는 CrossCoreOp 기반 handler dispatch로 대체.

#### (B) core tick — per-core 범용 주기 타이머
```cpp
void CoreEngine::start_tick_timer(uint32_t core_id) {
    auto& ctx = *cores_[core_id];
    ctx.tick_timer->expires_after(config_.tick_interval);
    ctx.tick_timer->async_wait([this, core_id](const boost::system::error_code& ec) {
        if (ec) return;
        if (tick_callback_) tick_callback_(core_id);
        if (running_.load(std::memory_order_acquire))
            start_tick_timer(core_id);
    });
}
```


### 1.6 Cross-Core Message Passing 아키텍처 (신규)

**[v5] 출처: 사용자-Claude 브레인스토밍 합의**

#### 1.6.1 현재 문제: Closure Shipping

현재 `cross_core_post`와 `cross_core_call`은 **closure shipping** 방식:
```cpp
// cross_core_call.hpp 184-185행
auto* task = new std::function<void()>(std::forward<F>(func));
```

**세 가지 근본 문제**:
1. `new std::function` per-message heap 할당 (SBO 초과 시 추가 할당)
2. closure가 코어 경계를 넘으며 캐시라인 오염
3. 풀링으로 해결하려 하면 반환 경로 필요 → shared-nothing 원칙 위반

#### 1.6.2 해결: Code와 Data 분리

```
Before: new std::function{ [code + captured_data] } → 포인터 전달
After:  사전 등록 handler(code) + immutable shared data(move) → op 기반 dispatch
```

외부 메시지의 `MessageDispatcher`(msg_id → handler)와 동일한 패턴을 내부 cross-core
통신에도 적용. 프레임워크 안에서 이미 검증된 패턴의 내부 확장.

#### 1.6.3 CoreMessage 구조

```cpp
struct CoreMessage {
    CrossCoreOp op{};         // handler dispatch key (uint16_t)
    uint32_t source_core{0};
    uintptr_t data{0};        // → SharedPayload* (immutable, refcounted)
};
static_assert(std::is_trivially_copyable_v<CoreMessage>);
// ~16B, 기존과 동일 크기. MPSC 큐 메모리 변화 없음.
```

**inline storage를 추가하지 않는 이유**:
- 모든 cross-core data를 immutable shared buffer로 통일 (크기 분기 제거)
- CoreMessage를 16B로 유지 → MPSC 슬롯 크기 불변, 캐시 효율 유지
- 48B 미만 페이로드도 shared buffer로 처리 (refcount atomic은 per-core 횟수라 무시 가능)

#### 1.6.4 Handler 사전 등록

```cpp
// CrossCoreOp 정의
enum class CrossCoreOp : uint16_t {
    Shutdown = 0,
    BroadcastToSessions,
    CollectMetrics,
    FindSessionRequest,
    FindSessionReply,
    // 서비스가 추가 등록 가능
};

// handler는 function pointer — static dispatch, icache friendly
using CrossCoreHandler = void(*)(CoreContext& ctx, uint32_t source_core, void* data);

// 서버 시작 시 1회 등록
engine.register_cross_core_handler(CrossCoreOp::BroadcastToSessions,
    [](CoreContext& ctx, uint32_t src, void* raw) {
        auto* req = static_cast<BroadcastRequest*>(raw);
        // ... 처리 ...
    });
```

#### 1.6.5 Immutable Shared Payload

**모든 cross-core data는 immutable shared buffer로 전달.**

```cpp
struct SharedPayload {
    std::atomic<uint32_t> refcount{0};
    // 서브클래스 또는 type-erased data
};

// 단일 대상 (point-to-point)
auto* payload = new ConcretePayload{...};
payload->refcount.store(1, std::memory_order_relaxed);
engine.post_cross_core(target, Op::Xxx, payload);
// target이 처리 후 release → refcount 0 → delete

// 브로드캐스트 (N개 코어)
auto* payload = new BroadcastPayload{...};
payload->refcount.store(N - 1, std::memory_order_relaxed);  // 자기 코어 제외
for (uint32_t core = 0; core < num_cores; ++core) {
    if (core == my_core) continue;
    engine.post_cross_core(core, Op::Broadcast, payload);
}
// 마지막 코어가 release → refcount 0 → delete
```

**소유권 모델: 메시지 자기 자신이 독립적으로 소유 (refcount 기반 자기 소유)**
- 호출자 세션이 종료돼도 payload는 유효 (fire-and-forget)
- 호출자는 broadcast 호출 시점에 소유권을 완전히 이전, 자기 ref 미보유
- 마지막 코어가 처리 완료하면 refcount → 0 → 자동 소멸
- 네트워크 패킷과 동일한 모델: 한번 보낸 패킷은 발신자가 죽어도 유효

**immutable이 보장하는 것**:
- MESI 프로토콜에서 모든 코어가 Shared(S) state로 동시 보유
- 캐시라인 바운싱 없음 (Invalidate 0회)
- atomic refcount만 바운싱 발생하지만 per-core 횟수 (per-session 아님)

**성능 분석 (1MB payload, 48코어 브로드캐스트)**:

| 항목 | per-core 복사 | immutable shared |
|------|-------------|-----------------|
| memcpy | 47 × 1MB = 47MB, ~3.1ms | 1회 1MB, ~0.07ms |
| 메모리 | 48MB | 1MB |
| atomic | 0 | 95회 × ~20ns ≈ 2μs |
| L3 fetch | 불필요 (로컬) | 48코어 병렬 fetch, ~330μs |
| **총 비용** | **~3.1ms + 48MB** | **~0.4ms + 1MB** |

small payload(48B 이하)에서도 immutable shared의 atomic 비용(~2μs)이
MPSC enqueue의 CAS 비용과 비슷한 수준이므로, 크기 분기 없이 통일해도
성능 손실이 무시 가능.

#### 1.6.6 cross_core_post 전환

**Before**:
```cpp
// closure shipping — per-message heap alloc
auto* task = new std::function<void()>(std::forward<F>(func));
CoreMessage msg{.type = CrossCorePost, .data = reinterpret_cast<uint64_t>(task)};
```

**After**:
```cpp
// message passing — data만 전달, handler는 사전 등록
auto* payload = new ConcretePayload{...};
payload->refcount.store(1);
CoreMessage msg{.op = CrossCoreOp::Xxx, .data = reinterpret_cast<uintptr_t>(payload)};
engine.post_to(target, msg);
```

- std::function 제거 → heap 할당 감소, virtual dispatch 제거
- handler는 function pointer → icache friendly
- 소유권 move → 반환 경로 불필요

#### 1.6.7 cross_core_call → request-response message passing 전환

**현재 문제**:
- `cross_core_call.hpp`: CAS + shared_ptr<State> + atomic status + timer race
- ~25줄의 race condition 분석 주석이 복잡도를 증명 [v6: 실측 수정]
- per-call 3회 heap 할당 (State, timer, task)

**해결: 같은 코어의 io_context는 single-threaded**

요청 코어에서 response handler와 timeout timer callback이 **같은 io_context에서
직렬 실행**. CAS도, shared State도, atomic도 필요 없음.

```
Core A                              Core B
──────                              ──────
① pending_map[req_id] = callback
② post_cross_core(B, FindSession, {session_id, req_id, reply_core=A})
③ timer.expires_after(timeout)
④ co_await timer
                                    ⑤ handler: find_session(id)
                                    ⑥ post_cross_core(A, FindSessionReply, {req_id, result})
⑦ drain_inbox → response handler
   pending_map[req_id] callback 실행
   → result 저장 + timer.cancel()
⑧ co_await 재개, result 반환
```

**레이스가 없는 이유**:
- 경우 1 (응답 먼저): drain에서 callback 실행, timer cancel → timer callback은 ec==cancelled, 무시
- 경우 2 (timeout 먼저): timer callback에서 pending_map.erase, timeout 반환 → 나중에 응답 도착 시 pending_map에 없음, 무시
- 같은 스레드이므로 동시 실행 자체가 불가능

```cpp
// per-core 로컬 — thread safety 불필요
struct PendingRequest {
    std::function<void(void*)> callback;
    boost::asio::steady_timer* timer;
};
boost::unordered_flat_map<uint64_t, PendingRequest> pending_requests_;  // core-local

// cross_core_request — cross_core_call의 대체
template <typename Result, typename Data>
auto cross_core_request(CoreEngine& engine, uint32_t target,
                        CrossCoreOp op, Data&& data,
                        milliseconds timeout = 5000ms)
    -> awaitable<Result<Result>>
{
    auto executor = co_await this_coro::executor;
    boost::asio::steady_timer timer(executor);
    timer.expires_after(timeout);

    uint64_t req_id = engine.next_request_id();
    std::optional<Result> result;

    // core-local map — 동일 스레드에서만 접근, lock 불필요
    engine.register_pending(req_id, {
        .callback = [&result, &timer](void* raw) {
            result = std::move(*static_cast<Result*>(raw));
            timer.cancel();
        },
        .timer = &timer
    });

    // 요청 전송
    data.request_id = req_id;
    data.reply_core = engine.current_core();
    engine.post_cross_core(target, op, std::move(data));

    // 응답 또는 timeout 대기
    auto [ec] = co_await timer.async_wait(as_tuple(use_awaitable));
    engine.remove_pending(req_id);

    if (!result) co_return error(ErrorCode::CrossCoreTimeout);
    co_return std::move(*result);
}
```

**비교**:

| | 현재 cross_core_call | message passing |
|---|---|---|
| heap 할당 | 3회 (State, timer, task) | **0회** (shared payload) |
| 동기화 | atomic CAS + shared_ptr | **없음** (single-thread 직렬화) |
| 타임아웃 | 크로스스레드 timer cancel race | **로컬 timer, race 없음** |
| 코드 복잡도 | CAS race 분석 주석 ~25줄 | **"같은 스레드" 한 문장** |

#### 1.6.8 drain_inbox에서의 handler dispatch

```cpp
void CoreEngine::drain_inbox(uint32_t core_id) {
    auto& core_ctx = *cores_[core_id];
    size_t processed = 0;

    while (processed < config_.drain_batch_limit) {
        auto msg = core_ctx.inbox->dequeue();
        if (!msg) break;

        // CrossCoreOp 기반 handler dispatch
        auto handler = cross_core_dispatcher_.find(msg->op);
        if (handler) {
            auto* data = reinterpret_cast<void*>(msg->data);
            handler(core_ctx, msg->source_core, data);
            // handler 내부에서 payload release (refcount--)
        }
        ++processed;
    }

    if (processed == config_.drain_batch_limit) {
        boost::asio::post(core_ctx.io_ctx, [this, core_id] {
            drain_inbox(core_id);
        });
    }
}
```

handler 입장에서는 data가 어디서 왔든 `void*`로 동일하게 접근. 프레임워크 내부 로직과
서비스 핸들러가 동일한 인터페이스.

## 2. 메모리 아키텍처 개선

### 2.1 `SessionPtr` → `intrusive_ptr` 전환 (non-atomic refcount)

**[출처: 사용자 2.2.1 + 토론 합의]**

**변경 내용**:
```cpp
class Session {
    uint32_t refcount_{0};  // non-atomic — per-core 전용
    friend void intrusive_ptr_add_ref(Session* s) noexcept { ++s->refcount_; }
    friend void intrusive_ptr_release(Session* s) noexcept;
};
using SessionPtr = boost::intrusive_ptr<Session>;
```

**아키텍처 불변조건 (확정)**:
> **SessionPtr은 코어 경계를 넘지 않는다.**
> cross_core_call/cross_core_post에서는 SessionId(uint64_t)만 전달하고,
> 대상 코어에서 `find_session(id)`로 코어 로컬 SessionPtr을 획득한다.

[v5] 이 불변조건은 1.6절의 message passing 전환과 자연스럽게 결합된다.
cross-core 메시지에서 SessionPtr 대신 SessionId를 전달하는 것이 message passing의
data-only 원칙에 부합.

**non-atomic이 가능한 이유와 캐시 효율 이점**:
- per-core 아키텍처에서 Session은 하나의 코어에 귀속
- "atomic을 잘 쓰는 것"이 아니라 "쓸 일을 없애는 것"이 shared-nothing의 진짜 이점

**[v2] pool vs heap 구분 전략**: `pool.owns(ptr)` 패턴.
**[v3] `SlabPool::owns()` 이미 구현 확인**: slab_pool.hpp 66행.
**[v3] `enable_shared_from_this` 제거 안전 확인**: 호출 0건.

**[v3] intrusive_ptr 전환 안전망**:
1. Debug assertion: refcount 오버플로/언더플로 검출
2. Session 소멸자 noexcept 검증
3. 전환 직후 전체 통합 테스트 + TSAN 실행

**검토 필요사항**:
- [x] refcount atomicity → **non-atomic 확정**. [v1]
- [x] enable_shared_from_this 사용처 → **0건, 안전 제거 가능**. [v3]
- [x] pool/heap 구분 → **`pool.owns(ptr)` 패턴**. [v2/v3]
- [x] ~~Session::~Session() 명시적 noexcept~~ → **암시적 noexcept 확인. 명시 추가는 구현 시 trivial.** [v5]

std::function 풀링이 아니라 문제의 근원을 없앰.

### B-4. per-session write queue (concurrent write 방지) [v3]

**문제**: 현재 `async_send`는 단일 세션에서 동시에 하나의 write만 허용.
process_frames의 순차 처리로 보장.

**향후 위험 시나리오** (Phase 7~8):
- 채팅 브로드캐스트: dispatch 핸들러가 같은 코어의 다른 세션들에 async_send
- push 알림: Redis Pub/Sub → cross_core_post → 세션에 async_send

**해결 방향**: per-session write queue (Phase 7 구현 시점)

**3.1/3.2 작업 시 고려사항**:
- ConnectionHandler에서 send_message() 추상화 계층 고려
- async_send의 `Result<void>` 에러 코드에 `WritePending` 추가 검토
