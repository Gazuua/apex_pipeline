**브랜치**: `feature/backlog_refactoring`

## 1. 목적

BACKLOG.md를 체계적 이슈 관리 도구로 탈바꿈하고, 전체 문서와 구현 간의 정합성을 확보한다.

### 스코프

1. **BACKLOG.md 템플릿화** — 2축 분류 체계, 히스토리 문서, 운영 가이드라인 확립
2. **문서-구현 정합성** — BACKLOG 항목 유효성, 로드맵, plans/progress/review 추적성, CLAUDE.md 규칙 전 레이어 검증

### 스코프 외

- 신규 백로그 항목 추가 (템플릿 안정화 후 별도 진행)

## 2. 2축 분류 체계

### 2.1 시간축 (1차 분류) — "언제 착수하는가"

| 등급 | 판단 질문 | 기준 |
|------|-----------|------|
| **NOW** | 이번 마일스톤에서 안 하면 다음 단계 진행이 막히는가? | 현재 마일스톤의 선행 조건/블로커. 이미 착수 결정된 항목 |
| **IN VIEW** | 다음 1-2 마일스톤 내에 필요해질 게 확실한가? | 가까운 미래에 필요. 구체적 트리거 조건 있음 |
| **DEFERRED** | 위 둘 다 아닌 경우 | 외부 조건 충족 시 재평가 |

**승격 규칙**: 마일스톤 전환 시 IN VIEW 재평가 → NOW 승격 여부 판단. DEFERRED는 트리거 조건 발생 시에만 재평가.

### 2.2 내용축 (2차 분류) — "얼마나 심각한가"

| 등급 | 판단 질문 | 기준 |
|------|-----------|------|
| **CRITICAL** | 안 고치면 시스템이 깨지거나 데이터가 위험한가? | 크래시, 데이터 손실/손상, 보안 취약점 |
| **MAJOR** | 안 고치면 이후 작업 비용이 눈에 띄게 늘어나는가? | 설계 부채, 테스트 부재, 문서 부재, 확장성 병목 |
| **MINOR** | 위 둘 다 아닌 경우 | 기능·안정성에 영향 없음. 코드 위생, 오타, 탐색적 최적화 |

**경계 케이스 원칙**: 두 등급 사이에서 고민되면 높은 쪽으로.

## 3. BACKLOG.md 구조

### 3.1 파일 헤더

```markdown
# BACKLOG

미해결 이슈 집약. 새 작업 시작 전 반드시 확인.
완료 항목은 즉시 삭제 후 `docs/BACKLOG_HISTORY.md`에 기록.
운영 규칙: `docs/CLAUDE.md` § 백로그 운영 참조.

다음 발번: {N}
```

### 3.2 시간축 섹션

```markdown
## NOW
(항목들...)

## IN VIEW
(항목들...)

## DEFERRED
(항목들...)
```

### 3.3 항목 템플릿

```markdown
### #{ID}. 이슈 제목
- **등급**: CRITICAL | MAJOR | MINOR
- **스코프**: {모듈 태그, 복수 가능}
- **타입**: bug | design-debt | test | docs | perf | security | infra
- **설명**: 이슈 상세. 파일 경로, 버전 의존 관계 등 휘발성 정보는 여기에 자연스럽게 기술.
```

### 3.4 항목 ID

- Auto-increment 정수. 자릿수 패딩 없음 (`#1`, `#12`, `#100`)
- 한 번 발번된 ID는 재사용하지 않음. ID 갭 허용
- `다음 발번` 카운터를 파일 헤더에서 관리
- **머지 충돌 시**: 높은 쪽 번호 + 1로 재발번

### 3.5 메타데이터 태그

**스코프** (어디에 영향을 주는가, 복수 가능):

```
core | shared | gateway | auth-svc | chat-svc | infra | ci | docs | tools
```

서비스 추가 시 태그 확장.

**타입** (어떤 성격의 작업인가):

```
bug | design-debt | test | docs | perf | security | infra
```

## 4. BACKLOG_HISTORY.md

**파일 경로**: `docs/BACKLOG_HISTORY.md`

### 4.1 파일 구조

```markdown
# BACKLOG HISTORY

완료된 백로그 항목 아카이브. 최신 항목이 파일 상단.

<!-- NEW_ENTRY_BELOW -->

---

(항목들...)

---
```

### 4.2 항목 템플릿

```markdown
### #{ID}. 이슈 제목
- **등급**: {등급} | **스코프**: {스코프} | **타입**: {타입}
- **해결**: YYYY-MM-DD HH:MM:SS | **방식**: {해결 방식} | **커밋**: {short hash, 선택}
- **비고**: {특이사항 또는 —}
```

### 4.3 해결 방식

| 방식 | 의미 |
|------|------|
| **FIXED** | 코드 수정으로 해결 |
| **DOCUMENTED** | 문서/주석 보강으로 해결 |
| **WONTFIX** | 검토 결과 수정 불필요 판단 |
| **DUPLICATE** | 다른 항목과 중복 (비고에 대상 명시) |
| **SUPERSEDED** | 다른 작업으로 자연 해소 |

### 4.4 운영 규칙

- 완료된 항목은 BACKLOG.md에서 삭제 → BACKLOG_HISTORY.md에 prepend
- 삽입 위치: `<!-- NEW_ENTRY_BELOW -->` 마커 바로 아래
- 마커를 삭제하거나 이동하지 않음. 마커 누락 시 파일 상단(헤더 직후)에 마커를 복원한 뒤 작업 진행
- 커밋 필드는 선택 — 해당 없는 경우(WONTFIX 등) 생략

## 5. 백로그 운영 가이드라인

### 5.1 접근 규칙

BACKLOG 항목을 추가·수정·착수·리뷰할 때, 반드시 실제 구현 상태(코드베이스, git 이력, 테스트 결과 등)를 검증하여 사실관계를 확인한다. 문서에 적힌 상태를 그대로 신뢰하지 않는다.

### 5.2 가이드라인 배치

- **`docs/CLAUDE.md`**: 백로그 운영 규칙 섹션 신설. 본 설계서 §2~§5의 내용을 **자립형(self-contained)으로 수록** — 2축 체계 판단 기준표, 승격 규칙, 히스토리 운영(prepend, 마커, 타임스탬프), 해결 방식 목록, 태그 목록(스코프/타입), ID 규칙, 접근 규칙. 본 설계서는 의사결정 이력으로 보존하되, 운영 가이드의 정본은 `docs/CLAUDE.md`
- **루트 `CLAUDE.md`**: 기존 "완료 항목은 즉시 삭제 (git이 이력 보존)" → "완료 항목은 즉시 삭제 후 `docs/BACKLOG_HISTORY.md`에 기록"으로 **교체**. `docs/CLAUDE.md` § 백로그 운영 참조 포인터 추가
- **`BACKLOG.md` 헤더**: 최소 요약만 (이슈 목록으로서의 본래 목적 유지)

## 6. 문서-구현 정합성 검증

### 6.1 레이어 A — BACKLOG 항목 유효성

**대상 파일**:
- `docs/BACKLOG.md` (기존 ~30개 항목)
- `docs/apex_core/backlog_memory_os_level.md` (5개 항목, 이 중 3개는 BACKLOG.md와 중복)
- `docs/apex_shared/review/20260315_094300_backlog.md` (~19개 항목)

**검증 방법**: 각 항목을 코드베이스/git 대조하여 이슈 존재 여부, 상태 기술 정확성 검증

**결과 처리**:
- 검증된 항목 → 새 포맷으로 BACKLOG.md 수록
- 이미 해결된 항목 → `docs/BACKLOG_HISTORY.md`에 SUPERSEDED 또는 FIXED로 기록
- 중복 항목 → 더 상세한 기술을 택하여 단일 항목으로 통합
- "해당 없음"으로 판정된 항목 → BACKLOG_HISTORY.md에 WONTFIX로 아카이브
- 별도 백로그 파일 2건 → 통합 완료 후 원본 삭제

**기존 → 신규 등급 매핑 가이드**:

시간축:
- 기존 "v0.6 선행/필수", "빠를수록 좋음" → **NOW** (현재 마일스톤 블로커)
- 기존 "v0.6", "v0.5 패치", 구체적 배치 있음 → **IN VIEW** (가까운 미래 필요)
- 기존 "Wave 배정 보류", "v1.0 이후", "별도 판단" → **DEFERRED**

내용축:
- 기존 Critical → 판단 기준표로 재평가 (대부분 **CRITICAL**)
- 기존 Important, Medium → 판단 기준표로 재평가 (대부분 **MAJOR**)
- 기존 Minor, Low, 코드 위생 → 판단 기준표로 재평가 (대부분 **MINOR**)

카테고리 섹션(단위 테스트, 서비스 체인 등) → **스코프 태그**로 대체

### 6.2 레이어 B — 로드맵 문서 정합성

- `docs/Apex_Pipeline.md` §9(버전 이력), §10(로드맵)을 main 상태와 대조

### 6.3 레이어 C — plans/progress/review 추적성

- 해결 가능한 갭 수정, 복원 불가능한 건 비고 기록

### 6.4 레이어 D — CLAUDE.md 규칙 정합성

- 루트 + 각 영역 CLAUDE.md를 현재 프로젝트 구조/규칙과 대조

## 8. JIRA 마이그레이션 적합성

모든 필드가 JIRA 기본/커스텀 필드에 자연 대응:

| 우리 필드 | JIRA 필드 |
|-----------|-----------|
| `#번호` | Issue Key (자체 발번, 매핑 테이블) |
| 등급 | Severity 커스텀 필드 |
| 스코프 | Component(s) |
| 타입 | Issue Type |
| 시간축 | Sprint 배정 / Status |
| 설명 | Description |
| 히스토리 | Resolution + 링크 |

유일한 주의점: JIRA는 Priority와 Severity를 보통 하나로 퉁치므로, 2축 분리를 유지하려면 Severity 커스텀 필드 추가 필요 (설정 5분).
