# v0.4 외부 어댑터 설계서

> **버전**: v0.4.0.0 — 외부 어댑터 (Kafka + Redis + PostgreSQL + Connection Pool)
> **작성일**: 2026-03-13
> **상태**: 승인됨
> **선행**: v0.3.0.0 (코어 성능 완성)

---

## 1. 목표

코어 프레임워크(`apex_core`)의 코루틴/Asio 아키텍처 위에 외부 시스템(Kafka, Redis, PostgreSQL)과의
비동기 통신 인프라를 구축한다. 모든 I/O를 Asio 이벤트 루프에 통합하여 단일 코루틴 모델 + 락프리 구조를 유지한다.

### 서브버전 구조

| 버전 | 내용 | 라이브러리 |
|------|------|-----------|
| v0.4.0.1 | 공통 추상화 + 코어 변경 (AdapterBase, ConnectionPool, AdapterError, Server API) | — |
| v0.4.1.0 | Kafka 어댑터 (Producer/Consumer) | librdkafka 2.x |
| v0.4.1.1 | KafkaSink (spdlog sink) + trace_id 자동 주입 | librdkafka 2.x |
| v0.4.2.0 | Redis 어댑터 (per-core 풀) | redis-plus-plus + hiredis |
| v0.4.3.0 | PostgreSQL 어댑터 (libpq fd → Asio) | libpq |
| v0.4.4.0 | Connection Pool 구현체 통합 (Redis/PG 풀 공통화 리팩터링) | — |

> v0.4.0.1(공통 추상화)을 먼저 구현한 뒤, v0.4.1(Kafka)과 v0.4.2~3(Data)을 병렬 진행 가능 (마스터 설계서 §10 참조).
> v0.4.4는 개별 어댑터 구현 후 풀 공통화 리팩터링 단계.

## 3. 코어 통합

### 서버 구성

어댑터는 서비스가 아닌 **인프라 컴포넌트**로 등록한다.

```cpp
Server({.port = 9000, .num_cores = 4})
    .add_adapter<KafkaAdapter>(kafka_config)
    .add_adapter<RedisAdapter>(redis_config)
    .add_adapter<PgAdapter>(pg_config)
    .add_service<GatewayService>()
    .run();
```

- 체이닝 순서는 무관 — 실제 초기화 순서는 `run()` 내부에서 보장
- 서비스에서 어댑터 접근: `server.adapter<KafkaAdapter>()`

### 초기화/종료 순서

```
run() 내부:
  adapters init → services on_start → event loop
```

### API 패턴

코루틴 thin wrapper — `awaitable<Result<T>>` 기반:

```cpp
auto& kafka = server.adapter<KafkaAdapter>();
co_await kafka.produce("topic", key, payload);

auto& redis = server.adapter<RedisAdapter>();
co_await redis.get(key);

auto& pg = server.adapter<PgAdapter>();
co_await pg.query("SELECT ...", params...);
```

구체적인 반환 타입(insert/update 응답 등)은 각 어댑터 구현 단계에서 라이브러리 응답 모델을 검토하여 확정.

## 4. 공통 추상화

### AdapterBase CRTP

```cpp
template <typename Derived>
class AdapterBase {
public:
    // 라이프사이클 (Server::run()이 호출)
    // Derived가 구현: do_init(), do_drain(), do_close()

    void init(CoreEngine& engine);       // 코어별 리소스 초기화 (풀 등)
    void drain();                        // "새 요청 거부" 시그널
    awaitable<Result<void>> close();     // flush + 커넥션 정리

    bool is_ready() const noexcept;      // init 완료 + drain 안 됨
    const auto& config() const;          // 어댑터별 TOML 설정
};
```

**소유 모델**: 어댑터는 전역 1개 인스턴스로 `Server`에 등록된다. 내부적으로 코어별 상태(풀 등)를
관리하며, `init(CoreEngine&)`에서 각 코어의 `io_context`에 대해 코어별 리소스를 생성한다.
서비스에서 `server.adapter<T>()`로 접근하면, 어댑터가 `CoreEngine::current_core_id()`를 이용해
현재 코어에 해당하는 풀/리소스로 자동 라우팅한다.

```
adapter<RedisAdapter>().get(key)
  → 내부: pools_[current_core_id()].acquire() → 커넥션에서 GET 실행
```

### ConnectionPool CRTP (Redis/PG 전용)

```cpp
template <typename Derived>
class ConnectionPool {
    struct Config {
        size_t min_size;
        size_t max_size;
        std::chrono::seconds max_idle_time;
        std::chrono::seconds health_check_interval;
    };

    // Derived가 구현: create_connection(), destroy_connection(), validate()

    awaitable<Result<Connection>> acquire();
    void release(Connection conn);
    void health_check_tick();           // TimingWheel 연동
    void shrink_idle();                 // max_idle_time 초과 정리
};
```

**특성:**
- **코어별 독립 인스턴스** — 락 불필요
- **auto-grow** — 부하 시 설정 범위 내 자동 확장
- **backpressure** — 풀 소진 시 `ErrorCode::AdapterError` + "pool exhausted"

Kafka는 Connection Pool을 사용하지 않음 — Producer가 전역 공유, Consumer가 코어별이라 풀링 패턴과 맞지 않음.

### AdapterError

```cpp
struct AdapterError {
    ErrorCode code = ErrorCode::AdapterError;
    uint32_t native_error = 0;          // 라이브러리 원본 에러코드
    std::string message;                // 사람 읽기용 상세
};
```

코어 `ErrorCode` enum에는 `AdapterError` 하나만 추가. 에러 범위 구조:
- Framework: 1-999
- Application: 1000-1999
- Adapter: 2000+ (`AdapterError = 2000`)

상세 에러는 `AdapterError` 구조체로 전달하여 코어 에러 체계 오염 방지.

## 5. Kafka 어댑터 (v0.4.1)

### 구조 (ADR-08 기반)

```
KafkaAdapter (AdapterBase CRTP)
├── KafkaProducer — 전역 공유 1개
├── KafkaConsumer — 파티션:코어 매핑
└── KafkaSink    — spdlog sink (null_mutex)
```

### Producer 모델

```
[Core 0] ──produce()──┐
[Core 1] ──produce()──┤──→ [KafkaProducer 1개] ──→ [Broker]
[Core 2] ──produce()──┘
```

- `rd_kafka_produce()`가 스레드세이프 → 코어에서 직접 호출
- 별도 SPSC 큐 불필요 (librdkafka 내부 큐가 역할 수행)
- delivery callback → Asio로 브릿지하여 비동기 결과 수신
- **ADR-08 업데이트 필요**: 기존 ADR-08은 SPSC 큐 경유를 기술하나, librdkafka 자체가 스레드세이프 + 내부 배치 처리를 하므로 직접 호출이 더 단순하고 효율적. 마스터 설계 결정 문서를 이에 맞게 갱신한다.

### Consumer 모델

```
[Broker Partition 0] ──→ [Core 0 Consumer]
[Broker Partition 1] ──→ [Core 1 Consumer]
[Broker Partition 2] ──→ [Core 2 Consumer]
```

- consumer group이 파티션을 코어에 분배
- `rd_kafka_queue_io_event_enable()`로 pipe/eventfd에 이벤트 포워딩 → 해당 fd를 Asio에 등록 (Linux)
- Windows: fd 등록 불가 → 폴링 타이머 fallback (5ms 주기)

### KafkaSink

- `spdlog::sinks::base_sink<spdlog::details::null_mutex>` 상속
- librdkafka produce가 스레드세이프이므로 spdlog 측 mutex 불필요
- 전용 토픽 분리 (비즈니스 메시지와 대역폭 경합 방지)
- trace_id 자동 주입 (세션 컨텍스트 → 로그)

### Asio 통합

```cpp
// Linux: rd_kafka_queue_io_event_enable → pipe fd → Asio 등록
int pipefd[2];
pipe(pipefd);
rd_kafka_queue_io_event_enable(queue, pipefd[1], "1", 1);
boost::asio::posix::stream_descriptor desc(ioc, pipefd[0]);
desc.async_wait(wait_read, [](auto ec) {
    char buf; read(pipefd[0], &buf, 1);  // 이벤트 소비
    rd_kafka_consumer_poll(rk, 0);
});

// Windows: 폴링 fallback
boost::asio::steady_timer timer(ioc);
timer.expires_after(5ms);
timer.async_wait([](auto ec) { rd_kafka_consumer_poll(rk, 0); });
```

## 6. Redis 어댑터 (v0.4.2)

### 구조

```
RedisAdapter (AdapterBase CRTP)
└── RedisPool (ConnectionPool CRTP) — 코어별 독립 인스턴스
    └── redis-plus-plus async 커넥션 N개
```

### Per-core 모델

```
[Core 0] ── RedisPool (커넥션 2~4개)
[Core 1] ── RedisPool (커넥션 2~4개)
[Core 2] ── RedisPool (커넥션 2~4개)
```

### Asio 통합

**1순위: hiredis fd를 Asio에 직접 등록** (단일 이벤트 루프 통합)
- 아키텍처 원칙("모든 I/O를 Asio에서")과 일치
- hiredis의 `redisAsyncContext`에서 fd를 꺼내 Asio에 등록
- redis-plus-plus를 얇은 API 래퍼로만 사용

**Fallback: redis-plus-plus 자체 async** (내부 이벤트 루프)
- hiredis fd 직접 등록이 기술적으로 불가능한 경우에만 사용
- 별도 스레드에서 이벤트 루프 실행, MPSC로 코어 스레드와 통신

구현 초기에 1순위 방식의 PoC를 수행하여 확정한다.

### 용도 (v0.4 시점)

- 세션 L2 백업 (SessionManager 연동)
- 범용 캐시 인프라
- v0.5에서 추가: JWT 블랙리스트, Pub/Sub

## 7. PostgreSQL 어댑터 (v0.4.3)

### 구조

```
PgAdapter (AdapterBase CRTP)
└── PgPool (ConnectionPool CRTP) — 코어별 독립 인스턴스
    └── libpq 비동기 커넥션 N개
```

### Asio 통합 (직접 구현 — 포트폴리오 핵심)

```
1. PQconnectStart()          → 비동기 커넥션 시작
2. PQsocket(conn)            → fd 획득
3. Asio에 fd 등록             → readable/writable 대기
4. PQconnectPoll()           → 커넥션 완료까지 반복
5. PQsendQuery()             → 비동기 쿼리 전송
6. Asio가 readable 알림      → PQgetResult() 수신
```

### 환경 구성

**모든 환경에서 PgBouncer 사용** (개발/테스트/프로덕션 동일):
- `apex_infra/docker-compose.yml` minimal 프로파일에 PgBouncer 기본 포함
- 어댑터는 PgBouncer(6432)에 연결, PgBouncer가 PG(5432)에 풀링
- 코어 수 증가에 따른 커넥션 폭증 문제를 인프라 레벨에서 해결 (설계서 §5 ADR)

### Windows 호환

Windows에서 `PQsocket()`은 SOCKET 핸들을 반환하며, `boost::asio::ip::tcp::socket::assign(protocol, PQsocket(conn))`으로 IOCP에 등록 가능.
단, libpq 소켓을 non-blocking 모드로 설정해야 함 (`PQsetnonblocking(conn, 1)`).

## 8. Graceful Shutdown 시퀀스

기존 코어 shutdown에 어댑터 단계를 삽입한다.

**현재 코드 순서** (`server.cpp` 기준):
```
Acceptor 중지 → 세션 close + drain → CoreEngine stop+join+drain_remaining → 서비스 stop → 종료
```

**변경 후 순서**:
```
SIGTERM
  │
  ▼
1. Acceptor 중지 (새 클라이언트 거부)
  │
  ▼
2. Adapter drain() — 새 요청 거부, 진행 중 허용
  │
  ▼
3. 세션 close + drain 폴링 (active_sessions == 0 대기)
  │  → 이 동안 서비스는 어댑터 사용 가능 (Kafka flush, PG 트랜잭션 완료 등)
  │
  ▼
4. 서비스 on_stop()
  │
  ▼
5. Adapter close() — Kafka flush, 커넥션 풀 반환, fd 정리
  │
  ▼
6. CoreEngine stop + join + drain_remaining
  │
  ▼
7. shutdown_logging()
```

**핵심 변경**: 기존 코드에서 CoreEngine stop이 서비스 stop보다 앞에 있었으나,
어댑터 close가 io_context 위에서 비동기로 동작해야 하므로 CoreEngine을 가장 나중에 정지한다.
`server.cpp`의 `finalize_shutdown()` 순서 재배치가 필요하며, 마스터 설계서(§5 Graceful Shutdown)도 동기화한다.

**핵심**: 2~3 사이에서 서비스가 마지막 작업 가능, 5에서 어댑터 깔끔하게 정리.


코어에 어댑터 설정 타입 의존성 없음.

## 11. 코어 프레임워크 변경 사항

`apex_core`에 대한 최소 변경:

### 1. Server에 add_adapter API 추가

```cpp
// server.hpp
template <typename T, typename... Args>
Server& add_adapter(Args&&... args);

template <typename T>
T& adapter() const;  // 내부에서 current_core_id()로 코어별 리소스 라우팅
```

### 2. Graceful Shutdown 시퀀스 재배치

`server.cpp`의 `finalize_shutdown()` 순서를 변경한다:
- 기존: acceptor → 세션 drain → CoreEngine stop → 서비스 stop
- 변경: acceptor → **adapter drain** → 세션 drain → 서비스 stop → **adapter close** → CoreEngine stop

CoreEngine이 가장 나중에 정지하도록 재배치. 어댑터 close가 io_context에서 비동기 flush를 수행해야 하므로.
마스터 설계서(§5 Graceful Shutdown)도 동기화 필요.

### 3. ErrorCode 확장

```cpp
// error_code.hpp
// Framework: 1-999 (기존)
// Application: 1000-1999 (기존 AppError = 1000)
AdapterError = 2000,  // Adapter: 2000+
```

이 3가지 외에 코어 프레임워크 변경 없음.

## 12. 구현 순서

전체 설계를 먼저 완료한 상태에서, 구현은 다음 순서를 따른다:
