**Goal:** v0.5 서비스 체인 완성 — 리뷰 이슈 8건 수정 + 인증 플로우 연동 + Chat 핸들러 PG/Redis 연동 + 서비스 부트스트래핑 + E2E 테스트 동작

**Architecture:** 기존 통신 인프라(Kafka envelope, MessageRouter, ResponseDispatcher)는 완성 상태. 핸들러의 placeholder를 실제 JWT/PG/Redis 호출로 채운다. Kafka consumer callback은 동기이므로, 핸들러를 `awaitable<void>`로 전환하고 `co_spawn`으로 현재 코어의 executor에 위임하는 패턴을 사용한다. Auth/Chat main.cpp를 완성하여 독립 프로세스 실행을 가능하게 한다.

## 핵심 설계 결정

### 디스패치 패턴: MessageDispatcher (O(1) hash map)

코어 프레임워크에 이미 `MessageDispatcher`(O(1) hash map) + `ServiceBase<T>::route<FbsType>()` 패턴이 완비되어 있다. Auth/Chat 서비스는 switch-case 대신 `MessageDispatcher`를 단독 멤버로 소유하고, 서비스 초기화(`start()`) 시 핸들러를 등록하는 패턴을 사용한다. `ServiceBase` 상속은 불필요 (Kafka 메시지 드리븐이라 SessionPtr이 meaningless).

```cpp
// apex_core/include/apex/core/message_dispatcher.hpp
class MessageDispatcher {
    using Handler = std::function<
        boost::asio::awaitable<Result<void>>(SessionPtr, uint32_t, std::span<const uint8_t>)>;
    void register_handler(uint32_t msg_id, Handler handler);
    awaitable<Result<void>> dispatch(SessionPtr session, uint32_t msg_id, std::span<const uint8_t> payload) const;
};
```

### 비동기 전환 패턴

Kafka consumer callback → 동기. PG/Redis 어댑터 → awaitable. 서비스는 io_context를 직접 소유하지 않는다. 어댑터는 `CoreEngine::current_core_id()` (thread-local) 기반으로 per-core 풀을 자동 라우팅하므로, 서비스는 어댑터 참조만 저장하면 된다. Kafka 콜백 → 코루틴 전환 시, main에서 주입받은 `any_io_executor`를 사용하여 `co_spawn`으로 디스패치한다 (Kafka callback은 동기이므로 `co_await this_coro::executor` 사용 불가). io_context 자체를 서비스에 직접 노출하면 executor 미스매치, data race, adapter 라우팅 오류가 발생한다.

```cpp
// 서비스는 어댑터 참조만 저장 — io_context 직접 소유 금지
class AuthService {
    apex::core::MessageDispatcher dispatcher_;
    boost::asio::any_io_executor executor_;  // Kafka→코루틴 브릿지용 (main에서 주입)
    apex::shared::adapters::pg::PgAdapter& pg_;
    apex::shared::adapters::redis::RedisAdapter& redis_;

    void start() {
        dispatcher_.register_handler(msg_ids::LOGIN_REQUEST,
            [this](auto session, auto msg_id, auto payload) -> awaitable<Result<void>> {
                return handle_login(payload, /* metadata from envelope */);
            });
        dispatcher_.register_handler(msg_ids::LOGOUT_REQUEST,
            [this](auto session, auto msg_id, auto payload) -> awaitable<Result<void>> {
                return handle_logout(payload);
            });
        dispatcher_.register_handler(msg_ids::REFRESH_TOKEN_REQUEST,
            [this](auto session, auto msg_id, auto payload) -> awaitable<Result<void>> {
                return handle_refresh_token(payload);
            });
    }

    // dispatch_envelope()은 동기 (Kafka callback에서 호출)
    // Kafka consumer thread → 코루틴 전환 브릿지
    void dispatch_envelope(std::span<const uint8_t> payload) {
        auto msg = parse_envelope(payload);  // 가벼운 동기 파싱

        // Kafka callback은 동기이므로 co_await 사용 불가.
        // main에서 주입받은 executor_ 사용.
        boost::asio::co_spawn(executor_,
            dispatcher_.dispatch(nullptr, msg.routing.msg_id, msg.fbs_payload),
            boost::asio::detached);
    }

    // 핸들러는 awaitable — PG/Redis co_await 가능
    awaitable<void> handle_login(...) {
        // 어댑터가 current_core_id()로 자동 라우팅
        auto user = co_await pg_.query(...);
        auto token = jwt_manager_.create_access_token(user.id, user.email);
        co_await session_store_.set(user.id, token);
        send_response(...);  // Kafka produce (동기)
    }
};
```

### main.cpp 구조

```cpp
int main() {
    auto config = parse_toml("auth_svc.toml");

    boost::asio::io_context io_ctx;
    KafkaAdapter kafka(kafka_cfg);
    RedisAdapter redis(redis_cfg);
    PgAdapter pg(pg_cfg);

    // io_context 자체가 아닌 executor만 전달 — 서비스는 io_context를 소유하지 않음
    AuthService auth(config, io_ctx.get_executor(), kafka, redis, pg);
    auth.start();  // MessageDispatcher에 핸들러 등록

    // io_context를 별도 스레드에서 run (코루틴 실행)
    std::jthread io_thread([&] { io_ctx.run(); });

    // Kafka consumer loop (메인 스레드)
    kafka.consume_loop();  // blocking
}
```

```
Wave B (순차):     Task 5 (MessageDispatcher 전환 + awaitable)
                        ↓
Wave C (병렬):     Task 6 | Task 7 | Task 8 | Task 9+10+11
                        ↓
Wave D (순차):     Task 12 → Task 13 → Task 14
                        ↓
Wave E (순차):     Task 15 → Task 16 → Task 17
                        ↓
Wave F:            Task 18 (빌드 + 검증)
```

## 리스크

| 리스크 | 대응 |
|--------|------|
| PG/Redis 어댑터의 awaitable API가 핸들러에서 기대하는 형태와 다를 수 있음 | Task 5에서 인터페이스 확인 후 조정 |
| MessageDispatcher의 Handler 시그니처(SessionPtr)가 Kafka 메시지 드리븐과 맞지 않을 수 있음 | nullptr 전달 또는 Kafka metadata를 SessionPtr에 래핑하는 어댑터 작성 |
| 어댑터의 per-core 자동 라우팅이 Kafka consumer thread에서 올바르게 동작하는지 검증 필요 | Kafka consumer thread의 core_id 매핑 확인, 필요 시 explicit core_id 전달 |
| E2E 프로세스 launch가 Windows에서 불안정 | boost::process 또는 CreateProcess API 고려 |
| PG 스키마(auth.users, chat_svc.chat_rooms 등)가 미정의 | docker-compose init-sql/ 확인, 없으면 생성 |
| RefreshToken PG 테이블(auth.refresh_tokens)이 없을 수 있음 | init-sql에 CREATE TABLE 추가 |
| gateway.toml에 [auth.exempt] 섹션이 없으면 파싱 실패 | 기본값 빈 set으로 fallback — auth_exempt_msg_ids 미설정 시 모든 요청 인증 필수 |
