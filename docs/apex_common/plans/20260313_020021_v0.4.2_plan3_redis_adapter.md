# v0.4.2 Redis 어댑터 — Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** hiredis fd를 Boost.Asio에 직접 등록하여 코루틴 기반 비동기 Redis 클라이언트를 구현한다. 코어별 독립 풀(RedisPool)을 관리하는 RedisAdapter를 제공하여 서비스에서 `co_await redis.get(key)` 형태로 사용 가능하게 한다.

**Architecture:** RedisAdapter는 AdapterBase CRTP를 상속하며, 내부에 코어별 RedisPool 인스턴스를 관리한다. RedisPool은 ConnectionPool CRTP를 상속하여 hiredis `redisAsyncContext` 커넥션을 풀링한다. 핵심은 hiredis fd를 Asio `tcp::socket::assign()`으로 등록하여 단일 이벤트 루프에서 Redis I/O를 처리하는 것이다.

**Tech Stack:** C++23, Boost.Asio, hiredis 1.3.0, redis-plus-plus 1.3.15 (thin API wrapper), Google Test, CMake + vcpkg

## 배경: hiredis Asio 통합 전략

### 핵심 원리

hiredis의 `redisAsyncContext`는 내부적으로 소켓 fd를 소유한다. 이 fd를 Boost.Asio의 `tcp::socket`에 `assign()`하면 Asio 이벤트 루프가 Redis 소켓의 read/write 이벤트를 직접 처리할 수 있다. 이는 프로젝트의 아키텍처 원칙("모든 I/O를 Asio 이벤트 루프에 통합")과 정확히 일치한다.

### 동작 흐름

```
redisAsyncConnect()
  → redisAsyncContext.c.fd 획득
  → boost::asio::ip::tcp::socket::assign(tcp::v4(), fd)
  → hiredis ev 콜백 등록 (addRead/delRead/addWrite/delWrite/cleanup)
  → addRead 콜백 → socket.async_wait(wait_read, ...) → redisAsyncHandleRead()
  → addWrite 콜백 → socket.async_wait(wait_write, ...) → redisAsyncHandleWrite()
  → 코루틴에서 co_await → hiredis 커맨드 콜백 → promise resume
```

### Windows 호환성

- Linux: `posix::stream_descriptor` 또는 `tcp::socket::assign(tcp::v4(), fd)`
- Windows: hiredis가 Winsock `SOCKET` 핸들을 반환 → `tcp::socket::assign(tcp::v4(), fd)`로 IOCP에 등록 가능
- 양쪽 모두 `assign()` 패턴이 동작하므로 플랫폼 분기 최소화

### 참고 구현

[ryangraham/hiredis-boostasio-adapter](https://github.com/ryangraham/hiredis-boostasio-adapter) — hiredis fd를 Boost.Asio 리액터에 등록하는 레퍼런스 구현. 본 프로젝트에서는 이 패턴을 C++23 코루틴 + `awaitable<Result<T>>` 기반으로 현대화한다.

### 설계

#### HiredisAsioAdapter — fd → Asio 브릿지

```cpp
/// hiredis redisAsyncContext의 fd를 Boost.Asio 소켓에 등록하여
/// hiredis의 이벤트 콜백을 Asio 이벤트 루프에 위임하는 어댑터.
///
/// 패턴: redisAsyncContext → ev 콜백 등록 → Asio socket async_wait
///
/// 하나의 redisAsyncContext에 대해 하나의 HiredisAsioAdapter 인스턴스.
class HiredisAsioAdapter {
public:
    /// redisAsyncContext를 Asio io_context에 바인딩.
    /// 내부적으로 fd를 추출하여 tcp::socket에 assign한다.
    HiredisAsioAdapter(boost::asio::io_context& io_ctx, redisAsyncContext* ac);
    ~HiredisAsioAdapter();

    // Non-copyable, movable (소켓 소유)
    HiredisAsioAdapter(const HiredisAsioAdapter&) = delete;
    HiredisAsioAdapter& operator=(const HiredisAsioAdapter&) = delete;

private:
    // hiredis ev 콜백으로 등록되는 static 함수들
    static void on_add_read(void* privdata);
    static void on_del_read(void* privdata);
    static void on_add_write(void* privdata);
    static void on_del_write(void* privdata);
    static void on_cleanup(void* privdata);

    void handle_read();
    void handle_write();

    boost::asio::ip::tcp::socket socket_;
    redisAsyncContext* ac_;               ///< 비소유. 생명주기는 RedisConnection이 관리.
    bool read_requested_ = false;
    bool write_requested_ = false;
};
```

#### RedisConnection — 단일 커넥션 래퍼

```cpp
/// 하나의 hiredis 비동기 커넥션을 래핑한다.
/// HiredisAsioAdapter를 소유하여 Asio에 통합.
/// 커맨드 발행 시 코루틴 awaitable로 결과를 반환한다.
class RedisConnection {
public:
    /// 비동기 커넥션 생성. io_context에서 실행.
    static awaitable<Result<std::unique_ptr<RedisConnection>>>
    create(boost::asio::io_context& io_ctx, const RedisConfig& config);

    ~RedisConnection();

    /// Redis 커맨드 실행 → redisReply 반환
    /// 내부: redisAsyncCommand() + 코루틴 resume
    awaitable<Result<redisReply*>>
    command(const char* format, ...);

    /// 커넥션 유효성 검증 (PING)
    bool validate();

    /// 커넥션 정리
    void disconnect();

    [[nodiscard]] bool is_connected() const noexcept;

private:
    RedisConnection(boost::asio::io_context& io_ctx,
                    redisAsyncContext* ac);

    boost::asio::io_context& io_ctx_;
    redisAsyncContext* ac_;
    std::unique_ptr<HiredisAsioAdapter> asio_adapter_;
};
```

#### 코루틴 브릿지 패턴

```cpp
// redisAsyncCommand의 콜백에서 코루틴을 resume하는 패턴:
awaitable<Result<redisReply*>> RedisConnection::command(const char* fmt, ...) {
    // 1. boost::asio::async_initiate 또는 experimental::use_promise
    // 2. redisAsyncCommand(ac_, callback, &promise_data, fmt, ...)
    // 3. callback에서 promise.set_value(reply)
    // 4. co_await promise → Result<redisReply*>
}
```

실제 구현은 `boost::asio::async_initiate`로 hiredis 콜백과 Asio completion token을 브릿지한다:

```cpp
template <typename CompletionToken>
auto async_redis_command(redisAsyncContext* ac,
                         const char* cmd,
                         CompletionToken&& token) {
    return boost::asio::async_initiate<CompletionToken,
                                        void(boost::system::error_code, redisReply*)>(
        [ac, cmd](auto handler) {
            auto* ctx = new CommandContext{std::move(handler)};
            redisAsyncCommand(ac, &command_callback, ctx, cmd);
        },
        token);
}
```

```cpp
// 핵심 구현 패턴:
HiredisAsioAdapter::HiredisAsioAdapter(io_context& io_ctx, redisAsyncContext* ac)
    : socket_(io_ctx), ac_(ac) {
    // fd를 Asio 소켓에 assign
    socket_.assign(boost::asio::ip::tcp::v4(), ac->c.fd);

    // hiredis ev 콜백 등록
    ac->ev.addRead = on_add_read;
    ac->ev.delRead = on_del_read;
    ac->ev.addWrite = on_add_write;
    ac->ev.delWrite = on_del_write;
    ac->ev.cleanup = on_cleanup;
    ac->ev.data = this;  // privdata로 this 전달
}
```

- [ ] **Step 3: hiredis_asio_adapter.cpp 구현**

```cpp
void HiredisAsioAdapter::handle_read() {
    if (!read_requested_) return;
    socket_.async_wait(
        boost::asio::ip::tcp::socket::wait_read,
        [this](boost::system::error_code ec) {
            if (!ec && read_requested_) {
                redisAsyncHandleRead(ac_);
                handle_read();  // 재등록 (연속 읽기)
            }
        });
}

void HiredisAsioAdapter::handle_write() {
    if (!write_requested_) return;
    socket_.async_wait(
        boost::asio::ip::tcp::socket::wait_write,
        [this](boost::system::error_code ec) {
            if (!ec && write_requested_) {
                redisAsyncHandleWrite(ac_);
                // write는 재등록하지 않음 — hiredis가 필요 시 addWrite 다시 호출
            }
        });
}
```

```cpp
// 핵심: redisAsyncCommand 콜백에서 코루틴 resume
struct CommandContext {
    using Handler = std::function<void(boost::system::error_code, redisReply*)>;
    Handler handler;
};

static void command_callback(redisAsyncContext*, void* reply, void* privdata) {
    auto* ctx = static_cast<CommandContext*>(privdata);
    auto* r = static_cast<redisReply*>(reply);
    if (r == nullptr) {
        ctx->handler(boost::asio::error::connection_reset, nullptr);
    } else {
        ctx->handler({}, r);
    }
    delete ctx;
}
```

### 설계

```cpp
/// 코어별 독립 Redis 커넥션 풀.
/// ConnectionPool CRTP를 상속하여 acquire/release/shrink/health_check 로직을 재사용.
/// 각 커넥션은 해당 코어의 io_context에 바인딩된 hiredis 비동기 커넥션.
class RedisPool : public ConnectionPool<RedisPool> {
public:
    using ConnectionType = std::unique_ptr<RedisConnection>;

    RedisPool(boost::asio::io_context& io_ctx,
              const RedisConfig& config,
              PoolConfig pool_config);
    ~RedisPool();

    // --- ConnectionPool CRTP 구현 ---
    ConnectionType do_create_connection();
    void do_destroy_connection(ConnectionType& conn);
    bool do_validate(ConnectionType& conn);

    /// 코루틴 thin wrapper: 풀에서 커넥션 acquire → 커맨드 실행 → release
    /// GET 명령어
    awaitable<Result<std::optional<std::string>>> get(std::string_view key);

    /// SET 명령어
    awaitable<Result<void>> set(std::string_view key,
                                 std::string_view value,
                                 std::chrono::seconds ttl = std::chrono::seconds{0});

    /// DEL 명령어
    awaitable<Result<int64_t>> del(std::string_view key);

    /// EXPIRE 명령어
    awaitable<Result<bool>> expire(std::string_view key, std::chrono::seconds ttl);

    /// 범용 커맨드 (redis-plus-plus 스타일 대비용)
    awaitable<Result<redisReply*>> execute(const char* format, ...);

private:
    boost::asio::io_context& io_ctx_;
    RedisConfig config_;
};
```

### 풀 동작 모델

```
서비스 코루틴:
  co_await redis_pool.get("session:1234")
    → pool.acquire()                    // ConnectionPool CRTP
      → idle_ 비었으면 do_create_connection()
        → RedisConnection::create(io_ctx_, config_)
          → redisAsyncConnect() + HiredisAsioAdapter 생성
      → idle_ 있으면 do_validate(conn)
        → conn->validate()  (PING)
    → conn->command("GET %s", key)     // 코루틴으로 결과 대기
    → pool.release(conn)               // ConnectionPool CRTP
    → Result<std::optional<std::string>> 반환
```

```cpp
///   Server({.port = 9000, .num_cores = 4})
///       .add_adapter<RedisAdapter>(redis_config)
///       .add_service<MyService>()
///       .run();

    expire(std::string_view key, std::chrono::seconds ttl);

    awaitable<Result<redisReply*>>
    execute(const char* format, ...);

    /// 현재 코어의 풀 상태 (모니터링용)
    [[nodiscard]] size_t active_connections() const noexcept;

private:
```

## 리스크 및 PoC 판단 기준

### 1순위 방식 (hiredis fd 직접 Asio 등록) PoC 판단

**성공 기준:**
- `redisAsyncConnect()` → `ac->c.fd` → `tcp::socket::assign()` → `async_wait(wait_read/write)` 체인이 Windows + Linux 모두에서 동작
- `redisAsyncCommand()` → 콜백 → 코루틴 resume 정상 동작
- `redisAsyncHandleRead/Write()` 호출 후 hiredis 내부 상태 정상

**실패 시 Fallback:**
- redis-plus-plus의 libuv async 모드 사용 (별도 이벤트 루프 스레드)
- MPSC 큐로 코어 스레드 ↔ libuv 스레드 통신
- 이 경우 `HiredisAsioAdapter`를 `RedisUvBridge`로 교체하되 `RedisConnection` 인터페이스는 유지

### Windows IOCP 주의사항

- hiredis가 Winsock `SOCKET`을 반환하므로 `tcp::socket::assign(tcp::v4(), (SOCKET)ac->c.fd)` 캐스트 필요할 수 있음
- IOCP에 이미 바인딩된 소켓은 `assign()` 불가 → hiredis가 소켓을 IOCP에 등록하지 않는지 확인 필요 (hiredis는 자체 이벤트 루프를 쓰지 않으므로 문제 없을 것으로 예상)
- PoC(Task 2)에서 Windows 빌드 + 실행까지 반드시 확인

### 커넥션 생명주기 주의

- `HiredisAsioAdapter` 소멸 시 `socket_.release()`로 fd를 반환하되 close하지 않아야 함 — hiredis가 `redisAsyncFree()`에서 fd를 close
- 순서: `redisAsyncDisconnect()` → hiredis cleanup 콜백 → `HiredisAsioAdapter` 소멸 → fd는 hiredis가 close
