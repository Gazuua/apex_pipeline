## 2. #91 SessionId 강타입화

### 현재 상태
```cpp
// session.hpp:23
using SessionId = uint64_t;
```

`corr_id`, `user_id`와 암묵적 변환 가능. 컴파일 타임 타입 안전성 없음.

### 설계

```cpp
// session.hpp
enum class SessionId : uint64_t {};

constexpr SessionId make_session_id(uint64_t v) noexcept { return static_cast<SessionId>(v); }
constexpr uint64_t to_underlying(SessionId id) noexcept { return static_cast<uint64_t>(id); }
```

- `enum class`이므로 `==`, `<=>` 비교 연산자는 기본 제공
- `std::hash<SessionId>` 특수화 추가 (unordered_map 키로 사용 중)
- `std::formatter<SessionId>` 또는 `operator<<` 특수화 (로깅용)

### Wire boundary 전략

Kafka envelope의 `MetadataPrefix::session_id`는 `uint64_t`로 직렬화된다.
직렬화 경계(wire boundary)에서는 `uint64_t`를 유지하고, 변환은 사용 지점에서 수행:
- `MetadataPrefix::session_id` — `uint64_t` 유지 (직렬화 코드 변경 최소화)
- 서비스 코드에서 사용 시 `make_session_id(meta.session_id)`로 변환
- `EnvelopeBuilder::metadata()` 파라미터도 `uint64_t` 유지, 호출부에서 `to_underlying()` 사용

**원칙**: 타입 안전성은 비즈니스 로직 레이어에서 강제, wire/직렬화 레이어는 원시 타입 유지.

### 영향 범위

~50곳 이상 기계적 치환:
- `apex_core/`: session.hpp, session_manager.hpp, connection_handler.hpp, server.hpp
- `apex_shared/`: 어댑터 (redis, kafka 관련 세션 ID 전달) — wire boundary는 `uint64_t` 유지, 사용 지점에서 `make_session_id()`/`to_underlying()` 변환
- `apex_services/`: gateway, auth-svc, chat-svc 전체

### 검증
빌드 성공 + 71 유닛 테스트 + 11 E2E 테스트 통과.

## 3. #89 core→shared 역방향 의존 해소

### 현재 상태

4개 파일이 core→shared 물리적 의존을 생성:

| 파일 (apex_core) | 형태 | 대상 (apex_shared) |
|---|---|---|
| `wire_header.hpp` | forwarding header | `shared/protocols/tcp/wire_header.hpp` |
| `frame_codec.hpp` | forwarding header | `shared/protocols/tcp/frame_codec.hpp` |
| `tcp_binary_protocol.hpp` | forwarding header | `shared/protocols/tcp/tcp_binary_protocol.hpp` |
| `service_base.hpp:11` | 직접 include | `shared/protocols/kafka/kafka_envelope.hpp` |

### 설계: Option A — WireHeader/FrameCodec core 복원

**근거**: v0.5 Wave 1에서 "프로토콜 구현을 shared로" 이동 시 WireHeader/FrameCodec까지 TcpBinaryProtocol과 함께 이동. 그러나 WireHeader(12바이트 와이어 포맷)와 FrameCodec(프레임 인코딩/디코딩)은 프로토콜 *구현*이 아니라 프레임워크 *인프라*. 이동 범위가 과했고, forwarding header라는 어색한 우회가 발생. 원래 설계 의도("core = concept + 공통 인프라, shared = 프로토콜 구현체")에 부합하는 복원.

#### 변경 사항

1. **WireHeader/FrameCodec 소스를 core로 복원**
   - `apex_core/include/apex/core/wire_header.hpp` — forwarding → 실제 구현으로 교체
   - `apex_core/include/apex/core/frame_codec.hpp` — forwarding → 실제 구현으로 교체
   - `apex_core/src/wire_header.cpp`, `apex_core/src/frame_codec.cpp` — 구현 복원
   - 네임스페이스: `apex::core`

2. **forwarding header 삭제**
   - `apex_core/include/apex/core/tcp_binary_protocol.hpp` 삭제
   - TcpBinaryProtocol은 shared에 유지 (프로토콜 *구현체*이므로 정당한 위치)

3. **shared 쪽 정리**
   - `apex_shared/lib/protocols/tcp/` — WireHeader/FrameCodec 소스 삭제, TcpBinaryProtocol만 잔류
   - TcpBinaryProtocol이 `apex::core::WireHeader`, `apex::core::Frame` include (shared→core 방향, 정상)

4. **service_base.hpp kafka_envelope.hpp 의존 해소**
   - core에 경량 구조체 `KafkaMessageMeta` 정의 (`session_id: uint64_t`, `corr_id: uint64_t`, `source_id: uint16_t` 등 최소 필드 — MetadataPrefix와 1:1 대응, 문자열 변환 불필요)
   - `KafkaHandler` 시그니처: `MetadataPrefix` → `KafkaMessageMeta` 사용
   - `kafka_route<T>()` 핸들러 시그니처도 `(const KafkaMessageMeta&, uint32_t, const FbsType*)` 로 변경
   - KafkaDispatchBridge가 `MetadataPrefix`를 파싱 후 `KafkaMessageMeta`로 변환하여 핸들러에 전달
   - `service_base.hpp`에서 `kafka_envelope.hpp` include 제거 — core는 `KafkaMessageMeta`만 알면 됨

5. **CMakeLists.txt 소스 목록 갱신** (core에 소스 추가, shared에서 제거)

### 결과
core→shared 물리적 의존 **0건**. 의존 방향: shared→core 단방향.

## 4. #3 Protocol concept FrameType 제약

### 현재 상태
```cpp
// protocol.hpp
template <typename P>
concept Protocol = requires {
    typename P::Config;
    typename P::Frame;
} && requires(RingBuffer& buf, const typename P::Frame& frame) {
    { P::try_decode(buf) } -> std::same_as<Result<typename P::Frame>>;
    { P::consume_frame(buf, frame) } -> std::same_as<void>;
};
```

Frame 내부 구조 미제약. `msg_id`, `payload` 접근 보장 없음.

### 설계

**설계 결정**: `FrameType` concept은 `payload()`만 요구한다. `msg_id()`는 요구하지 않는다.

**근거**: WebSocket Frame에는 msg_id 개념 자체가 없다. WebSocket의 msg_id는 payload 첫 4바이트에서 추출하는 별도 로직이며, 이는 Frame 디코딩이 아니라 디스패치 시점의 관심사다. `msg_id()` accessor를 강제하면 WebSocket Frame에서 payload를 매번 파싱해야 하는 부자연스러운 적합이 된다.

```cpp
// protocol.hpp
template <typename F>
concept FrameType = requires(const F& f) {
    { f.payload() } -> std::convertible_to<std::span<const uint8_t>>;
};

template <typename P>
concept Protocol = requires {
    typename P::Config;
    typename P::Frame;
} && FrameType<typename P::Frame>
  && requires(RingBuffer& buf, const typename P::Frame& frame) {
    { P::try_decode(buf) } -> std::same_as<Result<typename P::Frame>>;
    { P::consume_frame(buf, frame) } -> std::same_as<void>;
};
```

msg_id 추출은 connection_handler의 기존 `if constexpr` 프로토콜별 분기에서 처리 유지.
목표는 `if constexpr` 분기 제거가 아니라 **정적 타입 제약 추가**. 프로토콜마다 msg_id 추출 방식이 다를 수 있다는 현실을 수용한다.

#### 변경 사항

1. **Frame 구조체에 `payload()` accessor 추가** (#89에서 core로 복원된 Frame)
   - TCP Frame: `payload()` 반환 (기존 `frame.payload` 멤버 래핑)
   - 기존 public 멤버는 유지 (하위 호환)

2. **WebSocketProtocol::Frame에도 `payload()` accessor 추가**

3. **connection_handler.hpp에서 `payload()` accessor 사용 전환**
   - `frame.payload` → `frame.payload()` 패턴
   - msg_id 추출은 기존 `if constexpr` 분기 유지 (`frame.header.msg_id` for TCP, payload 파싱 for WebSocket)

4. **MockProtocol::Frame 갱신** — static_assert용 `payload()` accessor 추가

## 5. #66 CoreEngine tracked spawn

### 현재 상태
```cpp
// kafka_adapter.cpp:184
boost::asio::co_spawn(engine.io_context(0), [...], boost::asio::detached);
```

`co_spawn(detached)` → D7 outstanding 코루틴 추적 우회. shutdown 시 미완료 코루틴 가능.

### 설계: Option C — CoreEngine에 tracked spawn API

**근거**: wire_services()의 코루틴은 서비스 코루틴이 아니라 인프라 코루틴 (Kafka dispatch bridge 역할). 엔진 레벨 추적이 의미론적으로 적합. 시그니처 변경 최소화.

#### 변경 사항

1. **CoreEngine에 tracked spawn API 추가**
```cpp
// core_engine.hpp
template <typename F>
void spawn_tracked(uint32_t core_id, F&& coro_factory);
```
   - `core_id`로 대상 io_context를 내부에서 결정 (io_context 직접 전달 금지 — 잘못된 io_context 전달 위험 차단)
   - outstanding 카운터 관리 (atomic increment → 코루틴 실행 → decrement)
   - 예외 안전 (catch + 로깅)
   - shutdown 시 outstanding 코루틴 완료 대기

2. **kafka_adapter.cpp 수정**
   - `co_spawn(engine.io_context(0), ..., detached)` → `engine.spawn_tracked(0, ...)`

3. **ServiceBase::spawn()과 독립 유지**
   - 서비스 수준 추적(ServiceBase)과 엔진 수준 추적(CoreEngine)은 역할이 다름
   - **shutdown 순서**: (1) 새 코루틴 스폰 중단 (2) 서비스 코루틴(ServiceBase) 완료 대기 (3) 인프라 코루틴(CoreEngine) 완료 대기
   - 근거: 서비스 핸들러가 처리 중인 메시지를 완료한 후에 인프라 브릿지를 중단해야 응답 유실 방지

## 6. #56 ServiceBase io_context 캡슐화

### 현재 상태
`ServiceBase::io_ctx_`가 protected. 서비스가 `co_spawn(*io_ctx_, ...)` 직접 호출 가능.

### 설계

1. **`io_ctx_`를 private으로 이동**

2. **공개 API 추가 (protected)**
```cpp
// service_base.hpp (protected)
template <typename F>
void post(F&& fn);                                      // io_context에 작업 게시 (type erasure 없이 직접 전달)
boost::asio::any_io_executor get_executor() noexcept;   // timer 등에 필요
// spawn()은 이미 존재
```

`post()`를 `std::function` 대신 템플릿으로 정의하여 heap allocation 오버헤드 회피. `spawn()`과 일관된 패턴.

3. **서비스 코드 변경: 0건**
   - Gateway, Auth, Chat 모두 `io_ctx_` 직접 접근 없음 (확인 완료)
   - `FileWatcher`, `ConfigReloader` 등은 자체 `io_context&` 파라미터를 받는 별도 유틸리티

4. **컴파일 타임 가드레일**: 향후 서비스 개발자가 `io_ctx_` 직접 접근 시 컴파일 에러 → `spawn()` / `post()` / `get_executor()` 사용 유도

## 7. #90 ErrorCode Gateway 에러 분리

### 현재 상태
core `ErrorCode` enum에 Gateway 전용 에러 11개(100-199)가 혼입.
새 서비스 추가 시 core 수정 필요 (레이어 경계 위반).

### 설계: Option A — ServiceError sentinel + 서비스별 에러 enum

**에러 전달 구조**:
```
[프레임워크 에러] ErrorCode 하나로 완결
  ErrorCode::InvalidMessage (2) → 끝. service_error_code = 0 (무시)

[서비스 에러] ErrorCode::ServiceError (99) + service_error_code 조합
  ErrorCode::ServiceError (99) + GatewayError::JwtExpired (3)
  ErrorCode::ServiceError (99) + AuthError::JwtVerifyFailed (1)
```

#### 변경 사항

1. **core ErrorCode 수정**
   - Gateway 전용 코드 11개(100-199) 삭제
   - `ServiceError = 99` sentinel 추가 (기존 Gateway 에러 대역 100-199와 겹치지 않는 값)
   - `error_code_name()` 갱신
   - `AppError(1000)`, `AdapterError(2000+)`는 이번에 변경하지 않음 — 이유: 어댑터 에러는 프레임워크 인프라의 일부이며 서비스 독립 에러가 아님. 향후 서비스가 자체 어댑터 에러를 정의해야 할 때 `ServiceError` 패턴으로 전환 검토 (현재는 불필요)

2. **GatewayError enum 신설**
```cpp
// apex_services/gateway/include/apex/gateway/gateway_error.hpp
namespace apex::gateway {
enum class GatewayError : uint16_t
{
    ConfigParseFailed = 1,
    JwtVerifyFailed = 2,
    JwtExpired = 3,
    JwtBlacklisted = 4,
    RouteNotFound = 5,
    ServiceTimeout = 6,
    PendingMapFull = 7,
    RateLimitedIp = 8,
    RateLimitedUser = 9,
    RateLimitedEndpoint = 10,
    SubscriptionLimitExceeded = 11,
};
} // namespace apex::gateway
```

서비스별 에러 enum의 값 공간은 완전히 독립 — 1부터 시작, 기존 100 대역 유지 불필요.

3. **에러 응답 전송 경로**
   - `ErrorSender::build_error_frame()`에 optional `service_error_code` 파라미터 추가
   - FlatBuffers `ErrorEnvelope` 스키마에 `service_error_code: uint16` 필드 추가 (테이블 끝에 추가하여 하위 호환 유지 — 기존 메시지에서는 0으로 해석)

4. **서비스 코드 수정**
   - Gateway: `ErrorCode::JwtVerifyFailed` → `GatewayError::JwtVerifyFailed` 전환 (11곳)
   - Auth: `ErrorCode::JwtVerifyFailed` → `AuthError` enum 신설 + 이동 (1곳)
   - Chat: 영향 없음 (프레임워크 에러만 사용)

5. **E2E 테스트 수정**
   - 에러 코드 검증 테스트에서 `ServiceError` sentinel + 서비스 에러 코드 조합 확인으로 변경
