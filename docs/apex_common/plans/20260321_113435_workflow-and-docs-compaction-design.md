# BACKLOG-110, 111 설계 — 단계 기반 워크플로우 + CLAUDE.md 컴팩션

## 개요

두 백로그를 하나의 브랜치에서 진행한다.

- **BACKLOG-110**: 작업 종류를 분류하는 대신, 공통 단계 풀과 스킵 조건을 정의하여 모든 작업에 일관된 절차를 부여
- **BACKLOG-111**: CLAUDE.md 6개 파일의 구조는 유지하되 내용을 컴팩션 (중복 제거, 원본 참조 전환, 간결화)

실행 순서: #110 먼저 → #111. 단계 정의가 컴팩션의 기준점이 된다.

### 워크플로우 단계의 위치

루트 CLAUDE.md에 `## 워크플로우` 섹션을 신설하여 단계 풀과 스킵 조건을 기술한다. 위치는 `## 개요` 직후, `## 모노레포 구조` 직전. 기존 `### Git / 브랜치` 섹션의 머지 절차(6단계)는 빌드 스킵 조건 등 세부 규칙을 포함하므로 Git 섹션에 상세를 유지하고, 워크플로우 ⑦에서 포인터로 참조한다.

## 1. 단계 풀 정의 (#110)

### 1.1 공통 7단계

| # | 단계 | 핵심 행위 | 산출물 |
|---|------|-----------|--------|
| ① | **착수** | 브랜치 생성, 백로그 확인, 핸드오프 backlog-check | 브랜치 |
| ② | **설계** | 브레인스토밍 → 스펙 문서 | `docs/{project}/plans/` 문서 |
| ③ | **구현** | 코드 작성, clang-format | 소스 변경 |
| ④ | **검증** | 빌드 + 테스트 (Debug, 필요시 Release/ASAN/TSAN) | 빌드 성공 |
| ⑤ | **리뷰** | auto-review 실행, 이슈 수정 | `docs/{project}/review/` 문서 |
| ⑥ | **문서 갱신** | CLAUDE.md, Apex_Pipeline.md, BACKLOG.md, README, progress 등 | 갱신된 문서 |
| ⑦ | **머지** | lock 획득 → rebase → (빌드) → push → squash merge → lock 해제 → 브랜치 정리 | main에 머지 |

### 1.2 스킵 조건

| 단계 | 스킵 가능 조건 |
|------|---------------|
| ② 설계 | 변경이 단순하고 설계 판단이 불필요 (오타 수정, 1파일 이하 기계적 변경, 문서 전용) |
| ③ 구현 | 문서 전용 작업 |
| ④ 검증 | 문서 전용 작업 (코드 변경 없음) |
| ⑤ 리뷰 | 문서 전용 작업, 또는 변경 범위가 극히 작아 리뷰 ROI가 없는 경우 |
| ①⑥⑦ | **스킵 불가** — 모든 작업에 필수 |

판단 주체: 에이전트가 착수 시 스킵 조건을 평가하고, 이번 작업에 필요한 단계를 체크리스트로 생성.

### 1.3 체크리스트 메커니즘

착수 시 에이전트가 스킵 조건을 평가하여 TaskCreate로 체크리스트를 생성한다.

**코드 변경이 있는 일반 작업 예시:**

```
☐ ① 착수: 브랜치 생성, 백로그 확인
☐ ② 설계: 브레인스토밍 → 스펙 문서
☐ ③ 구현: 코드 작성 + clang-format
☐ ④ 검증: Debug 빌드 + 테스트
☐ ⑤ 리뷰: auto-review 실행 + 이슈 수정
☐ ⑥ 문서 갱신: CLAUDE.md, Apex_Pipeline.md 등
☐ ⑦ 머지: lock → rebase → push → squash merge
```

**문서 전용 작업 예시:**

```
☐ ① 착수: 브랜치 생성, 백로그 확인
  ② 설계: 스킵 (문서 전용)
  ③ 구현: 스킵 (코드 변경 없음)
  ④ 검증: 스킵 (코드 변경 없음)
  ⑤ 리뷰: 스킵 (문서 전용)
☐ ⑥ 문서 갱신: 대상 문서 작성/수정
☐ ⑦ 머지: lock → rebase → push → squash merge
```

### 1.4 강제 메커니즘

기존 hook 4개를 유지한다. 새 hook은 추가하지 않는다.

Hook은 **도구 호출 시** 발동한다 (단계가 아닌 tool matcher 기반). 아래 표의 "관련 단계"는 해당 hook이 가장 밀접한 워크플로우 단계를 나타낸다.

| Hook | Matcher | 관련 단계 | 동작 |
|------|---------|-----------|------|
| `validate-build.sh` | Bash | ④ 검증 | 빌드 도구 직접 호출 차단 → queue-lock 경유 강제 |
| `validate-merge.sh` | Bash | ⑦ 머지 | merge lock 미획득 시 `gh pr merge` 차단 |
| `validate-handoff.sh` | Bash | ①③⑦ | 매 Bash 호출 시 핸드오프 probe + merge gate + 브랜치 생성 리마인더 |
| `handoff-probe.sh` | Edit\|Write | ③⑥ | 모든 Edit/Write 시 핸드오프 probe, 소스 파일(.cpp/.hpp/.h) 편집 시 미ack 알림이면 차단 |

나머지 단계(설계, 리뷰)는 체크리스트 기반 자기 관리.

## 2. CLAUDE.md 컴팩션 (#111)

### 2.1 원칙

1. **원본이 있는 정보는 삭제** — vcpkg 의존성 목록(vcpkg.json), compose 구조도(yml), 환경변수 표(코드) 등. 원본과 문서 어긋남 리스크 제거
2. **중복은 한 곳으로** — 빌드 규칙은 루트에, 백로그 상세 규칙은 docs/CLAUDE.md에, 경고 정책은 루트에
3. **단순 규칙은 1줄로** — 저작권 헤더, 파일명 규칙 등 판단이 필요 없는 규칙. 다단계 절차 규칙은 현행 유지

### 2.2 파일별 방향

| 파일 | 현재 줄수 | 방향 | 예상 감소 |
|------|-----------|------|-----------|
| **루트 CLAUDE.md** | 123 | 워크플로우 단계 섹션 추가 (+20줄), 중복 제거·간결화 (-30줄) | 순감 ~10줄 |
| **docs/CLAUDE.md** | 112 | 백로그 운영 규칙(~70줄)은 권위 소스로 유지 (컴팩션 대상 아님). 코드 리뷰·브레인스토밍 섹션의 루트 중복분 삭제 | -15~20줄 |
| **apex_core/CLAUDE.md** | 53 | 의존성 목록 삭제, 루트와 겹치는 빌드 규칙 삭제 | -10~15줄 |
| **apex_tools/CLAUDE.md** | 34 | 루트와 겹치는 빌드 규칙 삭제 | -5줄 |
| **.github/CLAUDE.md** | 15 | 이미 간결. 변경 미미 | ~0 |
| **e2e/CLAUDE.md** | 165 | compose 구조도·환경변수 표·트러블슈팅 상세 삭제 (원본 참조로 대체) | -50~70줄 |

**총 예상**: 502줄 → ~390줄 (약 20~25% 감소), 워크플로우 단계 추가 포함.

### 2.3 진행 방식

파일별로 "이 부분 줄이겠다" → 사용자 확인 → 적용. 일괄 변경 없이 하나씩 합의하면서 진행.
