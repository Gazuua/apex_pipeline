## 2. 코어 메모리 아키텍처

### 2.1 설계 원칙

1. **Per-core 격리**: 각 할당기는 코어당 1개, thread-safety 불필요 (io_context 보장)
2. **수명 기반 계층화**: 객체 수명에 맞는 할당기 선택
3. **독립 chunk**: 각 할당기가 자체 malloc으로 chunk 확보 (공유 chunk 없음)
4. **concept 기반 다형성**: CRTP 대신 C++20 concept으로 인터페이스 제약
5. **점진적 도입**: 할당기별 독립 테스트·벤치마크 가능

### 2.2 할당기 선택 근거

브레인스토밍에서 5종(Bump, Arena, Slab, Buddy, mimalloc heap)을 비교 분석한 결과:

| 할당기 | 채택 | 역할 | 사유 |
|--------|------|------|------|
| **Bump** | O | 요청/코루틴 수명 임시 데이터 | 최고 성능 (포인터++), 개별 free 불필요한 패턴에 최적 |
| **Slab** | O | 고정 크기 장수명 객체 | O(1) 개별 free, Session/코루틴 프레임에 적합. 기존 구현 개선 |
| **Arena** | O | 트랜잭션 수명 데이터 | 블록 체이닝 + 벌크 해제, PG 트랜잭션 가드와 자연 결합 |
| Buddy | X | — | O(log n) 성능, 내부 단편화 심각, 서버 유저스페이스 부적합 |
| mimalloc heap | X (현재) | — | 범용 fallback으로 나중에 추가 가능. 현재는 3종으로 충분 |

**레퍼런스**: Seastar (ScyllaDB), DragonflyDB, PostgreSQL MemoryContext, 게임 엔진 Frame Allocator 등 프로덕션 시스템들이 수명별 할당기 계층화를 사용.

### 2.3 concept 정의

```cpp
namespace apex::core {

/// 모든 할당기가 만족하는 공통 concept.
template <typename T>
concept CoreAllocator = requires(T a, void* ptr) {
    { a.allocate(std::size_t{}, std::size_t{}) } -> std::same_as<void*>;
    { a.owns(ptr) }       -> std::same_as<bool>;
    { a.used_bytes() }    -> std::convertible_to<std::size_t>;
    { a.capacity() }      -> std::convertible_to<std::size_t>;
};

/// 개별 해제 가능 (SlabAllocator).
template <typename T>
concept Freeable = CoreAllocator<T> && requires(T a, void* ptr) {
    { a.deallocate(ptr) } -> std::same_as<void>;
};

/// 벌크 리셋 가능 (BumpAllocator, ArenaAllocator).
template <typename T>
concept Resettable = CoreAllocator<T> && requires(T a) {
    { a.reset() } -> std::same_as<void>;
};

} // namespace apex::core
```

**allocate 시그니처**: `void* allocate(size_t size, size_t align = alignof(std::max_align_t))`.
- 기본 alignment는 `alignof(std::max_align_t)` (MSVC/GCC 공통 16바이트)
- 호출자가 명시적으로 낮은 alignment를 지정 가능 (예: `char` 버퍼는 `align=1`)
- concept은 2-파라미터 시그니처를 요구하되, 구현에서 default argument 제공

**MSVC 주의**: chunk 할당 시 `std::aligned_alloc` 미지원. `malloc` 반환값은 `alignof(max_align_t)` 보장이므로 chunk 자체는 추가 정렬 불필요. chunk 내부 bump/slab 슬롯의 정렬은 할당기 내부에서 포인터 연산으로 처리.

### 2.4 BumpAllocator

**역할**: 요청/코루틴 수명의 임시 데이터. 매 요청 완료 시 `reset()`으로 전체 초기화.

**만족 concept**: `CoreAllocator` + `Resettable`

**구현 핵심**:
- 기동 시 `malloc(capacity)`로 단일 chunk 확보 (default 64KB, TOML 설정 가능)
- `allocate(size, align)`: cursor를 alignment 경계로 올린 후 size만큼 전진
- `reset()`: cursor를 시작 위치로 복원 (O(1))
- 개별 `deallocate()` 없음
- chunk 초과 시: `nullptr` 반환 (auto-grow 없음, 크기 설정으로 대응)

**적용 대상**: PG `query_params_async`의 SQL 복사 + 파라미터 배열 (#5 해결)

**파일 위치**: `apex_core/include/apex/core/bump_allocator.hpp`, `apex_core/src/bump_allocator.cpp`

### 2.5 SlabAllocator (기존 SlabPool 리네임 + 개선)

**역할**: 고정 크기 장수명 객체 (Session, 코루틴 프레임 등).

**만족 concept**: `CoreAllocator` + `Freeable`

**변경점 (#1 해결)**:
- `SlabPool` → `SlabAllocator` 리네임 (TypedSlabPool → TypedSlabAllocator 포함)
- Release 빌드에서도 double-free 감지 활성화:

```cpp
void SlabAllocator::deallocate(void* ptr) noexcept {
    if (!ptr) return;
    auto* node = static_cast<FreeNode*>(ptr);
    if (node->magic == SLAB_MAGIC_FREED) {
        // Release에서도 동작: 카운터 증가 + early return
        ++double_free_count_;  // 모니터링용 메트릭
        return;
    }
    node->magic = SLAB_MAGIC_FREED;
    node->next = free_list_;
    free_list_ = node;
    ++free_count_;
}
```

- `double_free_count()` 메트릭 API 추가
- 기존 auto-grow, peak_allocated 등 기능 유지
- **CoreAllocator concept 충족을 위한 API 추가**:
  - `used_bytes()` = `allocated_count() * slot_size_`
  - `capacity()` = `total_count() * slot_size_`
  - `allocate(size, align)`: size는 무시 (slot_size_ 고정), align은 생성 시 보장

**double-free 감지 한계**: magic marker(`SLAB_MAGIC_FREED = 0xF2EED000`) 기반이므로, T의 사용 중 데이터가 FreeNode 영역(첫 `sizeof(FreeNode)` 바이트)을 우연히 같은 값으로 덮어쓰면 감지 실패 가능. 이는 **best-effort 감지**이며, `double_free_count_`는 하한값(최소 감지 횟수)이다. 향후 별도 비트맵 기반 free 상태 추적으로 강화 가능.

**파일 위치**: 기존 `slab_pool.hpp/cpp` → `slab_allocator.hpp/cpp` 리네임

### 2.6 ArenaAllocator

**역할**: 트랜잭션 수명 데이터. 블록 체이닝으로 가변 크기 할당, 벌크 해제.

**만족 concept**: `CoreAllocator` + `Resettable`

**구현 핵심**:
- 블록(default 4KB, TOML 설정 가능) 단위로 `malloc` 체이닝
- 블록 내부는 bump 방식 (포인터 전진, alignment 보정)
- 블록 초과 시 새 블록 할당 후 체인에 추가
- `reset()`: 전체 블록 체인 해제 (첫 블록만 유지하여 재사용 가능)
- `max_bytes` 설정: 전체 블록 합계 상한 (default 1MB). 초과 시 `nullptr` 반환.

**사용 제약**: ArenaAllocator는 개별 해제(`deallocate`) 불가. 트랜잭션 수명이 짧을수록 효과적. 장시간 트랜잭션에서는 메모리가 `reset()`까지 누적되므로, `max_bytes` 설정으로 상한을 제어해야 한다.

**적용 대상**: PG 트랜잭션 가드(`PgTransaction`)와 결합 (#6과 연계)

**파일 위치**: `apex_core/include/apex/core/arena_allocator.hpp`, `apex_core/src/arena_allocator.cpp`

## 3. PostgreSQL 어댑터 개선

### 3.1 트랜잭션 가드 (#6)

RAII 스타일 트랜잭션 관리.

```cpp
class PgTransaction {
public:
    PgTransaction(PgConnection& conn, PgPool& pool);
    ~PgTransaction();

    [[nodiscard]] awaitable<Result<PgResult>>
    execute(std::string_view sql);

    [[nodiscard]] awaitable<Result<PgResult>>
    execute_params(std::string_view sql, std::span<const std::string> params);

    [[nodiscard]] awaitable<Result<void>> commit();
    [[nodiscard]] awaitable<Result<void>> rollback();

private:
    PgConnection& conn_;
    PgPool& pool_;
    bool finished_{false};
};
```

**진입점**: `PgConnection::begin_transaction(PgPool& pool)` → `co_await`로 `BEGIN` 실행 후 `PgTransaction` 반환.

**소멸자 전략 — poisoned 커넥션 폐기**:

코루틴 소멸자에서 `co_await` 불가하므로, 동기 `PQsendQuery("ROLLBACK")` 방식은 Asio io_context와 I/O 충돌 위험이 있다 (libpq와 Asio가 동일 fd에 동시 접근). 대신 **poisoned 전략**을 사용:

```cpp
PgTransaction::~PgTransaction() {
    if (!finished_) {
        // 커넥션을 poisoned로 마킹 → 풀에 반환하지 않고 폐기
        conn_.mark_poisoned();
        // PgPool::release()에서 poisoned 커넥션은 discard() 호출
        // → do_destroy_connection() → PQfinish()로 서버 측 세션 종료
        // → 서버가 미커밋 트랜잭션을 자동 ROLLBACK
    }
}
```

- `PgConnection::mark_poisoned()` / `is_poisoned()` API 추가
- `PgPool::release()`: poisoned 커넥션은 `discard()`로 폐기 (idle 큐에 반환하지 않음)
- PostgreSQL 서버는 클라이언트 연결 종료 시 미커밋 트랜잭션을 자동 ROLLBACK (PG 프로토콜 보장)
- PgBouncer transaction 모드에서도 동일하게 동작

**파일 위치**: `apex_shared/lib/adapters/pg/include/apex/shared/adapters/pg/pg_transaction.hpp`, `apex_shared/lib/adapters/pg/src/pg_transaction.cpp`

### 3.3 query_params_async malloc 제거 (#5)

`PgConnection` 생성 시 `BumpAllocator*`를 주입. 주입되면 bump에서 할당, nullptr이면 기존 방식(vector) fallback.

```cpp
PgConnection(boost::asio::io_context& io_ctx,
             std::string conninfo,
             BumpAllocator* request_alloc = nullptr);
```

**변경되는 핫패스**:
```cpp
// Before: 매 호출 힙 할당 2회
std::string sql_str(sql);
std::vector<const char*> param_values;

// After: BumpAllocator에서 포인터 전진만
char* sql_buf = static_cast<char*>(
    request_alloc_->allocate(sql.size() + 1, 1));  // align=1 (char 배열)
std::memcpy(sql_buf, sql.data(), sql.size());
sql_buf[sql.size()] = '\0';

auto* param_buf = static_cast<const char**>(
    request_alloc_->allocate(
        params.size() * sizeof(const char*),
        alignof(const char*)));  // align=8 (포인터)
```

## 4. Redis 어댑터 재설계

### 4.1 ConnectionPool → RedisMultiplexer 전환 (#3)

기존 acquire/release 모델을 폐기하고, 코어당 고정 커넥션 + 코루틴 멀티플렉싱으로 전환.

**구조**:
```
RedisAdapter (AdapterBase CRTP 유지)
  └─ 코어당 RedisMultiplexer 1개 (do_init에서 생성)
       └─ 고정 hiredis async 커넥션 1~2개
            └─ 여러 코루틴이 동시에 명령 제출 → 내부 큐잉 → 파이프라이닝
```

### 4.2 RedisMultiplexer

```cpp
class RedisMultiplexer {
public:
    RedisMultiplexer(boost::asio::io_context& io_ctx, const RedisConfig& config);

    /// 단일 명령. 내부적으로 파이프라이닝됨.
    [[nodiscard]] awaitable<Result<RedisReply>> command(std::string_view cmd);

    /// 다중 명령 배치.
    [[nodiscard]] awaitable<Result<std::vector<RedisReply>>>
    pipeline(std::span<const std::string> commands);

    [[nodiscard]] bool connected() const noexcept;
    awaitable<void> close();

private:
    struct PendingCommand {
        boost::asio::steady_timer resolver;   // co_await 대기용
        boost::asio::steady_timer timeout;    // 명령별 타임아웃
        Result<RedisReply> result;
    };

    void on_disconnect();                     // hiredis disconnectCallback
    awaitable<void> reconnect_loop();         // 지수 백오프 재연결

    std::deque<PendingCommand*> pending_;     // FIFO — Redis 응답 순서 보장
    std::unique_ptr<RedisConnection> conn_;
    boost::asio::io_context& io_ctx_;
    RedisConfig config_;

    // Reconnection 상태
    bool reconnecting_{false};
    uint32_t reconnect_attempts_{0};
};
```

**동작 원리**:
1. `command()` 호출 → `PendingCommand` 생성, `pending_` 큐에 push
2. `redisAsyncCommand()` 호출 (hiredis가 내부 write 버퍼에 큐잉)
3. 코루틴 suspend (`co_await resolver`)
4. hiredis callback → `pending_.front()` 결과 설정 → resolver cancel → 코루틴 resume
5. Redis 프로토콜이 응답 순서를 보장하므로 FIFO 매칭 가능

### 4.3 Reconnection 전략

기존 ConnectionPool은 `do_validate()` + `do_create_connection()`으로 불량 커넥션을 교체했지만, Multiplexer는 고정 커넥션이므로 별도 전략이 필요.

**커넥션 끊김 감지**: hiredis `redisAsyncSetDisconnectCallback` 등록.

**재연결 정책**:
- 지수 백오프: 100ms → 200ms → 400ms → ... → 최대 30초
- 최대 재연결 시도: TOML 설정 (default: 무제한, 서킷 브레이커 없음 — Redis 복구 대기)
- 재연결 중 pending 명령: **즉시 에러 반환** (`ErrorCode::AdapterNotReady`). 재시도는 서비스 코드 책임.
- 재연결 성공 시: `reconnect_attempts_` 리셋, 로그 기록

**명령별 타임아웃**: `RedisConfig::command_timeout` (기존 설정 유지, default 1000ms). 타임아웃 시:
- 해당 PendingCommand에 에러 설정 → 코루틴 resume
- **후속 명령은 영향 없음**: 타임아웃된 명령의 Redis 응답이 늦게 도착하면 pending_ 큐에서 이미 제거된 상태이므로 무시 처리 (응답 sequence 번호로 매칭)

### 4.4 RedisAdapter

```cpp
class RedisAdapter : public AdapterBase<RedisAdapter> {
public:
    explicit RedisAdapter(RedisConfig config);  // 기존 RedisConfig 유지
```

### 4.5 삭제 대상

- `RedisPool` 클래스 전체

## 6. ConnectionPool concept 전환 (#8)

### 6.1 CRTP 제거

```cpp
// 삭제: ConnectionPool<Derived, Conn> CRTP base

// 추가: concept (동기 풀 기본 계약)
template <typename T>
concept PoolLike = requires(T pool) {
    { pool.acquire() }
        -> std::same_as<apex::core::Result<typename T::Connection>>;
    { pool.release(std::declval<typename T::Connection>()) }
        -> std::same_as<void>;
    { pool.discard(std::declval<typename T::Connection>()) }
        -> std::same_as<void>;
    { pool.close_all() } -> std::same_as<void>;
    { pool.stats() }     -> std::convertible_to<PoolStats>;
};
```

**scope 명시**: `PoolLike`는 동기 acquire/release 기본 계약을 정의한다. `PgPool::acquire_connected()` 같은 비동기 확장은 PgPool 고유 API이며 concept의 범위 밖이다. concept은 "풀이라면 최소한 이것을 지원해야 한다"는 제약이지, 모든 API를 포괄하지 않는다.

### 6.2 PgPool 자체 완결화

ConnectionPool base의 공통 로직(`acquire`, `release`, `discard`, `shrink_idle`, `health_check_tick` 약 120줄)을 PgPool로 인라인 이동.

**추가**: `release()` 내부에서 poisoned 커넥션 체크:
```cpp
void PgPool::release(Connection conn) {
    if (conn && conn->is_poisoned()) {
        discard(std::move(conn));  // 풀에 반환하지 않고 폐기
        return;
    }
    // 기존 release 로직...
}
```

### 6.3 PoolStats 유지 (기존 구조 보존)

현재 코드베이스의 `PoolStats`를 그대로 유지:

```cpp
struct PoolStats {
    uint64_t total_acquired = 0;
    uint64_t total_released = 0;
    uint64_t total_created = 0;
    uint64_t total_destroyed = 0;
    uint64_t total_failed = 0;
};
```

기존 필드와의 호환성을 보장하여 테스트/모니터링 코드 변경을 최소화한다.

**파일 변경**:
- 삭제: `apex_shared/lib/adapters/common/include/apex/shared/adapters/connection_pool.hpp`
- 수정: `apex_shared/lib/adapters/pg/include/apex/shared/adapters/pg/pg_pool.hpp` (자체 완결)
- 추가: `apex_shared/lib/adapters/common/include/apex/shared/adapters/pool_concept.hpp` (PoolLike concept + PoolStats + PoolConfig)

## 10. 테스트 전략

### 단위 테스트 (신규)

- **BumpAllocator**: allocate/reset/overflow/alignment 경계/다양한 align 값
- **ArenaAllocator**: allocate/reset/block chaining/overflow/max_bytes 초과
- **SlabAllocator**: 기존 테스트 유지 + double-free Release 감지 + used_bytes/capacity API
- **SlabAllocator 감지 한계 테스트**: FreeNode 영역을 의도적으로 SLAB_MAGIC_FREED 값으로 덮어쓴 후 감지 실패 확인 (best-effort 문서화)
- **PgTransaction**: commit/rollback/소멸자 poisoned 폐기
- **RedisMultiplexer**: command/pipeline/응답 순서 보장/명령 타임아웃
- **RedisMultiplexer reconnection**: 끊김 감지/지수 백오프/재연결 중 에러 반환
- **CoreMetrics**: post_failures 카운터 증가
- **PoolLike concept**: static_assert로 PgPool이 concept 만족하는지 컴파일 타임 검증

### 기존 테스트 수정

- SlabPool → SlabAllocator 리네임 반영
- ConnectionPool 관련 Redis 테스트 → RedisMultiplexer 테스트로 교체
- PgPool 테스트: CRTP 제거 후 동일 동작 확인 + discard/poisoned 시나리오 추가

### 통합 테스트

- PG: 트랜잭션 가드 + Prepared Statement E2E
- Redis: 멀티플렉싱 동시 명령 E2E + 재연결 시나리오
- 기존 통합 테스트 4개 회귀 확인
