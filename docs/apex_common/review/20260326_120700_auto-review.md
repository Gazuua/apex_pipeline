## 2. 설계 분석

### 2.1 SocketBase Interface 설계 (PASS)

인터페이스가 필요 최소한의 pure virtual 메서드로 구성되어 있어 깔끔하다:

- `async_read_some(mutable_buffer)` — 비동기 읽기
- `async_write(const_buffer)` — 비동기 쓰기 (전량 전송 보장)
- `async_handshake()` — TLS handshake (Plain TCP는 no-op)
- `close() noexcept` — 소켓 닫기
- `is_open() const noexcept` — 상태 확인
- `get_executor() noexcept` — executor 접근
- `set_option_no_delay(bool)` — TCP 옵션
- `remote_endpoint(ec)` — 원격 엔드포인트

**설계 판단 타당성:**
- virtual dispatch ~2ns는 커널 syscall 대비 무시 가능 (0.001% 미만) — 프레임워크 벤치마크에서도 확인 가능
- Session/SessionManager를 비템플릿으로 유지하여 헤더 파일 의존성 전파를 완전히 차단
- `async_handshake()`를 SocketBase에 배치한 것은 좋은 판단 — ConnectionHandler가 TLS/Plain을 구분할 필요 없이 일관된 호출 가능

### 2.2 Transport Concept 리디자인 (PASS)

기존 `TransportContext` + `make_socket(io_ctx, TransportContext&)` 구조에서 `ListenerState` + `wrap_socket(tcp::socket, ListenerState&)` 구조로 전환한 점이 핵심.

**변경의 의미:**
- `TransportContext`가 `ssl::context*`를 보유하던 것이 의존성 오염의 근원이었음
- `ListenerState`는 각 Transport가 자유롭게 정의 (Plain: 빈 struct, TLS: `ssl::context`)
- `wrap_socket`이 raw `tcp::socket`을 받아 `unique_ptr<SocketBase>`를 반환 — accept 시점에서 타입 소거 완료

### 2.3 TlsSocket 배치 (PASS)

`TlsSocket`이 `apex_shared`에 배치된 것은 올바른 의존성 방향:
- `apex_core`는 `SocketBase` 인터페이스만 정의 (OpenSSL 미의존)
- `apex_shared`는 `TlsSocket` 구현 제공 (OpenSSL 의존은 여기에 격리)
- `TcpSocket`은 `apex_core`에 배치 (Boost.Asio `tcp::socket`만 의존)

### 2.4 Listener::listener_state_ 소유권 (PASS)

`listener_state_`가 `Listener` 인스턴스 멤버로 선언되어 Listener 수명 동안 유효.
모든 `wrap_socket` 호출이 이 참조를 사용하므로 dangling 위험 없음.

## 4. 이슈 분석

### 4.1 이슈 없음 (0건)

이 PR의 핵심 변경은 설계적으로 건전하고 구현이 정확하다. 아래 관점에서 검증 완료:

**의존성 역전 정확성:**
- `apex_core`는 더 이상 `<boost/asio/ssl/context.hpp>`를 include하지 않음
- CMake에서 `OpenSSL::SSL`, `OpenSSL::Crypto` 링크가 제거됨
- `TlsSocket`은 `apex_shared`에 격리되어 OpenSSL 의존이 해당 모듈 내로 한정

**소유권 모델 정확성:**
- `Session`이 `unique_ptr<SocketBase>` 소유 — 명확한 단일 소유자
- `ssl::context`는 `Listener::listener_state_`가 소유 — Listener 수명 동안 유효
- `wrap_socket()`에서 `listener_state_` 참조 전달 — TLS 소켓이 ssl::context를 빌려 사용

**SlabAllocator move 안전성:**
- `session_pool_.construct(id, std::move(socket), ...)`에서 allocate 실패 시 construct 미호출 → socket move 미발생
- heap fallback 경로에서 socket이 아직 유효하므로 안전하게 `std::move` 가능
- 코드 주석으로 이 인변이턴트가 명시되어 있음 (`session_manager.cpp:24`)

**테스트 커버리지:**
- SocketBase 인터페이스 전용 테스트 9건
- 기존 52+ 테스트가 `make_tcp_socket()` 래핑으로 전환되어 SocketBase 경로를 간접 검증
- TlsSocket은 OpenSSL 테스트 인프라 필요로 이 PR 범위 외 (합리적 판단)

**Transport concept 호환성:**
- `static_assert(Transport<DefaultTransport>)` — 컴파일타임 검증
- `static_assert(Transport<MockTransport>)` — Mock도 검증
- `static_assert(Transport<PlainTcpTransport>)` — shared 측도 검증
- `static_assert(Transport<TlsTcpTransport>)` — TLS 측도 검증

**성능 영향:**
- SocketBase virtual dispatch ~2ns는 커널 syscall (μs 단위) 대비 0.001% 미만
- Session hot path는 여전히 implicit strand (single-threaded) + write_pump 코루틴
- 벤치마크(bench_session_throughput)는 raw socket 측정으로 virtual dispatch 오버헤드와 무관
