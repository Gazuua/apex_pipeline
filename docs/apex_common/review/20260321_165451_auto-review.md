## 발견 이슈 (5건, 모두 수정 완료)

### L-1. EVAL 결과 is_integer() 타입 체크 미수행 (Minor → 수정)
- **파일**: `chat_service.cpp:292`
- **내용**: `eval_result->integer` 접근 전 `is_integer()` 타입 체크 미수행. Redis EVAL이 정수가 아닌 값을 반환하면 silent wrong behavior (0으로 ROOM_FULL 처리).
- **수정**: `!eval_result->is_integer()` 조건 추가

### S-1. Signal handler에서 spdlog registry mutex 데드락 (Major → 수정)
- **파일**: `crash_handler.cpp:75`
- **내용**: `spdlog::default_logger()` 호출이 내부 registry mutex를 잡음. 시그널 핸들러에서 같은 스레드가 이미 mutex를 잡은 상태면 데드락. async_logger의 thread pool mutex도 추가 위험.
- **수정**: `install_crash_handlers()` 시점에 `spdlog::default_logger_raw()`를 `std::atomic<spdlog::logger*>`에 캐싱. signal handler에서는 캐시된 포인터로 직접 접근하여 registry mutex 우회.

### S-2. finalize_shutdown()에서 uninstall_crash_handlers() 미호출 (Major → 수정)
- **파일**: `server.cpp` `finalize_shutdown()`
- **내용**: `install_crash_handlers()`는 `run()` 시작에서 호출되나, shutdown 경로에서 해제 없음. 호출측이 `shutdown_logging()`으로 spdlog를 파괴한 후 크래시 발생 시, crash handler가 파괴된 spdlog 접근 → UB.
- **수정**: `finalize_shutdown()` 끝부분 `control_io_.stop()` 직전에 `uninstall_crash_handlers()` 추가.

### S-3. Windows _write() 반환값 미처리 (Minor → 수정)
- **파일**: `crash_handler.cpp:35`
- **내용**: Linux에서는 `[[maybe_unused]] auto _`로 반환값 명시 무시, Windows에서는 미처리. `/W4` 경고 가능.
- **수정**: `(void)_write(...)` 캐스트 추가.

### CI-1. ASAN 빌드에서 crash handler가 sanitizer 시그널 핸들러 덮어쓰기 (Critical → 수정)
- **파일**: `crash_handler.cpp` `install_crash_handlers()`
- **내용**: `std::signal(SIGSEGV, crash_signal_handler)` 호출이 ASAN의 SIGSEGV 핸들러를 덮어써서 shadow memory 메커니즘 방해. `test_server_multicore` ASAN 테스트 실패의 직접 원인.
- **수정**: `is_sanitizer_active()` 컴파일 타임 체크 도입. `__SANITIZE_ADDRESS__` / `__SANITIZE_THREAD__` / `__has_feature` 감지하여 sanitizer 빌드에서 no-op.
