## CRITICAL

### C-1. SecureString `operator==`가 상수 시간 비교를 하지 않음 -- 타이밍 사이드채널

**파일**: `apex_shared/lib/src/secure_string.cpp:91-102`

`operator==(const SecureString&)`, `operator==(std::string_view)`, `operator==(const char*)` 세 오버로드 모두 `std::string::operator==`를 사용한다. 표준 라이브러리의 문자열 비교는 첫 번째 불일치에서 즉시 반환(early-return)하므로, 비교 시간으로부터 일치하는 prefix 길이를 추론할 수 있다.

SecureString의 용도가 password, secret key 등 보안 민감 데이터이므로, `operator==`가 공격자에게 타이밍 오라클을 제공하는 것은 클래스의 보안 목적과 모순된다.

**권장안**: HMAC 기반 비교 또는 상수 시간 바이트 비교(`crypto_memcmp` 패턴)로 교체.

```cpp
bool SecureString::constant_time_equal(std::string_view other) const noexcept
{
    if (data_.size() != other.size()) return false;
    volatile unsigned char result = 0;
    for (size_t i = 0; i < data_.size(); ++i)
        result |= static_cast<unsigned char>(data_[i]) ^ static_cast<unsigned char>(other[i]);
    return result == 0;
}
```

크기 불일치 시의 early-return은 허용 가능 (길이 자체는 대부분 공개 정보). 바이트 내용 비교만 상수 시간이면 된다.

현재 코드베이스에서 `SecureString::operator==`를 직접 호출하는 코드가 없으므로 당장 취약점이 노출되지는 않지만, 공개 API로 제공하는 이상 사용자가 password 비교에 쓸 것이 자명하다. SecureString이라는 이름이 "이 클래스를 쓰면 안전하다"는 계약을 내포하므로, 비교 연산도 그 계약을 지켜야 한다.

## MAJOR

### M-1. 설계서-구현 불일치 -- move-only에서 copyable로 변경, 설계서 미갱신

**파일**: `apex_shared/lib/include/apex/shared/secure_string.hpp:20-21` vs `docs/apex_common/plans/20260326_182655_secret_management_design.md:41-43` 및 `docs/apex_common/plans/20260326_183200_secret_management_plan.md:7,195-196`

설계서(design)에서 명시:
> "move-only -- 복사하면 평문 사본이 늘어남"
> `SecureString(const SecureString&) = delete;`
> `SecureString& operator=(const SecureString&) = delete;`

구현 계획서(plan)에서도:
> "Architecture: SecureString move-only 래퍼"
> 테스트: `CopyIsDeleted` -- `std::is_copy_constructible_v<SecureString>` == false 검증

실제 구현:
- 복사 생성자/대입 연산자가 `public`으로 제공됨
- 테스트에서 `CopyIsDeleted` 대신 `CopyCreatesIndependentZeroizedCopy`로 교체

copyable로 변경한 것 자체는 실용적인 판단일 수 있다 (config 구조체의 집합 초기화, PubSubListener에서 password 복사 등). 그러나 설계서와 구현이 불일치하면 향후 유지보수자가 설계서를 신뢰하고 move-only를 전제로 코드를 작성할 위험이 있다.

**권장안**: 설계 문서와 구현 계획서에 copyable 변경 사유를 명시하여 갱신. 클래스 주석(`/// Copies are allowed`)은 이미 있으므로 문서만 동기화하면 된다. 아울러 copyable의 보안 트레이드오프(평문 사본 수 증가)를 ADR로 기록할 것을 권장.

### M-2. `const char*`, `string_view` 생성자가 implicit -- 의도하지 않은 암묵적 변환 경로

**파일**: `apex_shared/lib/include/apex/shared/secure_string.hpp:26-27`

```cpp
SecureString(const char* s);       // NOLINT(google-explicit-constructor)
SecureString(std::string_view sv); // NOLINT(google-explicit-constructor)
```

설계서에서는 `explicit SecureString(std::string_view sv)`로 명시했다. 실제 구현에서 `explicit`을 제거하고 NOLINT를 달아 정적 분석 경고도 억제했다.

implicit 변환 경로가 열리면:
1. `SecureString s = "plaintext";` -- 리터럴에서 암묵적 생성. 의도적 사용은 편하지만, 의도하지 않은 곳에서 임시 SecureString이 생성되어 제로화 없이 소멸될 수 있다.
2. 함수 인자로 `const char*`를 받는 곳에 SecureString이 요구되면, 호출자 모르게 임시 SecureString -> 소멸 -> 제로화가 발생하여 성능 예측이 어려워진다.
3. `SecureString` 타입의 필드에 `= "default_value"` 형태의 기본값 초기화가 가능해지는데, 이것은 config 구조체의 `PgAdapterConfig::connection_string` 기본값에서 실제로 사용된다:

```cpp
apex::shared::SecureString connection_string = "host=localhost port=6432 dbname=apex user=apex";
```

이 기본값 패턴이 implicit 변환을 도입한 동기로 보인다. 하지만 이 한 곳을 위해 모든 곳에서 implicit을 허용하는 것은 pit of success 원칙에 반한다.

**권장안**: `explicit`을 복원하고, `PgAdapterConfig`의 기본값은 아래 패턴으로 변경:

```cpp
SecureString connection_string{std::string_view{"host=localhost port=6432 dbname=apex user=apex"}};
```

이미 설계서(plan)의 Task 5 Step 1에서 이 정확한 패턴을 제시했으나 구현에서 반영되지 않았다.

### M-3. PubSubListener::Config의 SecureString password가 복사되는 경로

**파일**: `apex_services/gateway/src/gateway_service.cpp:369-376`

```cpp
PubSubListener::Config pubsub_cfg{
    .host = config_.redis_pubsub_host,
    .port = config_.redis_pubsub_port,
    .password = config_.redis_pubsub_password,  // 복사
    .initial_channels = config_.global_channels,
};
g.pubsub_listener = std::make_unique<PubSubListener>(
    pubsub_cfg, ...);  // pubsub_cfg가 값으로 전달 -> 또 다시 이동/복사
```

`config_.redis_pubsub_password`에서 `pubsub_cfg.password`로 복사가 발생한다. 현재 SecureString이 copyable이므로 컴파일은 되지만, 설계 원칙("복사하면 평문 사본이 늘어남")과 충돌한다.

`PubSubListener` 생성자가 `const Config&` -> `Config` (값 전달)로 변경된 이유도 이 맥락으로 보인다 -- move-only SecureString이었다면 `const Config&`로는 생성이 불가하므로.

**권장안**: `config_` 자체를 `std::move`하거나, password 필드만 `std::move`하여 복사를 최소화:

```cpp
PubSubListener::Config pubsub_cfg{
    .host = config_.redis_pubsub_host,
    .port = config_.redis_pubsub_port,
    .password = std::move(config_.redis_pubsub_password),  // 이동
    .initial_channels = config_.global_channels,
};
```

GatewayService의 `config_`가 이후 password에 재접근하지 않는다면 이 방식이 안전하다. on_wire는 라이프사이클에서 1회 호출이므로 이동 후 재사용 문제는 없을 것이다.

### m-2. `shrink_to_fit()` 후 SSO 버퍼 잔존 가능성이 문서화되지 않음

**파일**: `apex_shared/lib/src/secure_string.cpp:25`

```cpp
data_.clear();
data_.shrink_to_fit();    // SSO 버퍼도 해제 유도
```

`shrink_to_fit()`은 표준에서 non-binding request다. 또한 SSO 범위(일반적으로 15-22 바이트)의 짧은 문자열은 heap이 아닌 스택 inline 버퍼에 저장되므로, `shrink_to_fit()`이 이 인라인 버퍼를 제로화하지 못한다. `SecureZeroMemory`/`explicit_bzero`가 `data_.data()` 포인터를 대상으로 먼저 실행되므로 SSO 인라인 데이터도 제로화는 된다. 하지만 `clear()` 후 `std::string` 구현에 따라 SSO 버퍼에 잔존 데이터가 남을 수 있다.

현재 구현에서 `clear()` 전에 제로화를 하므로 대부분의 경우 안전하지만, 클래스 문서/주석에 이 제한사항을 명시하면 사용자의 오해를 방지할 수 있다.

## 긍정적 평가

1. **제로화 구현 품질**: `SecureZeroMemory`/`explicit_bzero` 분기와 `clear()` 전 제로화 순서가 올바르다. 컴파일러 최적화 우회를 정확히 처리했다.

2. **기존 API 호환성**: `c_str()`, `empty()`, `view()` 인터페이스가 `std::string`과 동일한 패턴이므로, Kafka/Redis 어댑터의 소비 코드가 변경 없이 동작한다. 마이그레이션 비용이 최소화됐다.

3. **ESO 설계**: `enabled: false` 기본값으로 로컬 무영향, `refreshInterval: "0"`으로 로테이션 스코프 제한, `targetSecret`이 기존 `existingSecret`과 일치하여 서비스 코드 변경 없이 연동 가능. 접근법 A(실용적 최소)에 충실하다.

4. **PgPool::connection_string() 반환 타입 변경**: `const std::string&` -> `std::string_view`로의 변경이 적절하다. SecureString의 내부 `std::string`에 대한 참조를 반환하면 SecureString의 캡슐화를 깨뜨릴 수 있었는데, `string_view`로 변경하여 읽기 전용 접근만 허용한다. `connect_async`도 `string_view`를 받으므로 불필요한 복사가 없다.

5. **PgAdapter 기본 생성자 분리**: `= {}` 기본 인자에서 명시적 기본 생성자로 분리한 것은 SecureString의 기본 생성이 implicit이 아닌 경우를 대비한 방어적 변경이다.

6. **테스트 커버리지**: self-move assignment, SSO 우회를 위한 긴 문자열 제로화, ASAN/MSAN 가드 등 edge case를 잘 다루고 있다.

7. **의존성 방향**: `apex_shared`에 SecureString을 배치하고 `apex_services`에서 참조하는 구조가 모듈 경계를 준수한다. `apex_core`에 보안 관련 타입을 넣지 않은 것이 적절하다.
