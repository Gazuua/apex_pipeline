## 완료된 작업

### 1. Gateway 인프라 와이어링 (main.cpp)

| 컴포넌트 | 변경 내용 |
|----------|-----------|
| PubSubListener | Redis Pub/Sub 구독 + BroadcastFanout 콜백 연결. `global_channels` TOML 설정 파싱 추가 |
| BroadcastFanout | `pub:global:*` 패턴 → 전체 세션 브로드캐스트, 나머지 → ChannelSessionMap 타겟 전달 |
| ChannelSessionMap | 서비스 팩토리에서 shared_ptr로 생성, 각 GatewayService에 포인터 전달 |
| PendingRequests Sweep | `enable_shared_from_this` 기반 1초 주기 타이머. per-core post로 thread-safe sweep |
| RateLimitFacade | standalone RedisAdapter(port 6381) + per-core PerIpRateLimiter + RedisRateLimiter |

### 2. 채널 구독 시스템 메시지 (gateway_service.cpp)

- **msg_id=4 (SubscribeChannel)**: 클라이언트가 Pub/Sub 채널 구독 요청. Gateway는 채널명을 도메인 무관하게 처리 (MSA 원칙 준수)
- **msg_id=5 (UnsubscribeChannel)**: 구독 해제
- payload: FlatBuffers table, field 0 = channel string

### 3. E2E 테스트 수정

- **에러 응답 체크**: `msg_id < 1000` → `flags & ERROR_RESPONSE` (WireHeader 규격)
- **글로벌 라이프사이클**: `SetUpTestSuite`/`TearDownTestSuite` → `::testing::Environment` (Docker + 서비스 1회 기동)
- **Chat 테스트**: JoinRoom 후 `subscribe_channel("pub:chat:room:{id}")` 추가
- **fixture 헬퍼**: `subscribe_channel()`, `unsubscribe_channel()` 추가
- **gateway_e2e.toml**: `[pubsub]`, `[rate_limit.*]` 설정 추가

### 4. 버그 수정

- **sweep 타이머 크래시**: 재귀 람다 값 캡처 → 빈 `std::function` 호출 → `std::bad_function_call`. `enable_shared_from_this` 패턴으로 수정
- **PubSubListener Windows timeout**: hiredis `REDIS_ERR_IO` 체크 → null reply 체크로 간소화 (Windows hiredis 호환)
- **gateway_config_parser**: `global_channels` 배열 파싱 추가

## E2E 테스트 결과 (6/11)

### 통과 (6건)

| 테스트 | 검증 범위 |
|--------|-----------|
| AuthE2ETest.LoginAndAuthenticatedRequest | Client → Gateway → Kafka → Auth → JWT → Chat ListRooms 전체 파이프라인 |
| AuthE2ETest.UnauthenticatedRequestRejected | JWT 미바인딩 요청 → Gateway error flag 응답 |
| RateLimitE2ETest.PerUserRateLimit | 인증 후 20회 burst → Redis 기반 user rate limit 트리거 |
| RateLimitE2ETest.PerIpRateLimit | 미인증 50회 연결 → auth 거부 + IP rate limit |
| RateLimitE2ETest.PerEndpointRateLimit | CreateRoom 15회 burst → endpoint override(5/min) 트리거 |
| TimeoutE2ETest.ServiceTimeout | 미등록 msg_id(2999) → PendingRequests sweep → ServiceTimeout 에러 |

### 실패 (5건) — 다음 세션에서 수정

#### F-1. AuthE2ETest.RefreshTokenRenewal
- **증상**: 만료된 JWT로 요청 시 `msg_id=2008` (정상 ListRoomsResponse) 수신. 에러 플래그 없음
- **원인**: JWT `access_token_ttl_sec=2`로 설정했지만, 테스트의 `sleep_for(2s)` 후에도 JWT가 아직 유효
- **분석**: Gateway의 jwt-cpp `leeway` 설정 (기본 30초?) 또는 clock_skew가 만료 체크를 느슨하게 함
- **수정 방안**: `auth_svc_e2e.toml`에서 TTL을 1초로 줄이거나, `gateway_e2e.toml`의 `clock_skew`를 0으로 설정. 테스트의 sleep도 TTL+1초로 조정

#### F-2. ChatE2ETest.ListRooms
- **증상**: `total_count=0, rooms->size()=0` — 방 목록 비어있음
- **원인**: 테스트가 방을 생성(CreateRoom msg_id 2001)하고 응답을 `client.recv()`로 받지만, Chat 서비스가 PG INSERT에 실패했거나 응답을 제대로 반환하지 않음
- **분석**: CreateRoom 응답(msg_id 2002)의 room_id 확인 필요. Chat 서비스의 PG 연결 또는 스키마 문제 가능
- **수정 방안**: Chat 서비스 로그(`ChatService_e2e.log`) 분석. PG 쿼리 에러 확인. CreateRoom 응답 파싱 디버깅

#### F-3. ChatE2ETest.RoomMessageBroadcast
- **증상**: Bob이 브로드캐스트(msg_id 2013) 수신 timeout
- **원인**: Redis Pub/Sub 경로 미완성. Chat 서비스가 Redis PUBLISH는 하지만, Gateway의 PubSubListener가 해당 채널을 구독하고 있는지 불확실
- **분석 포인트**:
  1. `subscribe_channel("pub:chat:room:{id}")` 시스템 메시지가 Gateway에 도달하는지
  2. PubSubListener가 동적 `subscribe()` 후 실제 Redis SUBSCRIBE 명령을 보내는지 (현재는 reconnect 시에만 적용)
  3. Chat 서비스의 `build_pubsub_payload()` 형식이 BroadcastFanout의 `[4B msg_id BE] + [payload]` 파싱과 일치하는지
- **수정 방안**: PubSubListener에 즉시 SUBSCRIBE 명령 전송 기능 추가 (현재는 "reconnect 시 적용" 주석). 또는 PSUBSCRIBE `pub:*` 패턴 사용

#### F-4. ChatE2ETest.GlobalBroadcast
- **증상**: Alice/Bob 모두 GlobalChatMessage(msg_id 2043) 미수신
- **원인**: F-3과 동일 경로. `pub:global:chat` 채널이 PubSubListener의 `initial_channels`에 포함되어 있지만, PubSubListener의 timeout→reconnect 루프가 정상 작동하지 않으면 SUBSCRIBE가 안 됨
- **분석**: PubSubListener 로그에서 "PubSub reconnecting" 반복 여부 확인. Redis PubSub(port 6383) 연결 성공 후 SUBSCRIBE가 실행되는지 확인
- **수정 방안**: F-3과 동일. PubSubListener가 reconnect 후 즉시 SUBSCRIBE 실행하는 코드 경로 검증

#### F-5. TimeoutE2ETest.ServiceRecoveryAfterTimeout
- **증상**: `msg_id=1000, payload_hex=0c00...` — 이전 테스트의 잔여 Kafka 응답이 스트림에 남음
- **원인**: ServiceTimeout 테스트(msg_id 2999)가 Kafka에 produce → Chat 서비스에서 unhandled → 응답 없음. 하지만 Gateway의 Kafka consumer가 다른 pending response를 수신할 수 있음
- **분석**: Gateway가 이전 테스트의 미소비 Kafka 응답을 수신하여 현재 세션에 전달. 또는 이전 테스트의 Login 응답이 늦게 도착
- **수정 방안**: 테스트 시작 시 `client.recv(1s)` flush 루프 추가. 또는 ServiceRecovery 테스트에서 새로운 TCP 연결 사용
