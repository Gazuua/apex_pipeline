# v0.4 외부 어댑터 완료 기록

**버전**: v0.4.0.0 ~ v0.4.4.0
**기간**: 2026-03-13
**브랜치**: feature/v0.4-external-adapters
**상태**: 구현 완료, PR 대기

---

## 개요

C++23 코루틴 기반 외부 서비스 어댑터 레이어를 완성했다.
Kafka/Redis/PostgreSQL 3종 어댑터를 공통 추상화(AdapterBase CRTP, ConnectionPool CRTP) 위에 구현하고,
docker-compose 기반 통합 테스트 인프라를 구축했다.

## 구현 범위

### Plan 1: 공통 추상화 + 코어 변경 (v0.4.0)
- `AdapterBase<Derived>` CRTP: init/shutdown 생명주기, 코어별 인스턴스 관리
- `ConnectionPool<Derived, Conn>` CRTP: acquire/release/shrink_idle/health_check_tick/close_all
- `AdapterError` 구조체: ErrorCode + native_error + message
- `PoolConfig` 구조체: min_size, max_size, max_idle_time, health_check_interval
- `PoolStats` 구조체: total_acquired/released/created/destroyed/failed 카운터
- apex_core `ErrorCode::AdapterError` 추가

### Plan 2: Kafka 어댑터 (v0.4.1)
- `KafkaProducer`: 전역 공유, rd_kafka_produce() 직접 호출 (SPSC 큐 불필요)
- `KafkaConsumer`: 코어별 인스턴스, Linux io_event_enable / Windows timer polling
- `KafkaAdapter`: AdapterBase CRTP, produce poll 타이머 스케줄링
- `KafkaSink`: spdlog sink -> Kafka 토픽, JSON 구조화 로그, trace_id MDC

### Plan 3: Redis 어댑터 (v0.4.2)
- `HiredisAsioAdapter`: hiredis redisAsyncContext를 Boost.Asio에 직접 통합
- `RedisConnection`: 코루틴 기반 async_command, RESP 파싱 유틸리티
- `RedisPool`: ConnectionPool CRTP 상속, 동기 validate
- `RedisAdapter`: AdapterBase CRTP, 코어별 풀 라우팅

### Plan 4: PostgreSQL 어댑터 (v0.4.3)
- `PgConnection`: libpq fd -> Asio IOCP/epoll, async connect/query/execute
- `PgResult`: PGresult RAII 래퍼
- `PgPool`: ConnectionPool CRTP 상속, lazy async connect (acquire_connected)
- `PgAdapter`: AdapterBase CRTP, 코어별 풀 라우팅

### Plan 5: 인프라 + 통합 테스트 + 문서 (v0.4.4)
- docker-compose PgBouncer 추가 (transaction pooling, port 6432)
- ConnectionPool PoolStats 통계 카운터
- 통합 테스트 CMake 인프라 (APEX_BUILD_INTEGRATION_TESTS 옵션)
- Kafka/Redis/PG/KafkaSink 통합 테스트 4종
- ADR-08 업데이트, 마스터 설계서 동기화

## 설계 결정 (ADR-08 업데이트)

- **KafkaProducer**: SPSC 큐 불필요로 확정. rd_kafka_produce()가 스레드세이프하므로 코어에서 직접 호출.
- **KafkaConsumer**: Linux io_event_enable (pipe fd -> Asio stream_descriptor), Windows steady_timer polling (5ms).
- **ConnectionPool**: Redis는 동기 validate, PG는 비동기 validate + lazy connect. 구현 차이가 크므로 무리한 공통화보다 CRTP 공통 인터페이스 유지가 YAGNI에 부합.
