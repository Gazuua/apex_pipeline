## FIX-1: 에러 전파 파이프라인

### 결정 사항

- **핸들러 반환 타입**: `awaitable<void>` → `awaitable<Result<void>>` (강제 마이그레이션)
- **양쪽 시그니처 공존 안 함** — 프레임워크 계약 일관성 우선
- `co_return apex::ok()` (성공) / `co_return apex::error(ErrorCode::X)` (에러)

### 변경 흐름

```
Handler (awaitable<Result<void>>)
  → MessageDispatcher::dispatch()
    → Server::process_frames() Result 검사
      → 에러 시 ErrorSender::build_error_frame() 자동 전송
```

### 수정 대상

**핵심 타입 변경:**
- `MessageDispatcher::Handler`: `awaitable<void>(SessionPtr, uint16_t, span<const uint8_t>)` → `awaitable<Result<void>>(...)`
- `MessageDispatcher::dispatch()` 반환 타입: 내부 에러(UnknownMessage 등)와 핸들러 에러를 `Result<void>`로 통합
- `DispatchError` enum 제거 또는 `ErrorCode`로 통합

**ServiceBase 메서드:**
- `handle()`: 메서드 포인터 시그니처 `awaitable<void>` → `awaitable<Result<void>>`
- `route<FbsType>()`: 동일 변경 + FlatBuffers 검증 실패 시 `apex::error(ErrorCode::FlatBuffersVerifyFailed)` 반환

**Server 에러 전파 로직 (process_frames):**
```cpp
auto result = co_await dispatcher_.dispatch(session, msg_id, payload);
if (!result) {
    auto error_frame = ErrorSender::build_error_frame(msg_id, result.error());
    co_await session->async_send_raw(error_frame);
}
```

**예제/테스트:**
- echo_server, chat_server: 핸들러에 `co_return apex::ok()` 추가
- 전체 테스트 핸들러 시그니처 갱신
- 에러 자동 전파 테스트 신규 추가

## FIX-2: ProtocolBase CRTP 추상화

### 결정 사항

- **얇은 추상화** — 프레임 경계(decode/consume/encode)만 정의
- HeaderType 분리는 YAGNI (WebSocket 구현 시 확장)
- Server는 TcpBinaryProtocol을 직접 사용 (템플릿화는 WebSocket 추가 시)

### 새 파일

**`core/include/apex/core/protocol_base.hpp`**
```cpp
template <typename Derived>
struct ProtocolBase {
    static auto try_decode(RingBuffer& buf) {
        return Derived::try_decode_impl(buf);
    }
    static void consume_frame(RingBuffer& buf, const auto& frame) {
        Derived::consume_frame_impl(buf, frame);
    }
};
```

**`core/include/apex/core/tcp_binary_protocol.hpp`**
```cpp
struct TcpBinaryProtocol : ProtocolBase<TcpBinaryProtocol> {
    using FrameType = Frame;

    static std::expected<Frame, FrameError> try_decode_impl(RingBuffer& buf);
    static void consume_frame_impl(RingBuffer& buf, const Frame& frame);
};
```

### 수정 대상

- `Server::process_frames()`: `FrameCodec::try_decode()` → `TcpBinaryProtocol::try_decode()`
- `FrameCodec::consume_frame()` → `TcpBinaryProtocol::consume_frame()`
- WireHeader, FrameCodec: 기존 코드 유지 (TcpBinaryProtocol이 위임)
- CMakeLists.txt: 새 파일 추가

## ADR-21 갱신: 코루틴 프레임 할당 전략

### 변경 근거

업계 주요 프레임워크 조사 결과:
- **Seastar**: malloc 자체를 per-core allocator로 교체. promise_type 오버로드 아님
- **folly**: `coro::Task<T>`에 커스텀 할당 없음. jemalloc 의존
- **Boost.Asio**: `awaitable<T>` 내부 promise_type 수정 불가. HALO 최적화 의존
- **cppcoro**: HALO 최적화를 유도하는 설계. 커스텀 풀 할당 없음

**결론**: promise_type::operator new 풀 오버로드는 업계 미검증 접근

### 갱신 내용

기존:
> promise_type에 operator new/delete 오버로드 → 코어별 슬랩 풀에서 할당

변경:
> 고성능 범용 allocator(mimalloc 또는 jemalloc) 링크 + HALO 컴파일러 최적화 활용.
> 벤치마크에서 코루틴 프레임 할당이 병목으로 확인될 경우, 커스텀 코루틴 타입 도입 검토.
