## 2. 실행 전략

**의존성 기반 2-Phase** 실행. Bundle C 내부에 순서 의존성이 있으므로 C-prep(Phase 1)과 C-main(Phase 2)으로 분할한다.

### 2.2 워크트리

하나의 브랜치 `feature/v0.5-wave1`에서 전체 작업을 수행한다. Bundle 간 파일 충돌이 없도록 터치 영역을 사전 분리(§11 참조).

```
Phase 1 (병렬)          Phase 2 (병렬)          최종
┌─────────────┐        ┌─────────────┐        ┌──────────────┐
│ Bundle A    │        │ Bundle C-main│        │ auto-review  │
│ Bundle B    │──gate──│ Bundle E-1   │──gate──│ 이슈 수정     │
```

## 4. 프로토콜 레이어 아키텍처

### 4.1 Protocol concept (apex_core)

```cpp
template<typename P>
concept Protocol = requires {
    typename P::Config;
    typename P::Frame;
} && requires(RingBuffer& buf, const typename P::Frame& frame) {
    { P::try_decode(buf) } -> std::same_as<Result<typename P::Frame>>;
    { P::consume_frame(buf, frame) } -> std::same_as<void>;
};
```

Protocol concept은 core에 정의하여 **상위 레이어(shared, services)가 구현을 제공**하는 의존성 역전 구조를 만든다. `try_decode`와 `consume_frame`을 분리한 이유는 decode 실패 시 버퍼를 소비하지 않기 위함이다.

**`try_decode` 에러 매핑** (TcpBinaryProtocol 기준):

| FrameError | ErrorCode | 의미 |
|------------|-----------|------|
| `InsufficientData` | `InsufficientData`(13) | 버퍼에 완전한 프레임 없음 — 더 읽기 |
| `HeaderParseError` | `InvalidMessage` | 잘못된 헤더 — 연결 종료 |
| `BodyTooLarge` | `InvalidMessage` | 프레임 크기 초과 — 연결 종료 |

`InsufficientData`는 정상적 "더 읽기" 신호이므로 별도 에러코드(값 = 13)로 분리하여 `ConnectionHandler<P>::process_frames()`에서 명확히 구분한다:
- `ErrorCode::InsufficientData` → `break` (더 읽기)
- 그 외 에러 → 연결 종료

### 4.2 Server + Listener 구조

```
Server (비템플릿)
├── vector<unique_ptr<ListenerBase>>
├── template<Protocol P> listen(port, config) → Listener<P> 생성·등록
├── start() / drain() / stop()
│
ListenerBase (virtual — lifecycle만)
│   ※ 의도적 virtual: CRTP 일관성 예외.
│   start/drain/stop은 서버 시작·종료 시 1회 호출로 비용 ≈ 0.
│   Hot path는 ConnectionHandler<P> 템플릿으로 zero-overhead.
│
└── Listener<P> : ListenerBase
    ├── TcpAcceptor
    ├── MessageDispatcher          ← handler보다 먼저 선언 (소멸 순서 보장)
    └── PerCoreArray<ConnectionHandler<P>>
```

**ListenerBase**에는 반드시 아래 주석을 포함한다:

```cpp
// 의도적 virtual: CRTP 일관성 예외.
// start/drain/stop은 서버 시작·종료 시 1회 호출로 비용 ≈ 0.
// Hot path는 ConnectionHandler<P> 템플릿으로 zero-overhead.
class ListenerBase {
public:
    virtual ~ListenerBase() = default;
    // 현재 Server lifecycle은 동기 흐름. start()는 내부적으로 코루틴을
    // spawn하지만 자체가 코루틴일 필요 없음.
    virtual void start() = 0;
    virtual void drain() = 0;
    virtual void stop() = 0;
};
```

**Listener<P>**는 ListenerBase를 상속하며, 프로토콜별 accept loop와 ConnectionHandler 배열을 관리한다. **Listener<P>가 자체 MessageDispatcher를 소유**하여 PerCoreState의 dispatcher를 대체한다. 이유: 프로토콜별로 라우팅하는 메시지 셋이 다를 수 있음 (예: WebSocket은 0x0001~0x00FF, TCP는 0x0100~). PerCoreState에서 MessageDispatcher는 제거된다.

**ServerConfig::port 제거**: `listen<P>(port, config)` API 도입에 따라:
- `ServerConfig::port` 필드 제거 — 각 `listen<P>(port, config)` 호출로 대체
- `Server` 생성자에서 acceptor 바인딩 로직 제거 — `Listener::start()`에서 수행
- `Server::port()` 메서드 제거 — 여러 포트를 가질 수 있으므로 의미 없음
- `listen<P>()` 바인딩 실패 시 `std::system_error` 예외 throw (포트 충돌 등)

### 4.3 의존성 방향

```
apex_core (Protocol concept, Server, ConnectionHandler<P>, Session)
    ↑
apex_shared (TcpBinaryProtocol, WebSocketProtocol, adapters)
    ↑
apex_services (Server 인스턴스화 + listen<P> 호출)
```

- **core**는 concept만 정의하고 구체 프로토콜을 모른다
- **shared**는 concept을 만족하는 구현을 제공한다
- **services**는 필요한 프로토콜을 선택하여 Server에 등록한다

순환 의존 없이 기존 3-layer가 유지된다.

### 4.4 PerCoreState → Listener 마이그레이션 경로

**현재 구조**: PerCoreState가 ConnectionHandler, SessionManager, MessageDispatcher, BumpAllocator, ArenaAllocator를 모두 소유.

**변경 후**:

- **Listener<P>가 소유**: per-core `ConnectionHandler<P>`, `MessageDispatcher` (프로토콜별)
- **PerCoreState에서 제거**: ConnectionHandler, MessageDispatcher
- **PerCoreState에 남는 것**: SessionManager, BumpAllocator, ArenaAllocator (프로토콜 무관 리소스)
- **참조 전달**: Listener<P>는 PerCoreState의 SessionManager 참조를 받아 ConnectionHandler<P>에 전달

이 분리를 통해 프로토콜 종속 컴포넌트와 프로토콜 무관 리소스가 명확히 구분된다.

### 4.5 마이그레이션 대상

| 파일 | 현재 위치 | 이동 | 비고 |
|------|----------|------|------|
| ProtocolBase | `apex_core` | **삭제** | Protocol concept으로 대체 |
| TcpBinaryProtocol | `apex_core` | `apex_shared/lib/protocols/tcp/` | concept 만족하도록 리팩터링 |
| FrameCodec | `apex_core` | `apex_shared/lib/protocols/tcp/` | TcpBinaryProtocol과 함께 이동 |
| WireHeader | `apex_core` | `apex_shared/lib/protocols/tcp/` | 프레임 헤더 정의 |
| WebSocketProtocol | (신규) | `apex_shared/lib/protocols/websocket/` | Gateway용 신규 구현 |

### 4.6 WebSocketProtocol try_decode 전략

Beast의 `websocket::stream`은 자체 버퍼링으로 완성된 메시지 단위 수신을 처리한다. 따라서:

- **핵심 방향**: Beast가 framing을 담당하고, `WebSocketProtocol::try_decode()`는 메시지 추출만 수행
- Beast 내부 버퍼에서 메시지를 추출하므로 RingBuffer는 사용하지 않을 수 있음
- Protocol concept에 optional `handshake()` static method 추가를 고려:

```cpp
// concept에 optional: 없으면 handshake 생략
if constexpr (requires { P::handshake(socket, config); }) {
    co_await P::handshake(socket, config);
}
```

- 또는: WebSocketProtocol은 `ConnectionHandler` 수준에서 Beast stream으로 소켓을 래핑한 후 사용
- **구현 시 결정**: Beast API 학습 후 RingBuffer 우회 방식 또는 Beast → RingBuffer 브릿지 방식 중 확정. 상세는 구현 시 결정으로 마킹

### 4.7 main() 사용 예시

**단일 프로토콜 — 게임 서비스**

```cpp
Server server;
server.listen<TcpBinaryProtocol>(9100, TcpBinaryProtocol::Config{
    .max_frame_size = 64 * 1024
});
co_await server.start();
```

**듀얼 프로토콜 — Gateway (WebSocket + TCP binary)**

```cpp
Server server;
server.listen<WebSocketProtocol>(443, WebSocketProtocol::Config{
    .path = "/ws", .max_message_size = 1 * 1024 * 1024
});
server.listen<TcpBinaryProtocol>(9000, TcpBinaryProtocol::Config{
    .max_frame_size = 256 * 1024
});
co_await server.start();
```

**미래 확장 — TCP + UDP relay**

```cpp
Server server;
server.listen<TcpBinaryProtocol>(9100, tcp_config);
server.listen<UdpRelayProtocol>(9200, udp_config);  // 향후 추가
co_await server.start();
```

동일한 `listen<P>()` API로 프로토콜 종류에 관계없이 일관된 패턴을 유지한다.

## 5. Per-Session Write Queue

### 5.1 구조

```cpp
struct WriteRequest {
    std::vector<uint8_t> data;
};

class Session {
    // ...
    std::deque<WriteRequest> write_queue_;
    bool pump_running_ = false;
    size_t max_queue_depth_ = 256;

    Result<void> enqueue_write(std::vector<uint8_t> data);
    Result<void> enqueue_write_raw(std::span<const uint8_t> data);
    Task<void> write_pump();
};
```

- `enqueue_write()` — **동기**, 즉시 반환. 큐에 적재 후 pump가 미실행이면 기동
- `write_pump()` — **awaitable**, 큐에서 순차적으로 `async_write` 수행. 큐가 비면 종료. 에러 발생 시 `write_queue_.clear()` 후 세션 close

per-core single thread 모델이므로 큐 접근에 별도 동기화가 필요 없다. `std::deque`를 선택한 이유는 FIFO 특성과 양단 접근 효율 때문이다.

### 5.2 Backpressure

```cpp
Result<void> Session::enqueue_write(std::vector<uint8_t> data) {
    if (write_queue_.size() >= max_queue_depth_) {
        SPDLOG_WARN("Session {} write queue full (depth={})", id_, max_queue_depth_);
        return ErrorCode::BufferFull;
    }
    write_queue_.push_back(WriteRequest{std::move(data)});
    if (!pump_running_) {
        pump_running_ = true;
        boost::asio::co_spawn(socket_.get_executor(), write_pump(), boost::asio::detached);
    }
    return {};
}
```

- `max_queue_depth_`: 기본 256, Config로 조정 가능
- 초과 시 `ErrorCode::BufferFull` 반환 + `SPDLOG_WARN` 로그
- 호출자(서비스 로직)가 반환값으로 backpressure를 인지하고 대응

### 5.3 API 변경

| 기존 API | 신규 API | 변경 사유 |
|---------|---------|----------|
| `async_send()` (co_await) | `enqueue_write()` (동기) | 호출자가 I/O 완료를 기다릴 필요 없음 |
| `async_send_raw()` (co_await) | `enqueue_write_raw()` (동기) | 동일 |

`co_await` 제거로 서비스 로직이 write I/O에 블로킹되지 않는다. 실제 전송은 write pump가 비동기로 처리한다.

## 6. 어댑터 회복력 (Bundle B)

### 6.1 CircuitBreaker

```cpp
struct CircuitBreakerConfig {
    uint32_t failure_threshold = 5;
    std::chrono::milliseconds open_duration{30'000};
    uint32_t half_open_max_calls = 3;
};

enum class CircuitState { CLOSED, OPEN, HALF_OPEN };

class CircuitBreaker {
public:
    explicit CircuitBreaker(CircuitBreakerConfig config);

    template<std::invocable F>
        requires std::same_as<std::invoke_result_t<F>, Task<Result<void>>>
    Task<Result<void>> call(F&& fn);

    CircuitState state() const noexcept;

private:
    CircuitBreakerConfig config_;
    CircuitState state_ = CircuitState::CLOSED;
    uint32_t failure_count_ = 0;
    uint32_t half_open_calls_ = 0;
    std::chrono::steady_clock::time_point open_since_;
};
```

- **CLOSED → OPEN**: 연속 실패가 `failure_threshold`에 도달
- **OPEN**: 호출 거부 — `ErrorCode::CircuitOpen`(2002) 반환
- **OPEN → HALF_OPEN**: `open_duration` 경과 후 다음 호출 시
- **HALF_OPEN → CLOSED**: `half_open_max_calls` 내 성공
- **HALF_OPEN → OPEN**: 실패 발생 시 즉시

각 어댑터가 **composition**으로 CircuitBreaker를 소유한다. 상속이 아닌 composition인 이유: 회로 차단은 어댑터의 본질이 아닌 횡단 관심사이며, 어댑터마다 적용 범위(메서드 단위)가 다를 수 있기 때문이다.

### 6.2 Redis AUTH

```cpp
// create()는 동기 팩토리 — 연결 객체 생성 + 설정만 수행
Result<RedisConnectionPtr> RedisConnection::create(const RedisConfig& config) {
    auto conn = std::make_unique<RedisConnection>(config);
    if (!conn) return ErrorCode::AllocationFailed;
    return conn;
}

// AUTH는 RedisMultiplexer 연결 루프 내 코루틴에서 수행
// command()를 사용하지 않고 별도 authenticate() 메서드로 분리
Task<Result<void>> RedisMultiplexer::authenticate(RedisConnection& conn, const RedisConfig& config) {
    if (!config.password.empty()) {
        // command() 대신 raw_async_command 직접 사용 — command()는 reconnecting_ 상태 체크 등
        // 내부 가드가 있어 reconnect_loop 내에서 사용 시 문제가 될 수 있음
        auto auth_result = co_await conn.raw_async_command("AUTH", config.password);
        if (!auth_result) {
            conn.close();  // 실패 → 연결 폐기
            co_return auth_result.error();
        }
    }
    co_return {};
}
```

- `RedisConnection::create()`는 **동기 팩토리**로 유지 — 연결 객체 생성 + 설정만 수행
- AUTH는 `create()` 이후 별도 코루틴(RedisMultiplexer의 연결 루프 내)에서 수행
- `create()` 자체는 동기 유지하여 팩토리 패턴의 단순성을 보존
- **reconnect 시마다** `authenticate()` 호출로 AUTH 수행 (풀이 재생성하는 연결에도 적용)
- AUTH 실패 시 연결을 폐기하여 인증 안 된 연결이 풀에 유입되지 않도록 함

### 6.3 RedisReply ARRAY

```cpp
struct RedisReply {
    enum Type { STRING, INTEGER, BULK_STRING, ARRAY, NIL, ERROR };

    Type type;
    std::string str;
    int64_t integer = 0;
    std::vector<RedisReply> array;  // 추가: 재귀 구조

    bool is_array() const noexcept { return type == ARRAY; }
    bool is_nil() const noexcept { return type == NIL; }
    bool is_error() const noexcept { return type == ERROR; }
    // ...
};
```

- `array` 필드 추가 (`vector<RedisReply>`, 재귀 파싱)
- `is_array()`, `is_nil()` 등 타입 체크 헬퍼 제공
- RESP 프로토콜의 `*N\r\n` 형태를 재귀적으로 파싱

### 6.4 PgPool 재시도

```cpp
struct PgPoolRetryConfig {
    uint32_t max_acquire_retries = 3;
    std::chrono::milliseconds retry_backoff{100};  // 초기 backoff
};

Task<Result<PgConnectionPtr>> PgPool::acquire_with_retry() {
    for (uint32_t i = 0; i <= config_.max_acquire_retries; ++i) {
        auto conn = co_await acquire();  // 기존 내부용 유지
        if (conn) co_return conn;

        if (i < config_.max_acquire_retries) {
            auto delay = config_.retry_backoff * (1u << i);  // exponential
            co_await async_sleep(delay);
        }
    }
    co_return ErrorCode::PoolExhausted;
}
```

- `acquire_with_retry()` — exponential backoff로 재시도
- 기존 `acquire()`는 내부용으로 유지 (단일 시도)
- 외부 API는 `acquire_with_retry()`로 전환
- `PoolExhausted` 에러코드를 AdapterError(2000) 범위 내에 추가: `PoolExhausted = 2001`
- `CircuitOpen` 에러코드 추가: `CircuitOpen = 2002` — CircuitBreaker OPEN 상태에서 호출 거부 시 반환

### 6.5 Kafka DLQ

```cpp
// MessageCallback 반환타입 변경: void → Result<void>
// Message 래퍼 클래스 도입 없음 — 파라미터 리스트 그대로 유지
using MessageCallback = std::function<Result<void>(
    std::string_view topic, int32_t partition,
    std::span<const uint8_t> key,
    std::span<const uint8_t> payload,
    int64_t offset)>;

// Consumer의 poll 루프 (동기 timer 기반 polling — 코루틴 전환 불필요)
// callback 반환값만 검사하여 DLQ 라우팅 결정
auto result = callback_(topic, partition, key, payload, offset);
if (!result) {
    auto dlq_topic = fmt::format("{}-dlq", topic_);
    SPDLOG_WARN("Message processing failed (topic={}), routing to DLQ: {}",
                topic_, result.error().message());
    producer_.produce(dlq_topic, key, payload);
}
```

- `MessageCallback` 반환타입: `void` → `Result<void>` (파라미터 리스트 유지, Message 래퍼 없음)
- Consumer의 poll 루프는 동기(timer 기반 polling)이므로 코루틴 전환 불필요 — callback 반환값만 검사
- 실패 시 `{topic}-dlq`로 자동 produce + WARN 로그
- Consumer 생성 시 Producer 레퍼런스를 주입받아 DLQ produce에 사용

### 6.6 KafkaAdapter 상태 정리

```cpp
// 기존: bool draining_
// 변경: enum + atomic
enum class AdapterState : uint8_t {
    RUNNING,
    DRAINING,
    CLOSED
};

class KafkaAdapter {
    std::atomic<AdapterState> state_{AdapterState::RUNNING};

    bool is_running() const noexcept {
        return state_.load(std::memory_order_acquire) == AdapterState::RUNNING;
    }
    // ...
};
```

- `bool draining_` → `AdapterState` enum으로 상태 전이를 명확화
- `std::atomic<AdapterState>`로 스레드 안전 보장
- 3-state 모델: RUNNING → DRAINING → CLOSED (단방향 전이)

- `token: string` — JWT 또는 credential
- `request_type: uint8` — 0=발급, 1=검증, 2=블랙리스트

**auth_response.fbs**:
- `success: bool` — 인증 성공 여부
- `user_id: uint64` — 인증된 사용자 ID
- `error_message: string` — 실패 시 에러 메시지

**error_envelope.fbs**:
- `source_service: string` — 에러 발생 서비스명
- `error_code: uint16` — 에러 코드
- `message: string` — 에러 상세 메시지
- `timestamp_ms: uint64` — 에러 발생 시각 (ms)

### 7.3 진화 정책

> v0.5 최소 메시지 셋. 서비스 요구사항 변경 시 자유롭게 추가/수정/폐기 가능.
> **현재 스키마에 후속 설계가 종속되지 않도록 할 것.**

FlatBuffers 스키마이므로 필드 추가는 하위 호환, 필드 삭제는 deprecated 마킹으로 처리한다. v0.5 이후 서비스가 추가되면 스키마도 함께 확장한다.

### 7.4 기존 코어 스키마

`apex_core/schemas/`의 아래 스키마는 core에 **그대로 유지**한다:

- `echo.fbs` — 프레임워크 내부 에코 테스트
- `chat_message.fbs` — 예제용 메시지
- `heartbeat.fbs` — 프레임워크 내부 keepalive
- `error_response.fbs` — 코어 레벨 에러 응답

이들은 프레임워크 내부/예제 용도이며 서비스 간 공유 스키마와는 역할이 다르다.

### 9.3 Phase 2 의존성
