## 1. 개요

v0.6 운영 인프라 세 번째 마일스톤. K8s 배포 매니페스트와 Helm 차트로 서비스 오케스트레이션을 구성한다.

## 4. Namespace & 배포 전략

### 2-Release 분리

단일 Helm release로 두 namespace에 배포하면 namespace 명시를 모든 템플릿에 하드코딩해야 하고, Bitnami sub-chart의 namespace override가 까다로워진다. 대신 **2-release 분리**를 채택:

```bash
# Release 1: 인프라 (apex-infra namespace)
helm install apex-infra ./apex-pipeline \
  -n apex-infra --create-namespace \
  -f values.yaml \
  --set gateway.enabled=false \
  --set auth-svc.enabled=false \
  --set chat-svc.enabled=false \
  --set log-svc.enabled=false

# Release 2: 서비스 (apex-services namespace)
helm install apex-services ./apex-pipeline \
  -n apex-services --create-namespace \
  -f values.yaml \
  --set kafka.enabled=false \
  --set redis-auth.enabled=false \
  --set redis-ratelimit.enabled=false \
  --set redis-chat.enabled=false \
  --set redis-pubsub.enabled=false \
  --set postgresql.enabled=false \
  --set pgbouncer.enabled=false
```

`scripts/local-setup.sh`가 이 2단계를 자동화하므로 사용자 체감은 한 방 설치.

**장점**:
- 각 release가 단일 namespace에 배포 → Helm 표준 동작, namespace 명시 불필요
- 인프라만 독립 관리/업그레이드 가능
- Bitnami chart namespaceOverride 불필요

**프로덕션**: 인프라 release를 스킵하고 서비스 release만 배포 (외부 인프라 엔드포인트는 values-prod.yaml에서 지정).

### 서비스별 차이점

| | Gateway | Auth | Chat | Log |
|---|---|---|---|---|
| Ingress | O (nginx, wss://) | X | X | X |
| HPA | O (min:1, max:10) | X | X | X |
| PDB | O (maxUnavailable:1) | X | X | X |
| Service ports | 8443(ws), 8081(metrics) | 8082(metrics only) | 8083(metrics only) | 8084(metrics only) |
| extraVolumes | RSA 공개키 | RSA 키쌍 | — | — |
| 의존 인프라 | Kafka, Redis x3 | Kafka, Redis-auth, PG | Kafka, Redis-chat, Redis-pubsub, PG | Kafka |

## 7. Secret 주입 전략

### expand_env 활용 (init-container 불필요)

`apex::shared::expand_env()`가 `${VAR:-default}` 패턴을 이미 처리한다. K8s에서는:

1. **ConfigMap**: TOML 설정을 `${VAR:-default}` 패턴 그대로 마운트
2. **Secret → 환경변수**: K8s Secret을 `envFrom`으로 컨테이너에 주입
3. **expand_env**: 서비스 시작 시 TOML 파서가 환경변수를 자동 치환

```yaml
# Deployment (apex-common.deployment 템플릿)
containers:
  - name: {{ .Chart.Name }}
    envFrom:
      - secretRef:
          name: {{ .Values.secrets.existingSecret | default (printf "%s-secrets" (include "apex-common.fullname" .)) }}
    volumeMounts:
      - name: config
        mountPath: /app/config.toml
        subPath: config.toml
volumes:
  - name: config
    configMap:
      name: {{ include "apex-common.fullname" . }}-config
```

```toml
# ConfigMap 내 TOML (기존 패턴 그대로)
[redis]
host = "apex-pipeline-redis-auth-master.apex-infra.svc.cluster.local"
port = 6379
password = "${REDIS_AUTH_PASSWORD:-dev_redis_pass}"
```

**장점**: 서비스 코드 변경 없음. docker-compose에서 쓰던 TOML을 K8s DNS만 바꿔서 그대로 사용.

### Secret 리소스 환경 분리

| | 로컬 (`values.yaml`) | 프로덕션 (`values-prod.yaml`) |
|---|---|---|
| 크레덴셜 | 기본값 하드코딩 (`dev_redis_pass`) | `existingSecret` 참조 |
| RSA 키 | `rsaKeys.create: true` (테스트 키 내장) | `rsaKeys.existingSecret` 참조 |
| TLS | minikube 자체 서명 | cert-manager (BACKLOG-197) |

### 서비스별 Secret 매핑

| 서비스 | 필요한 환경변수 |
|--------|----------------|
| Gateway | `REDIS_AUTH_PASSWORD`, `REDIS_RATELIMIT_PASSWORD`, `REDIS_PUBSUB_PASSWORD` + RSA 공개키 볼륨 |
| Auth | `REDIS_AUTH_PASSWORD`, `PG_SERVICE_PASSWORD` + RSA 키쌍 볼륨 |
| Chat | `REDIS_CHAT_PASSWORD`, `REDIS_PUBSUB_PASSWORD`, `PG_SERVICE_PASSWORD` |
| Log | (현재 없음, Kafka 인증 추가 시 확장) |

### Gateway TLS 비활성화

K8s에서는 Ingress가 TLS termination을 수행하므로, Gateway 자체 TLS는 비활성화한다. K8s용 TOML ConfigMap에서 `[tls]` 섹션을 제거하고, Ingress → Gateway 간은 plain WebSocket(ws://) 통신.

### Ingress 설정 (Gateway)

Ingress 컨트롤러: **kubernetes/ingress-nginx** (NGINX Inc 버전이 아닌 커뮤니티 표준). WebSocket upgrade는 `proxy-http-version: "1.1"` 설정 시 자동 처리됨.

```yaml
metadata:
  annotations:
    nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"      # WS 장시간 연결
    nginx.ingress.kubernetes.io/proxy-send-timeout: "3600"
    nginx.ingress.kubernetes.io/proxy-http-version: "1.1"        # WS upgrade 자동 처리
    nginx.ingress.kubernetes.io/upstream-hash-by: "$remote_addr"  # sticky (선택)
spec:
  ingressClassName: nginx
  tls:
    - hosts: [{{ .Values.ingress.host }}]
      secretName: {{ .Values.ingress.tls.secretName }}
  rules:
    - host: {{ .Values.ingress.host }}
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: {{ include "apex-common.fullname" . }}
                port:
                  number: 8443
```

| 대상 | 로컬 (Bitnami) | 프로덕션 (외부) |
|------|----------------|----------------|
| Kafka | `apex-infra-kafka.apex-infra.svc.cluster.local:9092` | values-prod로 오버라이드 |
| Redis-auth | `apex-infra-redis-auth-master.apex-infra.svc.cluster.local:6379` | values-prod로 오버라이드 |
| PostgreSQL | `apex-infra-postgresql.apex-infra.svc.cluster.local:5432` | values-prod로 오버라이드 |
| PgBouncer | `apex-infra-pgbouncer.apex-infra.svc.cluster.local:6432` | values-prod로 오버라이드 |

### Probes 전략

모든 서비스에 **3종 프로브** 적용 (library chart 공통):

| 프로브 | 역할 | 설정 |
|--------|------|------|
| **startupProbe** | 콜드 스타트 보호. Kafka consumer group 리밸런싱, Redis/PG 연결 대기 | `failureThreshold: 30`, `periodSeconds: 5` (최대 150초) |
| **livenessProbe** | 프로세스 행 감지 → 자동 재시작 | `initialDelaySeconds: 10`, `periodSeconds: 15` |
| **readinessProbe** | 트래픽 수신 준비 확인 → Service 엔드포인트 등록 | `initialDelaySeconds: 5`, `periodSeconds: 5` |

K8s TOML ConfigMap에 `[metrics]` 섹션을 명시하여 metrics HTTP 서버를 활성화한다:
```toml
[metrics]
enabled = true
port = 8082    # 서비스별 상이
```

**구현 선행 조건**: 현재 각 서비스의 `main.cpp`에서 `ServerConfig`의 `.metrics = {}`로 하드코딩되어 있어 TOML의 `[metrics]` 섹션이 무시된다. 구현 단계에서 각 서비스의 `main.cpp`를 수정하여 TOML에서 파싱한 metrics 설정을 `ServerConfig`에 와이어링해야 한다. 이 변경이 없으면 Auth/Chat/Log의 HTTP 기반 liveness/readiness 프로브가 실패한다.
