# Auto-Review 전면 개편 설계서

**작성일**: 2026-03-14
**적용 범위**: v0.4.5
**대체 문서**: `20260314_180000_auto-review-improvements.md`

---

## 1. 개요

### 목적

1. **리뷰어 정밀도 향상** — 한 번에 제대로 찾아서 재리뷰 낭비 방지
2. **메인 에이전트 컨텍스트 절약** — 서브에이전트 자율성 극대화, 메인은 이슈 상세를 읽지 않음

### 범위

auto-review 시스템 전면 개편. 기존 5-리뷰어 + 메인 직접 운영 체제를 팀장 에이전트 기반 위임 모델로 전환.

### 핵심 변경점

| # | 기존 | 개편 |
|---|------|------|
| a | 리뷰 → 보고 → 별도 수정 (report-then-fix) | **find-and-fix** — 리뷰어가 발견 즉시 수정 |
| b | 메인이 리뷰어 디스패치 + 결과 취합 + 수정 지시 | **teammate mode** — SendMessage 기반 소통, 메인 컨텍스트 비소비 |
| c | 오케스트레이터(메인) 1계층 | **팀장 에이전트 신설** — 리뷰+수정+재리뷰 라운드 자율 운영 |

## 2. 아키텍처 — 에이전트 계층

```
메인 오케스트레이터 (auto-review.md 커맨드)
│
│  ① 팀장 디스패치 (TeamCreate)
│  ② 최종 결과 수령 (SendMessage)
│
└── 팀장 (review-coordinator)
    │  teammate mode — SendMessage 기반 소통
    ├── reviewer-docs-spec      원천 문서 정합성 (설계서·README·CLAUDE.md 간 일치)
    ├── reviewer-docs-records   기록 문서 리뷰 (plans/progress/review 형식·완결성)
    ├── reviewer-architecture   설계서 ↔ 코드 정합, 모듈 경계, 의존성 그래프
    ├── reviewer-logic          비즈니스 로직 정확성, 알고리즘 버그, 에러 처리 경로
    ├── reviewer-memory         할당기, RAII, lifetime, ownership, 릭 패턴
    ├── reviewer-concurrency    코루틴 안전성, 스레드 안전성, Boost.Asio 패턴, cross-core
    ├── reviewer-api            공개 API/concept 설계, 네이밍, 인터페이스 명확성
    ├── reviewer-test-coverage  테스트 커버리지 누락 탐지 (미테스트 코드 경로, 엣지케이스)
    ├── reviewer-test-quality   테스트 코드 품질 (assertion 적절성, 격리, ASAN/TSAN 호환)
    ├── reviewer-infra          CMake, vcpkg, CI/CD, Docker, 크로스컴파일러(GCC/MSVC)
    └── reviewer-security       입력 검증, 크레덴셜 노출, 인젝션, 보안 기본값
```

### 계층별 책임

| 계층 | 책임 | 하지 않는 것 |
|------|------|-------------|
| **메인** | Phase 0 초기화, 팀장 디스패치/재디스패치, PR 생성, CI 모니터링+push, 머지/정리 | 이슈 상세 읽기, 수정 지시, 리뷰어 직접 관리 |
| **팀장** | 리뷰어 팀 구성 (스마트 스킵 + 재량 판단), 파일 소유권 배정+충돌 조율, 라운드 관리, 최종 보고서 취합 | 리뷰 실무, 코드 수정 |
| **리뷰어 11명** | 자기 도메인 이슈 발견+즉시 수정 (find-and-fix), 에스컬레이션, 동료와 share | 소유권 밖 파일 수정 |

## 3. 리뷰어 구성 (11명)

### 기존 → 개편 매핑

| 기존 (5명) | 개편 (11명) | 변경 유형 |
|------------|-----------|----------|
| docs | **docs-spec** + **docs-records** | 2분화 |
| structure | **architecture** + **infra** | 2분화 |
| code | **logic** + **memory** + **concurrency** | 3분화 |
| test | **test-coverage** + **test-quality** | 2분화 |
| general | *(제거)* | 역할 분배 |
| *(신설)* | **api** | 신설 |
| *(신설)* | **security** | 신설 |

### 리뷰어별 상세 책임

| 리뷰어 | 도메인 | 주요 체크 항목 |
|--------|--------|---------------|
| **docs-spec** | 원천 문서 정합성 | Apex_Pipeline.md(마스터 설계서)·CLAUDE.md·README.md 간 버전 번호 일치, 용어 통일, 로드맵 현행화, 문서 위치 적합성. 리뷰 관점: "이 문서가 다른 원천 문서들과 일치하는가?" |
| **docs-records** | 기록 문서 리뷰 | plans/progress/review/ 하위 문서 파일명 타임스탬프(`YYYYMMDD_HHMMSS`) 검증, 문서 포맷·완결성, 계획-진행-리뷰 간 추적성. 리뷰 관점: "이 기록이 형식에 맞고 내용이 완결적인가?" |
| **architecture** | 설계 정합 | 설계서 ↔ 코드 일치, 모듈 경계 위반, 의존성 방향, 레이어 계층 준수, ADR 정합. docs-spec과의 역할 구분: architecture = 설계서 ↔ 코드 정합, docs-spec = 설계서 ↔ 다른 문서 정합 |
| **logic** | 비즈니스 로직 | 알고리즘 정확성, 에러 처리 경로 누락, 상태 전이 오류, 엣지 케이스 |
| **memory** | 메모리 관리 | 커스텀 할당기 정합, RAII 위반, lifetime/ownership 오류, 릭 패턴, aligned_alloc 분기 |
| **concurrency** | 동시성 | 코루틴 안전성, 데이터 레이스, Boost.Asio strand/executor 패턴, cross-core 시나리오, TSAN 억제 타당성 |
| **api** | 인터페이스 설계 | concept 정의(PoolLike, CoreAllocator 등), 공개 API 일관성, 네이밍 규칙, 타입 안전성, 사용성 |
| **test-coverage** | 테스트 커버리지 | 프로덕션 코드 기반 미테스트 코드 경로 탐지, 엣지케이스 누락, 새 API의 테스트 부재, 분기 커버리지 |
| **test-quality** | 테스트 코드 품질 | assertion 적절성(EXPECT_TRUE→EXPECT_EQ 등), 테스트 격리성, 네이밍 규칙, ASAN/TSAN/LSAN 호환성, 테스트 구조, 불필요한 의존성 |
| **infra** | 빌드/CI/인프라 | CMakeLists.txt, vcpkg.json, CI workflow, Docker, 크로스컴파일러(GCC/MSVC) 호환, suppressions |
| **security** | 보안 | 입력 검증, 크레덴셜/시크릿 노출, 인젝션 가능성, 보안 기본값, 권한 관리 |

### 분화 근거

- **docs → docs-spec + docs-records**: 원천 문서(설계서·README·CLAUDE.md)와 기록 문서(plans/progress/review)는 리뷰 관점이 근본적으로 다름. 원천 문서는 "다른 원천 문서들과 일치하는가?"를 검증하고, 기록 문서는 "형식에 맞고 내용이 완결적인가?"를 검증. 두 관점은 독립적이므로 병렬 수행 가능
- **code → logic + memory + concurrency**: C++23 코루틴 + 커스텀 할당기 프로젝트에서 로직/메모리/동시성은 완전히 다른 전문성. 한 리뷰어가 셋 다 보면 깊이 부족
- **structure → architecture + infra**: 고수준 설계 정합성(모듈 경계, ADR)과 빌드/CI 실무(CMake, Docker)는 별개 역량
- **test → test-coverage + test-quality**: coverage 리뷰어는 프로덕션 코드를 깊이 이해하여 "뭐가 테스트 안 됐나?"를 찾고, quality 리뷰어는 테스트 코드를 깊이 봐서 assertion·격리·구조를 검증. 전문성이 다르고, TDD 프로젝트라 테스트 코드가 방대하여 한 명이 커버하기 어려움. 두 리뷰어가 완전 독립 병렬 수행 가능
- **general 제거**: 감사(catch-all) 역할. 11명이 도메인별로 충분히 커버하면 불필요. 기존 general 담당(빌드, CI, 의존성)은 infra로, 라이선싱은 security로 이관
- **api 신설**: concept 기반 설계(PoolLike, CoreAllocator, FrameCodecLike 등)가 프레임워크의 핵심 계약. 전담 리뷰어 없으면 인터페이스 불일치가 누적
- **security 신설**: 서버 프레임워크에서 보안은 선택이 아닌 필수. 다른 리뷰어에 분산되면 체계적 점검 불가

## 4. 통신 프로토콜

### 메시지 유형 (6종)

| 유형 | 발신 → 수신 | 내용 |
|------|------------|------|
| **dispatch** | 메인 → 팀장 | 리뷰 실행 요청 (아래 디스패치 스키마 참조) |
| **assign** | 팀장 → 리뷰어 | 담당 파일 목록 + diff + 리뷰 지시 |
| **finding** | 리뷰어 → 팀장 | 발견+수정 결과 (수정됨 / 에스컬레이션 / 공유 필요) |
| **share** | 리뷰어 → 리뷰어 | 다른 도메인에 영향 주는 발견 공유 |
| **reassign** | 팀장 → 리뷰어 | 충돌 해결, 소유권 변경, 추가 리뷰 지시 |
| **report** | 팀장 → 메인 | 최종 요약 (수정 건수, 에스컬레이션 건수, 빌드 필요 여부) |

### 메인→팀장 디스패치 스키마 (dispatch)

메인이 팀장에게 리뷰를 위임할 때 전달하는 정보 구조:

| 필드 | 설명 | 예시 |
|------|------|------|
| **mode** | 리뷰 모드 | `task` / `full` |
| **changed_files** | 변경 파일 목록 (task 모드 시) | `["src/pool.cpp", "include/pool.hpp"]` |
| **diff_ref** | diff 요약 또는 참조 커밋 범위 | `main..feature/pool-refactor` |
| **smart_skip_config** | 스마트 스킵 매핑 테이블 참조 경로 | `apex_tools/auto-review/config.md` |
| **round_limit** | 전체 라운드 상한 | `10` |
| **safety_rules** | 안전장치 규칙 | 동일 이슈 5회 반복 에스컬레이션, 전체 10라운드 초과 에스컬레이션 |

- full 모드에서는 `changed_files` 생략 (전체 파일 대상)
- 디스패치 스키마는 팀장이 리뷰어 구성, 파일 소유권 배정, 라운드 관리를 **자율적으로** 수행하기 위한 최소 컨텍스트

### finding 보고 구조

```
[수정됨] 파일:라인 — 이슈 설명 + 수정 내용
[에스컬레이션] 파일:라인 — 이슈 설명 + 왜 직접 못 고치는지
[공유] @reviewer-xxx — 이 발견이 너 도메인에도 영향 줌
```

- **[수정됨]**: 소유권 파일 내에서 직접 수정 완료. 커밋까지 포함
- **[에스컬레이션]**: 수정 불가 사유 명시 (소유권 밖 / 설계 변경 필요 / 수정 방향 복수)
- **[공유]**: 대상 리뷰어를 태그하여 해당 도메인 검토 요청

### share 메시지 타이밍별 처리 규칙

share는 **"참고 정보"** 성격으로, 수신 리뷰어가 반드시 행동해야 하는 것은 아니다.

수신 리뷰어 상태별 처리:

| 수신 시점 | 처리 방식 |
|-----------|----------|
| **작업 중** | share 내용을 현재 리뷰/수정에 반영하여 계속 진행 |
| **작업 완료 후** | 수신 리뷰어가 직접 처리하지 않고 팀장에게 전달 → 다음 라운드에서 반영 |

- 작업 완료 후 도착한 share를 처리하기 위해 이미 끝난 리뷰어를 재가동하지 않는다 — 다음 라운드의 재리뷰 스마트 스킵에서 자연스럽게 포함됨

## 5. Find-and-Fix 모델

### 리뷰어 작업 흐름

```
assign 수령
  → 파일 읽기 / 분석
  → 이슈 발견
  → 직접 수정 가능?
     ├─ Yes → 수정 + 커밋 + finding[수정됨] 보고
     └─ No  → finding[에스컬레이션] 보고
  → 다른 도메인 영향?
     └─ Yes → finding[공유] + share 전송
```

### 수정 판단 기준

| 직접 수정 | 에스컬레이션 |
|-----------|-------------|
| 소유권 파일 내 완결 | 소유권 밖 파일에 영향 |
| 명확한 버그 (로직/타입/경계값) | 설계 변경이 필요한 구조적 문제 |
| 네이밍/주석/문서 불일치 | 수정 방향이 복수 (판단 필요) |
| 단일 파일 리팩토링 | 복합 이슈 (여러 리뷰어 도메인 교차) |

### 파일 소유권 관리

- **초기 배정**: 팀장이 스마트 스킵 매핑 + diff 분석으로 파일별 주 소유자 지정
- **공유 파일**: 주 소유자(수정 권한) + 참조 리뷰어(읽기+finding만) — 충돌 원천 차단
- **동적 재배정**: 소유자가 "내 도메인 아님" 판단 시 팀장에게 재배정 요청

#### 소유권 배정과 매핑의 관계

스마트 스킵 매핑(§6)은 **"어떤 리뷰어가 이 파일을 리뷰(읽기)할지"**를 결정하는 테이블이다. **수정 권한(소유권)**과는 별개의 개념:

- 매핑 예: `.cpp` → logic, memory, concurrency, api 4명이 리뷰 대상
- 소유권: 이 4명 중 **파일당 주 소유자는 1명만** 팀장이 배정
- 나머지 3명은 읽기 전용 참조 → 이슈 발견 시 share 또는 에스컬레이션

#### 소유권 배정 순서

1. 팀장이 전체 파일에 대해 소유권 배정 완료
2. 배정 결과를 assign 메시지에 포함하여 리뷰어에게 전달
3. 리뷰어들 병렬 리뷰/수정 시작

**소유권 배정 완료 전 수정 착수 금지** — 배정 전 수정 시작 시 충돌 위험

### 커밋 전략

리뷰어별 개별 커밋: `fix(review-{name}): {요약}`
최종 merge 시 squash되므로 개별 커밋은 세분화해도 무방.

### 에스컬레이션 처리

팀장이 에스컬레이션을 받으면 3가지 중 택 1:
1. 다른 리뷰어에게 **재배정** (소유권 이전)
2. 관련자 모아 **협의 지시** (share + reassign 조합)
3. 해결 불가 → **메인에 전달** (report에 포함)

## 6. 스마트 스킵 — 2단계 판단

### 설계 원칙

데이터 매핑은 팀장 판단을 **"보조"**하는 것이지 **"대체"**하는 게 아니다.

### 1단계: 데이터 매칭

파일 패턴 → 리뷰어 자동 선정.

| 파일 패턴 | 영향 리뷰어 |
|-----------|------------|
| `.cpp`, `.hpp` (src/) | logic, memory, concurrency, api, test-coverage |
| `test_*.cpp` | test-coverage, test-quality, logic |
| `*allocator*`, `*pool*` | memory |
| `*coroutine*`, `*async*` | concurrency |
| `concept*`, `PoolLike*` | api |
| `CMakeLists.txt`, `vcpkg.*` | infra |
| `*.yml` (CI), `Dockerfile` | infra |
| `*.md` (docs/) | docs-spec, docs-records, architecture |
| `Apex_Pipeline.md` | docs-spec, architecture |
| `README.md`, `CLAUDE.md` | docs-spec |
| `plans/`, `progress/`, `review/` | docs-records |

### 2단계: 팀장 재량 판단

데이터 매핑은 팀장 판단의 **"출발점"**이지 **"최종 결정"**이 아니다. 팀장이 diff 내용을 분석하여 **추가도 스킵도** 할 수 있다.

```
1단계: 데이터 매칭 (파일 패턴 → 리뷰어 기본 후보 선정)
2단계: 팀장 재량 판단 (diff 내용 분석)
       ├── 추가: 매핑에 안 걸렸지만 리뷰가 필요한 리뷰어 추가 디스패치
       └── 스킵: 매핑에 걸렸지만 변경 내용상 리뷰가 불필요한 리뷰어 스킵
```

#### 추가 디스패치

- 매핑에 안 걸리지만 리뷰 필요한 리뷰어를 **추가**
- 변경 내용/diff 의미를 분석하여 판단 (파일 확장자만으로 안 되는 케이스)
- 예: `.hpp` 변경이지만 보안 관련 입력 검증 코드 → security 추가

#### 스킵

- 매핑에 걸렸지만 변경 내용상 리뷰가 **불필요한** 리뷰어를 스킵
- 매핑은 파일 패턴 기반이므로 파일이 걸려도 실제 diff가 해당 도메인과 무관할 수 있음
- 예:
  - **테스트 코드 오타 수정** → `test_*.cpp` 매핑으로 test-coverage, test-quality, logic 3명이 후보지만, 단순 오타 수정이므로 팀장이 test-quality만 남기고 나머지 스킵
  - **문서 오타 수정** → `*.md` 매핑으로 docs-spec, docs-records, architecture 3명이 후보지만, 해당 문서의 담당 리뷰어 1명만 디스패치

### full 모드

full 모드에서는 팀장 재량 스킵 없이 **전원 디스패치** (기존 규칙 유지).

### 재리뷰 스마트 스킵

1. 수정된 파일 기반으로 영향 리뷰어 재선정
2. 직전 라운드 Clean(이슈 0건) 리뷰어는 스킵
3. 팀장 재량으로 추가/제외 조정

## 7. Phase 구조 통합

### 기존 → 개편 Phase 매핑

| Phase | 기존 | 개편 후 | 실행 주체 |
|-------|------|---------|----------|
| 0 | 초기화 | 변경 없음 | 메인 |
| 1+2 | 병렬 리뷰 + 수정 루프 | 팀장에게 위임 — 리뷰+수정+재리뷰 자율 운영 | 팀장 |
| 3 | PR 생성 | 변경 없음 — 팀장 보고서 기반 | 메인 |
| 4 | CI 수정 | CI 모니터링은 메인, 실패 시 수정은 팀장에게 재위임 | 메인 + 팀장 |
| 5 | 완료/머지 | 변경 없음 | 메인 |

### 팀장 내부 라운드 관리

```
Round N:
  디스패치 (assign)
  → 리뷰어 수정 (find-and-fix)
  → finding 취합
  → 빌드+테스트 검증
     ├─ 통과 → Clean? (이슈 0건)
     │          ├─ Yes → 종료, report 전송
     │          └─ No  → 재리뷰 스마트 스킵 적용 → Round N+1
     └─ 실패 → 실패 원인 파일의 소유 리뷰어에게 수정 지시 → 수정 후 재빌드
```

#### 라운드별 빌드 검증

각 라운드의 finding 취합 후, 팀장이 **로컬 빌드+테스트를 실행**하여 수정의 정확성을 즉시 확인한다.

- **왜 라운드마다 검증하는가**: 빌드/테스트를 CI(Phase 4)까지 미루면 잘못된 수정이 여러 라운드에 걸쳐 누적되어 원인 파악이 어려워진다. 라운드마다 검증하면 어떤 수정이 빌드를 깨뜨렸는지 즉시 특정 가능
- **실패 시 처리**: 팀장이 직접 수정하지 않는다. 빌드 실패 원인 파일의 소유 리뷰어에게 수정을 지시하고, 수정 완료 후 재빌드하여 통과를 확인한 뒤 다음 단계로 진행
- **통과 후 흐름**: 빌드 검증 통과 → Clean 판정(이슈 0건이면 종료) 또는 다음 라운드 진입

- 동일 이슈 **5회 반복** → 팀장이 메인에 에스컬레이션 (기존 안전장치 유지)
- **전체 라운드 상한 10회** — 서로 다른 이슈가 계속 발견되어 라운드가 무한 반복되는 케이스 방지
  - 10라운드 초과 시 팀장이 메인에 에스컬레이션: 현재까지의 수정 결과 + 미해결 이슈 목록을 report로 전달
  - 메인이 판단: 추가 라운드 허용 (상한 재설정) / 현재 상태로 PR 진행 / 수동 개입

### 리뷰어 실패/타임아웃 처리

리뷰어 응답 없음 또는 에이전트 크래시 시 팀장이 감지하여 처리:

| 상황 | 팀장 대응 |
|------|----------|
| 리뷰어 응답 없음/크래시 | 해당 리뷰어 담당 파일을 다른 리뷰어에게 **재배정** or 다음 라운드로 **이월** |
| 재배정 불가 (도메인 전문성 필요) | 해당 도메인을 **"미리뷰"**로 표시하고 report에 포함 |
| 최종 보고서 | 실패한 리뷰어의 도메인을 **미리뷰 영역**으로 명시 |

메인이 report에서 미리뷰 영역을 확인한 뒤 판단:
- 해당 도메인 리뷰어를 재생성하여 **재시도**
- 메인이 직접 **수동 리뷰**
- 리스크 수용 후 **진행**

### CI 실패 시 흐름

1. 메인이 CI 모니터링 + 실패 감지
2. 팀장에게 CI 로그 전달 + 수정 지시
3. 팀장이 실패 유형 판단:
   - 코드 문제 → 관련 리뷰어 배정
   - 인프라 문제 → reviewer-infra 배정
4. 수정 완료 → 메인에 report → 메인이 push + CI 재실행

## 8. 확장성

### 리뷰어 추가 절차

1. `agents/reviewer-{name}.md` 작성
2. 스마트 스킵 매핑에 파일 패턴 추가 (`config.md`)
3. 끝 — 팀장/메인 코드 수정 불필요

### 가능한 이유

- 팀장은 리뷰어 목록을 **하드코딩하지 않음** — 디스패치 시점에 `agents/` 디렉토리에서 가용 리뷰어 동적 탐색
- 통신 프로토콜은 리뷰어 수에 **비의존적** — 메시지 유형 6종은 N명에서도 동일
- 매핑은 **데이터 테이블** — 패턴 한 줄 추가로 연결

### 향후 확장 예시 (v0.5+)

| 리뷰어 | 시점 | 도메인 |
|--------|------|--------|
| reviewer-protocol | v0.5 | 서비스 간 통신 (FlatBuffers, 메시지 스키마) |
| reviewer-adapter | v0.5 | 외부 어댑터 사용 패턴 (Kafka/Redis/PG) |
| reviewer-perf | v0.6 | 핫패스 분석, 벤치마크 회귀 |
