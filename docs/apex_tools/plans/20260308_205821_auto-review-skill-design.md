# Auto-Review Skill 설계서

## 개요

자동화된 리뷰 + 수정 프로세스를 Claude Code 슬래시 커맨드(`/auto-review`)로 제공.
멀티에이전트 병렬 리뷰 → 병렬 수정 → 재리뷰 루프(Clean까지) → PR → CI 통과까지 100% 자동.

## 호출 방식

```
/auto-review task   # 현재 작업 브랜치 변경분만 리뷰 (git diff main...HEAD)
/auto-review full   # 전체 프로젝트 리뷰 (diff 무관)
```

## 스킬 위치

프로젝트 로컬 플러그인: `apex_tools/claude-plugin/`

```
apex_tools/claude-plugin/
├── .claude-plugin/
│   └── plugin.json
├── commands/
│   └── auto-review.md        ← 오케스트레이터 (프로세스 흐름)
└── agents/
    ├── reviewer-docs.md       ← 문서 정합성 에이전트
    ├── reviewer-structure.md  ← 구조/구현 정합성 에이전트
    ├── reviewer-code.md       ← 코드 리뷰 에이전트
    ├── reviewer-test.md       ← 테스트 리뷰 에이전트
    └── reviewer-general.md    ← 기타 리뷰 에이전트
```

## 셋업

구현 완료 후 제공할 셋업 가이드:

1. **플러그인 등록** — 프로젝트 루트에서 CLI 명령 또는 프로젝트 설정 파일로 로컬 플러그인 등록
   - 정확한 등록 방법은 구현 시 `plugin-dev:plugin-structure` 스킬로 확인
2. **사용법** — `/auto-review task` 또는 `/auto-review full`로 호출
3. **셋업 문서화** — README 또는 CLAUDE.md에 클론 후 1회 실행할 셋업 명령 기재

## 프로세스 흐름

```
┌─ 호출: /auto-review [task|full]
│
├─ Phase 1: 리뷰
│   ├─ 리뷰 범위 결정 (task: diff 기반 / full: 전체)
│   ├─ 5개 리뷰어 병렬 디스패치
│   │   └─ 각 리뷰어는 자율적으로 서브에이전트 추가 분할 가능
│   ├─ 결과 취합 → 종합 리뷰 보고서 작성
│   └─ 이슈 0건이면 → Phase 3으로 점프
│
├─ Phase 2: 수정 + 재리뷰 루프
│   ├─ 이슈 목록을 파일 비중복 기준으로 분할
│   ├─ 멀티에이전트 병렬 수정
│   ├─ 수정 후 Phase 1 재실행 (재리뷰)
│   ├─ 동일 이슈 5회 반복 → 중단 + 유저 보고
│   └─ Clean(0건) 달성까지 반복
│
├─ Phase 3: PR 생성
│   ├─ 리뷰 보고서 커밋
│   ├─ gh pr create
│   └─ gh run watch (CI 대기)
│
├─ Phase 4: CI 수정 루프
│   ├─ CI 실패 시 → 로그 분석 → 수정 → 재푸시
│   ├─ gh run watch (재대기)
│   ├─ 동일 CI 실패 5회 반복 → 중단 + 유저 보고
│   └─ CI 전체 통과 → Phase 5
│
└─ Phase 5: 완료
    ├─ 최종 완료 보고서 작성 (리뷰 라운드 요약, 수정 내역, CI 결과)
    ├─ 보고서 커밋 + 푸시
    └─ 유저에게 완료 보고 (merge는 유저가 수동)
```

## 리뷰어 에이전트 설계

### 공통 규칙

- 이슈 심각도: **Critical / Important / Minor** 3단계
- 출력 포맷 통일: `파일:라인` 참조, 원인, 영향, 수정 방안
- 자율 병렬 분할: 리뷰 범위가 크면 서브에이전트로 분할 후 취합
- `task` 모드: diff 범위 내 파일만 대상
- `full` 모드: 전체 프로젝트 대상

### 리뷰어별 담당

| 리뷰어 | 담당 | 주요 체크 항목 |
|--------|------|---------------|
| **docs** | 문서 정합성 | 마스터 문서, 설계서, README, MEMORY가 최신 상태 반영하는지 + 각 문서의 내용을 읽고 해당 문서가 현재 위치에 적합한지 판단(CLAUDE.md 문서 위치 규칙 기준) + 파일명(YYYYMMDD_HHMMSS 포맷) 준수 여부 |
| **structure** | 구조/구현 정합성 | 프로젝트 구조, 파일 배치, 모듈 경계가 설계 철학과 일치하는지 |
| **code** | 코드 품질 | 의도대로 구현됐는지, 버그, 보안, 성능, 설계 패턴 |
| **test** | 테스트 커버리지 | 누락 테스트, 엣지 케이스, 잘못된 assertion, 테스트 격리 |
| **general** | 기타 | 위 4개가 놓칠 수 있는 영역 자율 판단 (빌드, CI, 의존성, 라이선싱 등) |

## 안전장치

### 이슈 반복 감지

- 각 이슈를 `파일경로:카테고리:이슈요약` 키로 추적
- 라운드마다 누적 카운트
- 동일 이슈가 **5회** 반복되면 "해결 불가"로 마킹, 루프 중단, 유저 보고

### CI 실패 반복 감지

- 동일 CI 잡이 같은 원인으로 **5회** 실패하면 루프 중단, 유저 보고

## 설계 결정

- **접근 방식**: 오케스트레이터 + 서브 프롬프트 (모놀리식 대비 유지보수 용이)
- **2단계 병렬화**: 리뷰어 5개 병렬 + 각 리뷰어 내부 자율 서브에이전트 분할
- **CI 대기**: `gh run watch` 블로킹 방식
- **자동 범위**: 리뷰~CI 통과까지 / merge+push는 유저 수동
- **리뷰 루프 제한**: 라운드 수 무제한, 동일 이슈 5회 반복 시에만 중단
