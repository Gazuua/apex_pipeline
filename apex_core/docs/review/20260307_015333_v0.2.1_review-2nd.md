## 발견된 이슈 (코드)

### Critical (3건)

**C-1. TimingWheel `schedule()` — ticks_from_now 범위 검증 부재**
- **파일:** `src/timing_wheel.cpp:53`
- **발견:** 코어 인프라 에이전트
- `ticks_from_now >= num_slots_` 입력을 아무 검증 없이 받아들임
- 같은 슬롯에 미만료 엔트리가 누적되어 `tick()` O(1) 보장이 깨질 수 있음
- **수정 제안:** `assert(ticks_from_now < num_slots_)` 또는 에러 반환 추가

**C-2. ServiceBase `handle()`/`route()` — raw `self` 포인터 캡처 수명 위험**
- **파일:** `include/apex/core/service_base.hpp:77-83`
- **발견:** 프레임워크 에이전트 + clangd LSP
- `auto* self = static_cast<Derived*>(this)` 후 람다에서 캡처
- 서비스 소멸 후 디스패처에 핸들러가 남으면 댕글링 포인터
- `stop()`에서 등록된 핸들러를 unregister하지 않음
- 현재 Server가 서비스+디스패처를 동시 소유하므로 실질적 문제는 없으나, 프레임워크 안전성 관점에서 취약
- **수정 제안:** `on_stop()`/소멸자에서 핸들러 unregister, 또는 수명 제약을 헤더 주석에 명시

**C-3. Server::Config — clangd `default_member_initializer_not_yet_parsed` 에러**
- **파일:** `include/apex/core/server.hpp:29-35`
- **발견:** clangd LSP
- `Config` 구조체의 기본 멤버 초기화자가 클래스 내부 템플릿에서 designated initializer로 사용될 때 clang 파싱 에러
- MSVC에서는 문제 없으나, clang 기반 크로스 컴파일 시 빌드 실패 가능
- **수정 제안:** `Config`를 `Server` 클래스 외부에 별도 정의하거나, 명시적 기본 생성자 사용

### Important (13건)

#### 수명/안전성 (5건)

**I-1. Server `stop()`/`start_tick_timer()` — `[this]` 캡처 수명 위험**
- **파일:** `src/server.cpp:60, 156`
- **발견:** 네트워크 에이전트 + clangd LSP
- `post(io_ctx_, [this]() {...})`와 `tick_timer_.async_wait([this](...) {...})`에서 Server 조기 소멸 시 댕글링
- 현재 사용 패턴에서는 안전하나, 전제 조건(io_ctx.run() 완료 전 Server 소멸 금지)이 코드에 표현되지 않음
- **수정 제안:** 수명 제약 문서화, 또는 shared_ptr guard 추가

**I-2. Session `set_state()`가 public — 상태 불변식 위반 가능**
- **파일:** `include/apex/core/session.hpp:61`
- **발견:** 네트워크 에이전트 + clangd LSP
- 외부에서 `Active -> Connected` 역전이 등 임의 호출 가능
- **수정 제안:** private으로 변경 + `friend class SessionManager`, 또는 전이 메서드(`activate()`) 제공

**I-3. TcpAcceptor/Server `running_` non-atomic + 일관성 부재**
- **파일:** `include/apex/core/tcp_acceptor.hpp:45`, `server.hpp:84`
- **발견:** 네트워크 에이전트 + clangd LSP
- `CoreEngine::running_`은 `atomic<bool>`, `Server`/`TcpAcceptor`는 일반 `bool`
- 테스트에서 메인 스레드가 `stop()` 호출, accept_loop은 io_ctx 스레드에서 실행 — 이론적 data race
- **수정 제안:** `atomic<bool>`로 변경하거나 stop()을 post() 기반으로 통일

**I-4. FrameCodec `try_decode()` — 두 번째 `linearize()` span 무효화 불변식**
- **파일:** `src/frame_codec.cpp:14, 34`
- **발견:** 프레임워크 에이전트 + clangd LSP
- 첫 번째 linearize 결과를 두 번째 linearize가 무효화할 수 있음
- 현재 `header`를 값 복사하여 안전하나, 리팩토링 시 UAF 위험
- **수정 제안:** 헤더 10바이트를 로컬 배열로 복사 후 파싱하여 의존성 제거

**I-5. Server `TMP_BUF_SIZE` vs 사용자 지정 `recv_buf_capacity` 불일치 가능**
- **파일:** `include/apex/core/server.hpp:88-91`
- **발견:** 네트워크 에이전트
- `Config{.recv_buf_capacity = 2048}` 시 `TMP_BUF_SIZE(4096) > recv_buf_capacity`
- 정상 세션이 버퍼 오버플로로 불필요하게 끊어짐
- **수정 제안:** 생성자에서 `assert(config_.recv_buf_capacity >= TMP_BUF_SIZE)` 추가

## API 일관성 검토 (clangd LSP 기반)

| 관점 | 상태 | 비고 |
|------|------|------|
| `running()` 반환 타입 | 일관 | CoreEngine, Server, TcpAcceptor 모두 `bool` 반환 |
| `running_` 저장 타입 | **불일치** | CoreEngine: `atomic<bool>`, Server/TcpAcceptor: `bool` |
| Non-copyable 패턴 | 일관 | 모든 핵심 클래스에 copy/move 삭제 |
| 에러 반환 방식 | 일관 | `std::expected` 일관 사용 |

| `stop()` 의미론 | **미묘한 차이** | CoreEngine: 즉시 실행, Server: post로 지연 실행 |
| const 정확성 | 양호 | `[[nodiscard]]` + `const noexcept` 일관 적용 |

## 종합 판정

**전체 코드 품질: 높음**

v0.2.0 리뷰의 Critical 15건이 모두 올바르게 수정되었고, 코루틴 전환이 일관되게 반영되어 있다. clang-tidy strict 모드에서 경고 0건은 코드 품질이 높다는 객관적 지표이다.

신규 Critical 3건은 모두 **현재 사용 패턴에서는 안전하지만, 프레임워크 확장 시 위험해지는 잠재적 이슈**이다:
- C-1 (TimingWheel 범위 검증): assert 한 줄로 수정 가능
- C-2 (ServiceBase self 캡처): 주석 명시 또는 unregister 추가
- C-3 (Server::Config clang 호환): Config 구조체 위치 이동

Important 13건 중 수명/안전성 5건(I-1~I-5)은 v0.3.0 전에 검토 권장, 성능 4건(I-6~I-9)은 프로파일링 후 판단, 코드 품질 4건(I-10~I-13)은 점진적 개선 가능.

테스트 커버리지는 정상 경로(happy path)에서 우수하나, **에러 경로와 방어 코드 검증이 상대적으로 부족**하다. Session 컴포넌트의 테스트가 가장 미흡하며, HandlerFailed 경로 테스트(TC-1)는 반드시 추가해야 한다.
