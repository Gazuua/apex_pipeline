## v0.2.2 변경사항 정합성

## Important — 코드 (9건)

### I-1. TimingWheel `tick()` — delete 후 콜백 호출, ID 재사용 시점 문제

- **파일:** `src/timing_wheel.cpp:130-136`
- **발견:** LSP + 코어 인프라 에이전트
- **문제:**
  ```cpp
  entries_[expired_id] = nullptr;
  free_ids_.push_back(expired_id);  // ID 즉시 재사용 풀에 반환
  --active_count_;
  delete e;                         // Entry 메모리 해제
  on_expire_(expired_id);           // 콜백에서 schedule() 시 같은 ID 재할당 가능
  ```
  콜백 내 `schedule()` 호출 시 `free_ids_`에서 방금 반환한 ID를 재사용하여 논리적 혼란 발생.
  현재 사용처(`SessionManager::on_timer_expire`)에서는 schedule()을 호출하지 않아 **현재 안전**하나, 향후 확장 시 시한폭탄.
- **수정 제안:**
  - `free_ids_.push_back`을 모든 콜백 완료 후 일괄 처리, 또는
  - `delete e`를 콜백 이후로 지연 (풀 할당 전환 시 자연 해결)

### I-2. TimingWheel `schedule()` — 매 호출 `new Entry()` 힙 할당 (v0.2.1 I-7 재등록)

- **파일:** `src/timing_wheel.cpp:66`
- **발견:** LSP + 코어 인프라 에이전트
- **문제:** `SessionManager::touch_session()`이 `cancel() + schedule()` 패턴이므로 매 클라이언트 메시지마다 `delete + new` 발생. 250만 동접 환경에서 초당 수백만 회 힙 할당 가능.
- **수정 제안:**
  - **방안 A (권장):** `touch_session()`에서 `cancel()+schedule()` 대신 `TimingWheel::reschedule()`을 사용. Entry 재할당 없이 in-place 갱신. I-1 이슈도 함께 우회됨.
  - **방안 B:** `TypedSlabPool<Entry>`를 TimingWheel 내부에 소유, `new/delete` → 풀 할당 전환.

### I-3. SlabPool `deallocate()` — Release 빌드에서 double-free 탐지 불가 (v0.2.1 I-12 재등록)

- **파일:** `src/slab_pool.cpp:81-91`
- **발견:** LSP + 코어 인프라 에이전트
- **문제:** `assert(owns(ptr))`만 존재. Release에서 제거됨. double-free 시 같은 메모리가 두 번 할당되어 corruption. `owns()` 검사도 슬롯 경계 정렬 미검증 — 청크 중간 주소 해제 시 free-list 오염.
- **수정 제안:**
  1. 매직 넘버 기반 경량 검증: 각 슬롯 헤더에 `ALLOCATED`/`FREED` 마커 기록, deallocate 시 상태 확인
  2. `owns()` 내 슬롯 정렬 검증 추가: `(p - chunk.data) % slot_size_ == 0`

### I-4. Release 빌드 assert 무력화 — 3개 컴포넌트 일괄

- **파일:** `src/timing_wheel.cpp:54`, `src/server.cpp:29-30`, `src/ring_buffer.cpp:70,76`
- **발견:** 코어 인프라 + 네트워크 에이전트
- **문제:** 아래 3곳의 `assert`가 Release(`NDEBUG`)에서 완전히 제거됨:

  | 위치 | assert 내용 | Release 시 영향 |
  |------|------------|----------------|
  | TimingWheel `schedule()` | `ticks_from_now < num_slots_` | 좀비 엔트리 → 세션 누수 |
  | Server 생성자 | `recv_buf_capacity >= TMP_BUF_SIZE` | RingBuffer overflow |
  | RingBuffer `commit_write/consume` | `n <= writable/readable_size()` | buffer overrun |

- **수정 제안:** hot path가 아닌 곳은 `if (...) throw`, hot path는 `if (...) return` 또는 clamp 처리.

### I-5. Server `process_frames()` — ErrorResponse 전송 실패 무시

- **파일:** `src/server.cpp:152, 157`
- **발견:** 네트워크 에이전트
- **문제:** `co_await session->async_send_raw(error_frame)`의 반환값(`bool`)을 무시. 전송 실패 시 세션이 이미 close되었을 수 있으나, `process_frames`의 while 루프에 `session->is_open()` 가드가 없어 close된 세션에서 추가 프레임 처리 시도.
- **수정 제안:** while 루프 조건에 `&& session->is_open()` 추가, 또는 전송 실패 시 `break`.

### I-6. TcpAcceptor `accept_loop` — 에러 재시도 backoff 없음

- **파일:** `src/tcp_acceptor.cpp:44-47`
- **발견:** 네트워크 에이전트
- **문제:** EMFILE(fd 소진) 등 일시적 에러에서 `continue`로 즉시 재시도 → CPU 100% busy-loop. TODO 주석만 존재, 미구현.
- **수정 제안:** 최소 고정 지연(`co_await steady_timer(100ms)`) 또는 지수 백오프 추가.

### I-7. ProtocolBase `consume_frame` — 타입 안전성 손실

- **파일:** `include/apex/core/protocol_base.hpp:16-18`
- **발견:** 프레임워크 에이전트
- **문제:** MSVC 우회로 `template <typename FrameT>`를 사용하여 아무 타입이나 전달 가능. 잘못된 타입 전달 시 `consume_frame_impl` 내부에서야 에러 발생, 메시지 불친절.
- **수정 제안:** `static_assert(std::is_same_v<FrameT, typename Derived::FrameType>)` 추가. MSVC에서도 불완전 타입 문제가 발생하는지 확인 필요 — 발생하면 주석으로 제약 명시.

### I-8. ServiceBase `handle()`/`route()` — 중복 등록 시 ID 추적 부정확

- **파일:** `include/apex/core/service_base.hpp:98, 119`
- **발견:** 프레임워크 에이전트
- **문제:** 같은 `msg_id`로 `handle()`을 재호출하면 `registered_msg_ids_`에 동일 ID가 중복 push. `stop()` 시 같은 ID에 대해 `unregister_handler` 이중 호출.
- **수정 제안:** `registered_msg_ids_`를 `std::unordered_set<uint16_t>`으로 변경, 또는 등록 전 중복 체크.

### I-9. SessionManager `create_session()` — ServerConfig `recv_buf_capacity` 전달 누락

- **파일:** `src/session_manager.cpp:22-23`
- **발견:** 네트워크 에이전트
- **문제:** `create_session()`이 Session 기본값(8192)을 사용. ServerConfig에 `recv_buf_capacity`가 있지만 SessionManager에 전달되지 않아 설정 불일치 가능.
- **수정 제안:** SessionManager 생성자에 `recv_buf_capacity` 파라미터 추가, `create_session()`에서 전달.

## 종합 판정

**전체 코드 품질: 높음**

v0.2.2의 핵심 목표인 에러 전파 파이프라인(FIX-1)과 ProtocolBase CRTP(FIX-2)가 정확히 구현되었다. v0.2.1 Critical 3건 + TC-1이 모두 수정되어 Critical 0건 달성. clang-tidy strict 경고 0건.

신규 Important 9건 중 핵심은:
- **I-1~I-4 (Release 안전성):** assert 기반 검증이 Release에서 무력화되는 공통 패턴. 프로덕션 서버 프레임워크로서 런타임 검증 강화가 최우선 과제.
- **I-5~I-6 (에러 경로):** ErrorResponse 전송 실패 무시 + accept backoff 부재는 프로덕션 안정성에 직결.
- **I-7~I-9 (설계 보강):** ProtocolBase 타입 안전성, ServiceBase 중복 등록, Config 전달 누락은 프레임워크 확장성에 영향.

### 수정 우선순위 권장

1. **즉시 수정 (v0.2.3):** I-1~I-6 (Release 안전성 + 에러 경로), TI-1~TI-2 (테스트 보강)
2. **권장 수정 (v0.2.3):** I-7~I-9 (설계 보강)
3. **점진적 개선:** S-1~S-9, TS-1~TS-4
