## 불일치 항목 (3건)

### M-1. 에러 전파 파이프라인이 설계와 다르다 — 코드 수정 필요

- **설계:** `design-decisions.md:87-89`
  - 핸들러에서 `co_return apex::error(ErrorCode::X)` → 자동 ErrorResponse
  - 커스텀 필요 시 `session.send()` 후 `co_return apex::ok()`
  - 핸들러가 에러 리턴했는데 응답 안 보냈으면 자동 ErrorResponse 생성
- **코드:**
  - 핸들러 시그니처: `awaitable<void>` — void 반환이라 에러를 리턴할 수 없음
  - `result.hpp`에 `ok()`/`error()` 헬퍼 정의되어 있지만 디스패치 파이프라인에 미연결
  - `server.cpp:144` — `DispatchError::UnknownMessage`만 처리
- **조치:** 코드 수정 (FIX-1)

### M-2. `alignas(64) 전역 적용` — 문서 수정 완료 ✅

- **설계:** `design-decisions.md:120` (원본) "alignas(64) 전역 적용 (false sharing 제거)"
- **코드:** MpscQueue, SlabPool에만 적용. per-core 단일 스레드 컴포넌트(TimingWheel, RingBuffer, Session 등)는 미적용
- **판단:** 단일 스레드 컴포넌트에서 false sharing은 발생 불가. 코드가 올바르고 문서가 과장됨
  - v0.2.1 2차 리뷰 I-8에서 MpscQueue Slot 패딩도 벤치마크 전 보류로 결정한 선례 참고
- **조치:** 문서 수정 완료 — "멀티스레드 공유 구조에 적용 (per-core 전용 컴포넌트는 대상 아님)"

### M-3. `BackpressureError` vs `QueueError` 네이밍 — 문서 수정 완료 ✅

- **설계:** `design-decisions.md:109` (원본) `std::expected<void, BackpressureError>`
- **코드:** `mpsc_queue.hpp:16-18` `QueueError::Full`
- **조치:** 문서 수정 완료 — `QueueError`로 통일

## 판단 완료 항목 (5건)

### A-1. Server가 CoreEngine을 사용하지 않는다 → 별도 Phase로 계획

- **설계:** io_context-per-core가 아키텍처 핵심
- **코드:** `Server`는 단일 `io_context` 기반. `CoreEngine`과 독립적으로 존재
- **판단:** 통합이 필요하나, 코어별 SessionManager/Dispatcher 분리 + SO_REUSEPORT + MPSC 라우팅 등 Phase 5급 작업량
- **조치:** PHASE-NEXT-1로 계획 (v0.4.0 목표)

### A-2. `ProtocolBase<Derived>` CRTP가 없다 → 코드 수정 필요

- **설계:** `design-decisions.md:67` "CRTP ProtocolBase\<Derived\>로 추상화, vtable 없음"
- **코드:** `ProtocolBase` 클래스 자체가 없음. WireHeader + FrameCodec 구체 클래스만 존재
- **판단:** 다음 에이전트가 WebSocket 프로토콜 추가 시 확장 지점이 없음. CRTP 베이스를 미리 만들어야 함
- **조치:** 코드 수정 (FIX-2)

### A-3. `ServiceBaseInterface` vtable vs "vtable 회피" 원칙 → 문서 수정 완료 ✅

- **설계:** `design-decisions.md:20` (원본) "CRTP 정적 다형성 (vtable 회피)"
- **코드:** `ServiceBaseInterface`에 virtual 메서드 (`start()`, `stop()` 등)
- **판단:** 의도적 하이브리드 설계
  - 핫패스 (메시지 디스패치): array O(1) → std::function → CRTP self — vtable 없음
  - 콜드패스 (서비스 생명주기): `vector<unique_ptr<ServiceBaseInterface>>` — virtual 필수
  - CRTP만으로는 이종 서비스 컨테이너 불가능
- **조치:** 문서 수정 완료 — "핫패스에서 vtable 회피, 콜드패스인 서비스 생명주기 관리는 virtual 허용"

### A-4. 코루틴 프레임 풀 할당 미구현 → 코드 수정 필요

- **설계:** `design-decisions.md:99`, ADR-21 "promise_type에 operator new/delete 오버로드 → 코어별 슬랩 풀에서 할당"
- **코드:** `boost::asio::awaitable<void>` 기본 힙 할당 사용. 커스텀 promise_type 없음
- **판단:** 코루틴 전환이 이미 완료된 상태이므로 반영 필요. 단, Boost.Asio `awaitable`의 promise_type은 라이브러리 내부 타입이라 직접 오버로드가 어려울 수 있음. 커스텀 `apex::task<T>` 도입 검토 필요
- **조치:** 코드 수정 (FIX-3)

### A-5. cross_core_call → awaitable\<T\> 미구현 → A-1 이후로 보류

- **설계:** ADR-20 "cross_core_call(target_core, lambda) → awaitable\<T\>"
- **코드:** `CoreEngine::post_to()`는 fire-and-forget (`bool` 반환)만 존재
- **판단:** 소비자(멀티코어 Server)가 없으므로 지금 구현하면 테스트 불가능한 죽은 코드. PHASE-NEXT-1 (Server-CoreEngine 통합) 이후 함께 구현
- **조치:** PHASE-NEXT-2로 계획 (PHASE-NEXT-1 의존)

## 수정 예정 리스트

### 코드 변경 (3건) — v0.3.0 목표

**FIX-1: 에러 전파 파이프라인 연결 (M-1)** `우선순위: 높음`

핸들러 시그니처를 `awaitable<void>` → `awaitable<Result<void>>`로 변경하고,
`process_frames()`에서 핸들러 반환값 검사 → 에러 시 자동 ErrorResponse 전송.

- 수정 대상: MessageDispatcher, ServiceBase, Server, 예제, 관련 테스트 전체
- `result.hpp`의 `ok()`/`error()` 헬퍼를 디스패치 파이프라인에 연결

**FIX-2: ProtocolBase\<Derived\> CRTP 추상화 (A-2)** `우선순위: 중간`

`ProtocolBase<Derived>` CRTP 베이스 클래스 추가.
`TcpBinaryProtocol : ProtocolBase<TcpBinaryProtocol>` 구현 (기존 WireHeader+FrameCodec 래핑).
향후 WebSocketProtocol 추가 가능 구조 확보.

- 수정 대상: wire_header, frame_codec, server, session

**FIX-3: 코루틴 프레임 풀 할당 (A-4)** `우선순위: 중간`

커스텀 코루틴 타입 `apex::task<T>` 도입 또는 promise_type 오버로드로
코루틴 프레임을 SlabPool에서 할당. Boost.Asio awaitable과의 호환성 검증 필요.

- 수정 대상: 핸들러 코루틴 타입, MessageDispatcher, ServiceBase

### 별도 Phase (2건) — v0.4.0+ 목표

**PHASE-NEXT-1: Server ↔ CoreEngine 통합 (A-1)**

Server가 CoreEngine을 소유/사용. 코어별 SessionManager, MessageDispatcher 분리.
기존 단일 io_context Server는 `SimpleServer`로 유지 (테스트/예제용).

**PHASE-NEXT-2: cross_core_call → awaitable\<T\> (A-5)**

전제 조건: PHASE-NEXT-1 완료. 요청 ID + 응답 콜백 래퍼로 코어 간 요청-응답 패턴 구현.

## 종합 판정

**설계 문서 정합성: 양호**

18개 핵심 설계 결정이 코드에 정확히 반영되어 있다. 불일치 3건 중 2건(M-2, M-3)은 문서 표현 문제로 즉시 수정 완료했고, 1건(M-1)은 코드 수정이 필요한 실질적 파이프라인 미연결이다.

판단 필요 5건은 설계 문서가 **v1.0.0까지의 전체 비전**과 **v0.2.1 현재 구현**을 "확정된 결정"으로 구분 없이 나열한 것이 원인이다. A-1(Server-CoreEngine 통합)과 A-5(cross_core_call)는 별도 Phase로 분리했고, A-2(ProtocolBase), A-4(코루틴 풀)는 현재 코드에 반영 가능하므로 v0.3.0 수정 대상으로 포함했다.

문서 수정 3건(M-2, M-3, A-3)은 본 검증과 동시에 반영 완료하여, 설계 문서가 현재 코드 상태를 정확히 기술하도록 갱신했다.
