# v0.2.4 Server-CoreEngine 통합 설계

**일자:** 2026-03-08
**버전 목표:** v0.2.4
**선행:** v0.2.3 (main 머지 완료), 기존 계획서(20260307_224818) 브레인스토밍 결과 반영
**상태:** 설계 확정, 구현 미착수

---

## 목표

Server를 CoreEngine과 통합하여 멀티코어 shared-nothing 아키텍처를 실현한다.
기존 싱글코어 전용 Server API를 전면 재설계하고, 코어 간 요청-응답 패턴(cross_core_call)을 구현한다.

## 설계 원칙

| 원칙 | 적용 |
|------|------|
| **프레임워크가 모든 것을 소유** | Server::run()이 io_context, 스레드, 시그널 전부 관리 |
| **shared-nothing** | 서비스 인스턴스, SessionManager, 상태 모두 코어별 독립 |
| **코어 간 통신은 명시적** | cross_core_call/post로만 코어 간 상호작용 |
| **글로벌 상태는 외부** | Redis/DB로 제어, 프로세스 내 글로벌 상태 없음 |
| **하위 호환 불필요** | 기존 API 전면 재설계, 예제/테스트 전면 갱신 |

## 주요 설계 결정

### D-1: Server::run() 블로킹 (확정)

**선택:** Option B — `server.run()` 올인원 블로킹

- 기존 `Server(io_context&, Config)` + `start()` 패턴 제거
- Server가 모든 io_context와 스레드를 내부에서 생성/소유
- Seastar `app_template::run()` 모델 참조

**근거:**
- apex_core는 "프레임워크가 실행 흐름을 소유"하는 철학 (ServiceBase CRTP)
- io_context 수명을 사용자에게 위임하면 멀티코어에서 소유권 혼란
- accept io_context 소유권 문제가 자연스럽게 해소

### D-1 후속: Signal Handling (확정)

**선택:** C 패턴 — 기본 시그널 처리 + thread-safe stop()

- `handle_signals = true` (기본): SIGINT/SIGTERM → graceful shutdown
- `stop()` thread-safe 노출 — 프로그래밍적 종료 (헬스체크 실패 등)
- Actix-web의 `.disable_signals()` opt-out 모델 참조

### D-2: cross_core_call 타입 소거 (확정)

**선택:** raw new/delete + unique ownership + API 래퍼

- MpscQueue의 trivially_copyable 제약 유지 (락프리 큐 수정은 위험)
- 힙에 CrossCoreTask 할당, 포인터를 CoreMessage.data(uint64_t)로 전달
- shared_ptr 불필요 — 소유권이 선형(호출→대상→호출)
- reinterpret_cast는 cross_core_call API 내부에 캡슐화
- SlabPool 최적화는 벤치마크에서 병목 확인 후 도입
- Seastar도 큐 자체는 포인터만 전달, 타입 소거는 큐 바깥에서 처리

**근거:**
- cross_core_call 빈도는 초당 수만 이하 (프레임워크 내부 오퍼레이션 중심)
- drain 대기(100us)가 할당 비용(30-40ns)의 1000배 이상 → 힙 할당은 병목 아님
- 단일 큐 유지 — 65536 용량에서 시스템 메시지 기아 불가

### D-3: 멀티코어 브로드캐스트 (확정)

**선택:** Redis Pub/Sub 경유

- cross_core_post 브로드캐스트는 단일 프로세스 내에서만 동작
- 프로덕션에서 Gateway N대 스케일아웃 시 어차피 Redis Pub/Sub 필요
- 동일 경로로 통일하여 코드 단순화
- cross_core_post는 프레임워크 내부 오퍼레이션 전용 (shutdown, 메트릭 등)

### ServiceBase 모델 (확정)

**선택:** Seastar식 코어별 독립 인스턴스

- `add_service<T>()` → 코어 수만큼 인스턴스 생성
- 서비스 state가 코어별 완전 분리 → data race 원천 차단
- 글로벌 상태는 Redis/DB로 제어 (MSA stateless 원칙)
- Seastar `sharded<T>` 모델 참조

**근거 — A의 단점이 apex_core에서 발현되지 않음:**
- 무거운 초기화: thin 서비스, 외부 상태는 Redis/DB
- 브로드캐스트 오버헤드: Redis Pub/Sub로 대체
- 서비스 간 로컬 호출: Kafka 경유 원칙, 직접 호출 안 함

## 아키텍처

### 전체 구조

```
Server::run()
├── accept io_context (메인 스레드)
│   ├── TcpAcceptor (accept_loop 코루틴)
│   └── signal_set (SIGINT/SIGTERM)
│
├── CoreEngine
│   ├── core[0]: io_context + thread
│   │   ├── drain_timer → MPSC drain + session_mgr.tick()
│   │   ├── EchoService 인스턴스
│   │   └── SessionManager (이 코어의 세션만)
│   ├── core[1]: 동일 구조
│   ├── core[2]: 동일 구조
│   └── core[3]: 동일 구조
│
├── MessageDispatcher (공유, read-only after setup)
└── next_core_ (atomic 라운드로빈)
```

### Server::run() 내부 흐름

```
run()
├── 1. PerCoreState 초기화 (코어별 SessionManager)
├── 2. 서비스 팩토리로 코어별 인스턴스 생성 + 핸들러 등록
├── 3. CoreEngine 설정 (message_handler, drain에서 tick 호출)
├── 4. CoreEngine::start() (워커 스레드 spawn)
├── 5. TcpAcceptor 시작 (accept_loop co_spawn)
├── 6. signal_set 등록 (handle_signals=true면)
├── 7. accept io_context run (메인 스레드 블로킹)
├── 8. CoreEngine::join() (워커 스레드 합류)
└── 9. 리소스 정리
```

## Server API

### ServerConfig

```cpp
struct ServerConfig {
    // 네트워크
    uint16_t port = 9000;
    std::string bind_address = "0.0.0.0";

    // 멀티코어
    uint32_t num_cores = 1;                       // 최소 1, "싱글코어 = 코어 1개짜리 멀티코어"
    size_t mpsc_queue_capacity = 65536;
    std::chrono::microseconds drain_interval{100};

    // 세션
    uint32_t heartbeat_timeout_ticks = 300;
    uint32_t tick_interval_ms = 100;
    size_t recv_buf_capacity = 8192;
    size_t timer_wheel_slots = 1024;

    // 시그널
    bool handle_signals = true;
};
```

### Server Public API

```cpp
class Server {
public:
    explicit Server(ServerConfig config);
    ~Server();

    template <typename T, typename... Args>
    Server& add_service(Args&&... args);  // 체이닝 지원

    void run();   // 블로킹
    void stop();  // thread-safe

    template <typename Func>
    auto cross_core_call(uint32_t target_core, Func&& func,
                         std::chrono::milliseconds timeout = 5000ms)
        -> awaitable<Result<std::invoke_result_t<Func>>>;

    template <typename Func>
    bool cross_core_post(uint32_t target_core, Func&& func);

    uint32_t core_count() const noexcept;
};
```

### 사용 예시

```cpp
int main() {
    Server({.port = 9000, .num_cores = 4})
        .add_service<EchoService>()
        .run();
}
```

### Server 내부 멤버

```cpp
class Server {
    ServerConfig config_;
    std::unique_ptr<CoreEngine> core_engine_;
    boost::asio::io_context accept_io_;             // accept 전용
```

## CoreEngine 변경

### API 변경

```cpp
class CoreEngine {
public:
    // 변경: run() → start()/join() 분리
    void start();   // 스레드 spawn (non-blocking)
    void join();    // 스레드 join (blocking)
    // run() 제거

    // 기존 유지
    void stop();
    bool post_to(uint32_t target_core, const CoreMessage& msg);
    void broadcast(const CoreMessage& msg);
    void set_message_handler(MessageHandler handler);
    boost::asio::io_context& io_context(uint32_t core_id);

    // 추가
    void set_drain_callback(std::function<void(uint32_t)> callback);
    void drain_remaining();  // shutdown 후 잔여 메시지 정리
};
```

### CoreMessage Type 확장

```cpp
enum class Type : uint8_t {

    DrainComplete,
```

별도 tick_timer 불필요. CoreEngine의 drain_timer 콜백에서 처리:

## Accept & Socket Migration

### 방식: async_accept(target_executor)

Boost 1.78+의 `acceptor.async_accept(executor)` 오버로드 사용.
accept 시점에 소켓이 타겟 코어의 io_context에 바인딩된다.

**기존 계획서 오류 수정:**
기존 계획서는 `socket = std::move(socket)` 후 다른 코어에서 사용하는 방식이었으나,
Boost.Asio에서 소켓은 move 후에도 원래 executor에 바인딩된 상태를 유지한다.
이로 인해 async 연산의 completion이 accept 스레드에 post되어 data race가 발생한다.

### accept 흐름

```
TcpAcceptor (메인 스레드)
  └── accept_loop
        │ core_id = next_core_++ % num_cores
        │ socket = co_await acceptor_.async_accept(target_executor)
        │   → 소켓이 처음부터 타겟 코어에 바인딩
        ▼
        post(target_io, [session, core_id] {
            co_spawn(read_loop(session, core_id))
        })
```

## cross_core_call 메커니즘

### CrossCoreTask 구조체

```cpp
struct CrossCoreTask {
    uint64_t request_id;
    uint32_t source_core;
    std::function<void()> execute;
    std::function<void()> set_result;
    std::atomic<bool> cancelled{false};
    boost::asio::steady_timer* resume_timer;
    boost::asio::io_context* caller_io_ctx;
};
```

### 소유권 흐름 (선형, unique ownership)

```
호출 코어 (core 0)                    대상 코어 (core 2)
───────────────────                   ───────────────────
1. new CrossCoreTask
2. resume_timer 무한 대기 세팅
3. post_to(core 2, msg)
4. co_await timer (suspend)
                                      5. drain에서 msg 수신
                                      6. if (cancelled) { delete task; return; }
                                      7. execute() → 결과 저장
                                      8. post(caller_io, timer->cancel())
10. timer cancel → resume
11. 결과 읽기
12. delete task
```

### resume 메커니즘: steady_timer cancel 패턴

- 호출 측: `steady_timer`를 `time_point::max()`로 세팅 → 무한 대기
- 대상 코어: 완료 시 호출 코어의 io_context에 `timer.cancel()` post
- `operation_aborted`를 "완료" 시그널로 해석

### 타임아웃 처리

- timeout_timer 만료 → `task->cancelled = true` → ErrorCode::CrossCoreTimeout
- 소유권이 대상 코어로 이전 (대상 코어가 cancelled 확인 후 delete)

### enqueue 실패 처리

- `post_to` 실패(큐 만석) → 호출 측이 즉시 `delete task` → ErrorCode::CrossCoreQueueFull

### ErrorCode 추가

```cpp
CrossCoreTimeout = 200,
CrossCoreQueueFull = 201,
```

## Graceful Shutdown

### 종료 시퀀스

```
server.stop()
  ├── ① acceptor_.stop()                  새 연결 차단
  ├── ② 각 코어에 세션 정리 post           코어 스레드가 살아있는 동안
  │     session_mgr.for_each(close)
  ├── ③ 모든 코어 정리 완료 대기           atomic counter
  ├── ④ core_engine_->stop()              io_context.stop()
  ├── ⑤ core_engine_->drain_remaining()   잔여 MPSC 메시지의 힙 포인터 정리
  ├── ⑥ accept_io_.stop()                 run() 블로킹 해제
  └── ⑦ core_engine_->join()              워커 스레드 합류 (run()에서)
```

### 핵심: 순서 근거

세션 정리(②)는 코어 스레드가 살아있어야 post 가능.
따라서 반드시 CoreEngine stop(④) 전에 수행한다.
잔여 메시지의 힙 포인터(CrossCoreTask*)는 drain_remaining(⑤)에서 delete하여 메모리 누수 방지.

## 테스트 전략

### 단위 테스트

| 테스트 | 검증 |
|--------|------|
| ServerStartStop | num_cores=2, run/stop 정상 |
| AcceptRoundRobin | 4 연결 → 코어 0,1,2,3 분배 |
| SessionCoreIsolation | 코어별 세션 독립성 |
| ServicePerCoreInstance | 코어 수만큼 인스턴스 생성 |
| CrossCoreCallBasic | 람다 실행 + 결과 수신 |
| CrossCoreCallTimeout | ErrorCode::CrossCoreTimeout |
| CrossCoreCallQueueFull | ErrorCode::CrossCoreQueueFull |
| CrossCorePostFireAndForget | 대상 코어 실행 확인 |
| CrossCorePostTargetCancelled | 타임아웃 후 메모리 누수 없음 |

### 통합 테스트

| 테스트 | 검증 |
|--------|------|
| MultiCoreEchoE2E | 멀티코어 에코 정상 |
| GracefulShutdown | 전 코어 세션 정리 + 스레드 종료 |
| SignalHandling | SIGINT → graceful shutdown |

### 회귀

기존 테스트를 num_cores=1로 실행. 코드 경로 동일하므로 별도 분기 없이 통과.

### 비결정론적 테스트 대응

- 테스트에서 drain_interval = 1us
- 결과 확인은 promise/future (sleep 금지)
- 타임아웃 여유있게 설정 (기본 5s)
