## Important (4건)

### N-1. TimingWheel tick() — 콜백 예외 시 메모리 누수 + ID 소실

- **파일:** `src/timing_wheel.cpp:147-152`
- **발견:** 코어 인프라 에이전트
- **문제:** Phase 3b에서 `on_expire_(expired_id)` 콜백이 예외를 던지면 해당 엔트리의 `delete e` + `free_ids_.push_back`이 실행되지 않아 메모리 누수 + ID 소실 발생. 이후 expired_buf_ 잔여 엔트리도 동일.
- **현재 영향:** SessionManager의 `on_timer_expire`는 예외를 던지지 않으므로 현재 안전. 다만 TimingWheel은 범용 컴포넌트이므로 방어 필요.
- **수정 제안:** Phase 3b 루프 내 try-catch 감싸기, 또는 `Callback` 타입에 noexcept 계약 명시.

### N-2. TcpAcceptor backoff_timer — async_wait 에러 미처리

- **파일:** `src/tcp_acceptor.cpp:51-52`
- **발견:** 프레임워크+네트워크 에이전트 + LSP 에이전트
- **문제:** `backoff_timer.async_wait(use_awaitable)`은 timer cancel 시 `system_error` 예외를 throw. `async_accept`은 `as_tuple(use_awaitable)`로 안전 처리하는 반면 비일관적. stop() 시 acceptor.close()만 하고 timer는 cancel하지 않으므로, backoff 대기 중 stop 호출 시 최대 100ms 지연 후 코루틴 종료. `detached`가 예외를 삼키지만 정리되지 않은 종료.
- **수정 제안:** `as_tuple`으로 통일하거나, backoff_timer를 멤버로 승격 후 `stop()`에서 cancel.

### N-3. heartbeat_timeout_ticks ≥ timer_wheel_slots 선검증 없음

- **파일:** `src/server.cpp` 생성자 / `include/apex/core/server.hpp:22-27`
- **발견:** LSP 에이전트
- **문제:** v0.2.3에서 `schedule()`이 `ticks_from_now >= num_slots_`이면 `out_of_range`를 throw하도록 변경. 그런데 Server/SessionManager 생성자에서 `heartbeat_timeout_ticks < next_power_of_2(timer_wheel_slots)` 선검증이 없음. 기본값(300 < 1024)은 안전하지만 사용자가 `{.heartbeat_timeout_ticks = 1024, .timer_wheel_slots = 512}` 설정 시 첫 세션 연결에서 예외 지연 발생.
- **수정 제안:** `recv_buf_capacity` 검증처럼 Server 생성자에서 선검증 추가.

### N-4. OwnsSlotAlignment 테스트 — false positive

- **파일:** `tests/unit/test_slab_pool.cpp:138-147`
- **발견:** 테스트 에이전트
- **문제:** 테스트 이름은 "OwnsSlotAlignment"이지만 `owns()`를 한 번도 호출하지 않음. allocate → deallocate → free_count 확인만 하여 기존 `AllocateAndDeallocate`와 완전 중복. `owns()`는 private이므로 직접 검증 불가.
- **수정 제안:** 테스트 제거 후 `deallocate` 내부 간접 보호로 대체, 또는 `owns()` friend 선언으로 직접 테스트.
